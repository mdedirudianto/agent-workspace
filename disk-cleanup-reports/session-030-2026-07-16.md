**Date:** 2026-07-16

**Disk before:** 91.2% used — 21.6GB free
**Disk after:** 87% used — 29GB free
**Recovered:** ~7.4GB

---

## Context

One day after S29's bulk inactive-project purge (47GB free / ~81% used on 2026-07-14), disk had regrown to 21.6GB free / 91.2% used — a ~25GB swing in under 48 hours. Investigation showed most of the regrowth was **legitimate**: grobiz (2.8GB) and eventopia (3.2GB) `node_modules` came back because both are active projects, and the grobiz Docker Compose stack (7 containers: nginx/redis/mariadb/scheduler/worker/socketio/erpnext) was actually running this session — contradicting S28/S29's assumption that Docker was dormant. `pasarbesar`/`ytgrab` correctly stayed purged (no reinstall). The real actionable regrowth was the standard cache offenders plus one new one.

Note: `df /` reports misleading numbers on this Mac (APFS container vs. volume split). Real usage came from `diskutil apfs list` / `df -H /System/Volumes/Data`.

## Actions Taken

| # | Item | Command | Recovered |
|---|------|---------|-----------|
| 1 | npm caches | `rm -rf ~/.npm/_npx ~/.npm/_cacache` | 740MB → 0 |
| 2 | codex-runtimes cache | `rm -rf ~/.cache/codex-runtimes` | 1.8GB → 0 |
| 3 | bun install cache | `rm -rf ~/.bun/install/cache` | 1.2GB → 0 |
| 4 | codex tmp | `rm -rf ~/.codex/.tmp` | 188MB → 0 |
| 5 | Gradle | `rm -rf ~/.gradle` | 3.9GB → 0 |
| 6 | meetgrab Chrome debug profile | `rm -rf ~/.meetgrab-chrome-debug` | 1.3GB → 0 |
| 7 | Docker unused images | `docker image prune -a -f` | 1.739GB reclaimed (grobiz stack + volumes untouched) |

Approved as a batch (all 4 tiers) after the recommendation was presented. Result: 21.6GB → 29GB free (91.2%→87%).
