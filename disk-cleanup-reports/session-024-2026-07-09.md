## Addendum (same day) — `klo` npm/pnpm/bun/node_modules follow-up

User asked to also clean any pnpm/npm/bun/node_modules found inside `klo`'s home. This resolved the item 8 mystery and closed out the account.

**Root cause of the item 8 failure:** `sudo -u klo <cmd>` does **not** reset `$HOME` on this system — it was silently running with `HOME=/Users/dedi` inherited from the caller, so pnpm was reading (and erroring against) `dedi`'s config/cache paths the whole time. Confirmed via `sudo -u klo sh -c 'echo HOME=$HOME'` → printed `/Users/dedi`. Fix: use `sudo -H -u klo <cmd>` instead, which correctly sets `HOME=/Users/klo`.

| Item | Finding | Action |
|---|---|---|
| pnpm store | With `-H` fix, `pnpm store prune` ran cleanly — **0 orphaned packages**, the 1.4GB store is fully in use | No space recovered; bug fixed for future sessions |
| npm cache | Already reduced to 1.0MB by S24's earlier `_cacache` removal | `sudo -H -u klo npm cache clean --force` run for completeness, no-op |
| bun | `~/.bun` does not exist under `klo` | Nothing to clean |
| node_modules | Found 5 small dirs (~150MB total): `.nvm/versions/node/v20.20.0/lib/node_modules` (15M), Zed editor's bundled prettier (8.3M) + json-language-server (93M), pnpm's own self-install (18M, the `9.15.4 -> 9.15.4_tmp_20722` symlink looked like debris but is the live active install), openclaw's bundled node v22.22.0 (18M) | All tool-internal, not stray project installs — left alone, deleting would break Zed/nvm/pnpm/openclaw |

**Net result:** no additional space recovered (klo was already lean after the `.ollama` removal), but the `sudo -u` vs `sudo -H -u` HOME bug is now documented so future klo-side commands don't silently misfire.

**Disk note:** between the original S24 cleanup and this addendum, free space dipped from 19GB to ~10GB (95% used) — traced to Xcode DerivedData + CoreSimulator regrowing again (~3.2GB, same grobiz mobile-build pattern flagged in the Repeat-Offender table below) plus normal fluctuation. Not a new emergency; no action taken this addendum beyond the klo checks above.

**Process note:** the standard non-sudo `du -h -d1 ~` sweep used in S1-S23 cannot see other user accounts' disk usage — permission-denied errors are silent unless explicitly checked. Going forward, an early step in any "disk almost full" investigation should be `sudo du -h -d1 /System/Volumes/Data` to catch cross-account or system-level growth before assuming it's the usual `dedi`-side repeat offenders.

**Process note (addendum):** any command run as `sudo -u klo` (or any other secondary account) must use `sudo -H -u klo` — without `-H`, `$HOME` stays whatever the calling user's shell has, which silently redirects config/cache-reading tools (pnpm, npm, etc.) at the wrong account's files instead of erroring loudly.
