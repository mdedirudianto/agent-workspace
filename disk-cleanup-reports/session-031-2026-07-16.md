# Session 31 — Xcode/iOS Simulator Crisis + Fast Gradle Regrowth

**Date:** 2026-07-16

**Disk before:** 98% used — 4.7GB free
**Disk after:** 88% used — 26GB free
**Recovered:** ~21.3GB

---

## Context

Hours after S30 (87% used / 29GB free), disk had collapsed to **98% used / 4.7GB free** — a ~24.6GB swing in a single afternoon. This was a different animal from S30's regrowth: Xcode/iOS Simulator tooling had ballooned by ~32GB combined (new iOS 26.3 runtime download, 5 freshly-created simulator devices, DerivedData), and `~/.gradle` had already fully regrown to 4.0GB just hours after being wiped in S30 — the fastest regrowth cycle seen across all 31 sessions. Given the severity (real risk of running out of disk entirely), investigation and cleanup happened in one pass rather than the usual two-step review/approve.

Root cause of the Xcode/Simulator growth is still open — no active iOS/mobile dev project is known to be in progress (see [Next session] below).

## Actions Taken

| # | Item | Command | Recovered |
|---|------|---------|-----------|
| 1 | Xcode DerivedData | `rm -rf ~/Library/Developer/Xcode/DerivedData` | 3.5GB → 0 |
| 2 | VSCode ShipIt cache | `rm -rf ~/Library/Caches/com.microsoft.VSCode.ShipIt` | 1.3GB → 0 |
| 3 | Docker installer leftover | `rm -rf ~/Library/Application\ Support/com.docker.install` | 2.1GB → 0 |
| 4 | Gradle | `rm -rf ~/.gradle` | 4.0GB → 0 |
| 5 | Maestro CLI cache | `rm -rf ~/.maestro-cli` | 643MB → 0 |
| 6 | iOS Simulator devices | `xcrun simctl erase all` | 7.0GB → 192MB |

Approved as two tiers (A: caches/gradle/maestro, B: simulator erase); Tier C (16GB runtime image) explicitly declined. Result: 4.7GB → 26GB free (98%→88%).

### Attempted but blocked

- `/Library/Developer/CoreSimulator/Caches` (3.8GB) — `rm -rf` failed with `Operation not permitted` even under `sudo` and with the harness sandbox explicitly disabled. The dyld shared cache files show no visible ACLs/flags via `ls -lO@`, consistent with SIP (System Integrity Protection) path-based enforcement rather than a fixable ownership/permission issue. Not pursued further — reclaiming it would require disabling SIP in Recovery Mode, out of proportion for 3.8GB of regenerable cache.
