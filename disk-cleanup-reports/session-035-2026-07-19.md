## Context

User flagged disk as "tight" for the second check-in in two days. Reanalysis found a real crisis this time, not routine churn: free space had collapsed from 23GB (start of S34's "no regrowth" check) to **6.2GB free / 97.5% used** on the main 245.1GB APFS container. `df` alone under-reported the severity (`Capacity` column showed a misleading 67% at one point mid-session due to how macOS reports the Data-role volume) — `diskutil apfs list` was used instead for an authoritative container-level read this session and should be preferred going forward when disk numbers look inconsistent.

New discovery: this Mac actually has **two separate APFS containers** sharing the physical disk. The main 245.1GB container holds `/Users`, `/Applications`, system volumes, etc. A second, independent **17.8GB container holds the iOS Simulator runtime disk image** (`iOS 26.3.1`, single runtime via `xcrun simctl runtime list`) and was itself at **97.4% used / 459MB free** — not discovered or tracked in any prior session. It's required for grobiz mobile-branch simulator testing, so it's not a cleanup target, but it's now a standing thing to check each session since it has no slack left either.

The dominant growth driver continues to be `grobiz/.worktrees/`: 11GB (S33) → 15GB (yesterday's analysis) → **19GB** today, now 8 active branches (two new this morning: `button-font-scale`, `playwright-nav`), all with commits in the last ~35 hours, none merged. User confirmed this is still-needed active dev work — left untouched again.
