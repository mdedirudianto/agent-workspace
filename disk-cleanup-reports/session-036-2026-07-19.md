**Disk before:** 97.2% used — 6.8GB free

## Context

Same-day follow-up to S35, which ended at 19.1GB free / 92.2% used. Within roughly two hours, three new grobiz worktrees appeared (`button-font-scale`, `fabric-gap-chips`, `playwright-nav`) — each with a fresh `node_modules` (1.5–2.9GB) — and free space collapsed back to **6.8GB free / 97.2% used**. This confirms `grobiz/.worktrees/` (flagged in S35 as "the only lever left big enough to matter") is now regrowing faster than any cache-based sweep can keep up with: 11 worktrees total, up from 8 yesterday and 7 the day before.

Rather than touching the worktrees themselves (branches/commits, all legitimate active mobile-fix work), the user asked for a narrower cut: strip `node_modules` from worktrees that haven't been touched recently, keeping the worktree/branch intact so `pnpm install` can restore it later. Selection criterion was last-commit age — over 3 hours idle, checked to the minute via `git log -1 --format=%ct` per worktree — with one explicit manual exception (`font-scale-overflow`, 18.5h idle, kept anyway per user instruction).

## Actions Taken

| # | Item | Command | Recovered |
|---|------|---------|-----------|
| 1 | `node_modules` in 7 idle worktrees (listed above) | `rm -rf .worktrees/<name>/node_modules` ×7 | ~1.2GB (host-visible) |
| 2 | pnpm store prune (reclaim now-orphaned package content) | `pnpm store prune` | 2,845 packages / 146,369 files removed, ~1.6GB |

First `rm -rf` pass timed out mid-loop at 2 minutes (killed while deleting `service-items`); re-ran the remaining single `rm -rf` to finish, then confirmed all 7 were gone before running the store prune (which also needed to run as a background task — took longer than the 2-minute inline limit against an ~11GB store).

All approved by the user with an explicit cutoff ("more than 3h ago") and one named exception (`font-scale-overflow`).
