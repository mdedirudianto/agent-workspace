# Session 2 — First per-hotel tenant instance: `technoparkmalang.hoteru.co.id`

**Date:** 2026-06-18

First real per-hotel tenancy on `hoteru.co.id`, for **Technopark Malang** (TEFA teaching hotel,
SMKN 3 Malang). Stood up as a **fully isolated, dedicated clone** — separate repo clone, backend
process, Postgres DB, Redis logical db, JWT secrets, and TLS — rather than via shared multi-tenant
routing (which still doesn't exist; see session-001). This is the brute-force "one clone per hotel"
model: the booking front-end and management SPA bake their API base + hotel id at build time.

## What drove the design

- Used the **`feat/integration-area-a-b`** branch (not `main`) — it introduces the bespoke per-hotel
  guest site `apps/web/technopark/` (`@hoteru/technopark`, Next 16, `output: standalone`) plus a
  multi-tenant backend that resolves the tenant from an **`x-hotel-id`** header. `main` has no such app.
- Both public + management apps requested (guest booking site **and** staff back-office).
- Dedicated API subdomain `api.technoparkmalang.hoteru.co.id`.
- **Seed policy (new standing rule):** only the **hotel + one ADMIN user + real room types** — never
  the shared `prisma/seed.ts` (it wipes the DB and creates a demo Jakarta hotel + dummy
  bookings/guests/payments). Wrote a minimal idempotent `prisma/seed.technopark.ts`.

## Deploy footprint (this tenant)

- **App host** `app` (`devops@10.0.0.5`), clone `~/hoteru-technopark` @ `feat/integration-area-a-b`:
  - `hoteru-tpm-api` — backend Fastify, `10.0.0.5:3120` (cwd `apps/backend`, loads `.env` via `dotenv/config`).
  - `hoteru-tpm-web` — `apps/web/technopark` Next standalone, `HOSTNAME=10.0.0.5 PORT=3121`.
  - `management` SPA — built static, served from `proxy:/var/www/hoteru-tpm-app`.
  - PM2 config `~/hoteru-technopark/hoteru-tpm.config.cjs` (named `*.config.cjs` so PM2 reads it as
    config, not a script). `pm2 save` done.
- **DB host** `db` (`10.0.0.1`): Postgres database **`hoteru_tpm`**, owner role **`hoteru_tpm`**
  (dedicated random password). **Redis logical db 9** (`redis://10.0.0.1:6379/9`). Schema via
  `prisma db push` (not migrate — matches project convention, avoids the flagged PaymentGateway
  migration hazard; clean on a fresh DB).
- **Proxy** `proxy` (`46.250.234.153`): vhost `technoparkmalang.hoteru.co.id.conf` (3 server blocks +
  `:80`→`:443`). Management SPA static at `/var/www/hoteru-tpm-app/`.
- **TLS:** one LE cert `technoparkmalang.hoteru.co.id` via **DNS-01**, SANs:
  `technoparkmalang` + `api.technoparkmalang` + `app.technoparkmalang`. Same CF token
  `/etc/letsencrypt/cloudflare/hoteru.ini`. Auto-renew via certbot timer.

## ⚠️ Cloudflare 2-level subdomain TLS gotcha

CF's free Universal SSL covers only the apex + **one-level** `*.hoteru.co.id`. Orange-proxying the
**two-level** `api.`/`app.technoparkmalang.hoteru.co.id` produced a TLS **handshake failure at the CF
edge** (origin was fine). Resolved by setting those two records to **grey-cloud (DNS-only)** so they
hit the proxy directly over our LE cert. Apex (`technoparkmalang`, one-level) stays orange.
To restore orange on the 2-level names, the zone needs **Total TLS + Advanced Certificate Manager**.

## Redeploy / ops notes

- After code change: rebuild the affected app(s). Packages must be built first
  (`pnpm build:packages`) or `@hoteru/types` won't resolve for `management`. For the Next apps,
  `rm -rf .next` before build to force `NEXT_PUBLIC_*` re-bake; then copy `.next/static` + `public`
  into `.next/standalone/apps/web/technopark/` before `pm2 restart hoteru-tpm-web`.
- Changing any `VITE_*` / `NEXT_PUBLIC_*` requires a **rebuild**, not just a restart (inlined at build).
- Management static: `rsync apps/management/dist/` → `proxy:/var/www/hoteru-tpm-app/` (drop `mockServiceWorker.js`).
