# Session 3 ŌĆö `app.hoteru.uk` login fix: production DB was never seeded

**Date:** 2026-07-16

User reported "invalid email" logging into `app.hoteru.uk` (management SPA demo) with the
documented demo credentials `admin@hoteru.com` / `admin123`.

## Root cause investigation

Followed systematic-debugging: confirmed the failure at the API layer before touching anything.

| Check | Result |
| --- | --- |
| `https://app.hoteru.uk` | 200 ŌĆö SPA loads fine |
| `https://api.hoteru.uk/health` | 200 |
| PM2 `hoteru-api` on `app` | online, 13d uptime, 0 restarts |
| `POST /api/auth/staff/login` (real route, found via `/docs/openapi.json`) | **401** `{"success":false,"error":"Invalid email or password"}` |
| `hoteru` DB on `db` ŌĆö `users`, `hotels`, and all 8 other data tables | **0 rows in every table** |

Not a build/deploy/rebuild issue ŌĆö frontend, API, and routing were all healthy. The account
genuinely didn't exist: **prod DB migrations were applied at session-001 but `db:seed` was never
run** (already flagged as an open follow-up in the session-001 report). `/api/auth/staff/create`
can't bootstrap the first admin either ŌĆö it requires an existing admin bearer token *and* an
existing `hotelId`, and there was no hotel.

Found `apps/backend/prisma/seed.ts` creates exactly one hotel ("Hoteru Grand Jakarta") + 4 users,
and its own console output names the same credentials the user was given
(`admin@hoteru.com` / `admin123`, `manager@...`, `staff@...`, `reception@...`) ŌĆö confirming this
seed is what the demo environment was always meant to run.

## Fix applied

```
cd ~/hoteru/apps/backend && pnpm db:seed
```

Seed script clears (no-op, everything was already empty) then creates: 1 hotel, 4 staff users,
5 room types, 31 rooms, 12 hotel amenities, 12 guests, 26 seasonal rates, 6 promotions,
21 bookings, 16 payments.

## Verification

| Check | Result |
| --- | --- |
| `POST /api/auth/staff/login` (admin@hoteru.com/admin123) | 200, JWT access+refresh tokens returned |
| Browser login at `https://app.hoteru.uk/login` | "Welcome back!" toast, redirected to dashboard |
| Dashboard data | 21 bookings, 12 guests, 31 rooms (29% occupancy), Rp 103.750.000 revenue ŌĆö matches seed counts |
| Screenshot | [dashboard-login-success.png](../screenshots/hoteru-session-003-2026-07-16/dashboard-login-success.png) |

`demo.hoteru.uk` (guest-facing `public` app, same DB) now also has real data to render instead of
being empty.

## Notes for next session

- This is the shared multi-tenant-model demo DB (`hoteru`), **not** a per-hotel tenant ŌĆö running
  the full `prisma/seed.ts` here was correct. The standing rule from session-002
  (never run shared `seed.ts` against a real per-hotel tenant DB like `hoteru_tpm`) still applies
  unchanged; it does not apply to this shared demo DB.
- Demo credentials are publicly shown on the login page itself (`admin@hoteru.com` / `admin123`),
  so there is no confidentiality expectation here.
