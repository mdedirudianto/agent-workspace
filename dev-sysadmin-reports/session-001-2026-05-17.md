## Findings

### Critical

| # | Finding | Detail |
|---|---------|--------|
| **C1** | **No firewall, no fail2ban, brute-force flooding** | UFW inactive. 5,748 failed SSH attempts in current log rotation. Top attacker has 1,387 tries. Nothing is rate-limiting. |
| **C2** | **`PermitRootLogin yes` + `PasswordAuthentication yes`** | Direct root via password is allowed. Combined with C1, this is the single biggest exposure on the box. |
| **C3** | **PostgreSQL bound to `0.0.0.0:5432` with wide `/20`–`/22` md5 allows** | Any host in 5 different provider subnets can attempt password auth on any DB. `listen_addresses='*'` should be `localhost` (or explicit IPs only). |
| **C4** | **Frappe gunicorn on `0.0.0.0:8000` + socketio on `9000`** | ERPNext app server is publicly reachable bypassing nginx. Should be `127.0.0.1` only. |
| **C5** | **iccn-web crash loop: 2,670,394 restarts, 2.24 GB error log** | `MODULE_NOT_FOUND: /home/deploy/iccn/apps/web/.next/standalone/apps/web/server.js`. The standalone build was deleted/never produced; PM2 has been restarting it nonstop. The 2.24 GB log alone is 1.5% of the disk. |
| **C6** | **No log rotation** | 2.5 GB total PM2 logs. Same issue as `app` session-001 (which had a 4.7 GB single log). |

## Audit Corrections (post-investigation)

Two findings from the initial audit turned out to be wrong after deeper investigation in Phase 3/4:

| Original Finding | Correction |
|------------------|------------|
| **H1 — "18 PM2 processes are dead stubs at ~2 MB RSS"** | Only 4 of 18 are truly dead (`ytgrab-be`, `ytgrab-fe`, `miaw-web`, `wagrab-team`). The other 14 use a bash-wrapper pattern: PM2 launches `/usr/bin/bash -c <script>`, the bash parent stays at ~2 MB while the real node child runs at normal memory. Verified by `pgrep -af <name>` returning live node processes for `pahala`, `pasarbesar`, `teamy`, `washop`, `wagrab-web`, `wagrab-admin`, `rekrutme`, `feyfa-admin`, `feyfa-voucher`, etc. The PM2 memory column is therefore an unreliable liveness signal on this host. |
| **H4 — "Frappe stack runs both natively (`uid 1000`) AND in docker"** | `/home/frappe` does not exist on the host. The uid-1000 gunicorn / socketio / scheduler processes seen in `ps` are the docker `grobiz-erpnext-1` / `grobiz-socketio-1` / `grobiz-scheduler-1` containers — their internal processes show up in the host's process listing because docker shares the PID namespace by default, but the paths and uids belong to the container. There is no native Frappe install to reconcile; only the docker stack exists. |
