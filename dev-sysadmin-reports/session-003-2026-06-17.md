# Session 3 ‚Äî Swap Saturation Remediation (PM2 reclaim + swap grow)

**Date:** 2026-06-17
**Server:** `root@dev` (`154.26.130.96`)
**OS:** Ubuntu 22.04.5 LTS
**Goal:** Diagnose and relieve a near-full swap / active memory-pressure condition on the dev host, without a rebuild and without disrupting recently-deployed apps.

---

## Symptom

Swap reported "at peak." Confirmed on inspection: **swap 5.2 G / 6 G used (only 815 M free)** with PSI memory pressure showing the box actively stalling ‚Äî `some avg10=22%`, `full avg10=20%`. Load had spiked (`9.85` 15-min) and was calming (`0.83` 1-min) at time of inspection. Uptime 61 days.

This was **not** just cold paged-out memory ‚Äî PSI `full ~20%` means ~1/5 of wall-clock time was lost to memory reclaim stalls.

## Part 2 ‚Äî PM2 Memory Caps (`max_memory_restart`)

To stop apps ballooning again (pasarbesar had reached 1.4 G RSS over 2 months), `max_memory_restart` caps were applied so PM2 auto-recycles a process that crosses its ceiling.

### Critical gotcha: bash-wrapped apps defeat the cap

Many apps were launched as `bash -c "npm start"` / `dotenv -e .env -- sh -c 'next start'`. PM2's `max_memory_restart` monitors **only the process PM2 directly forked** ‚Äî i.e. the bash/dotenv wrapper (~4 kb RSS) ‚Äî **not** the real `next-server` grandchild that actually holds the memory. This is why the original `pm2 list` showed pasarbesar as "4.0kb" while it truly used 1.4 G. **A cap on a wrapper-launched app never trips.**

Fix for the three proven next-server leakers: relaunch them to exec the Next binary directly so PM2 monitors the real process.
- Pass `-H`/`-p` explicitly (was supplied by the dotenv wrapper); Next.js loads `.env` itself at runtime, so app env is preserved.
- `node_modules/.bin/next` is a package-manager cmd-shim ‚Äî for npm/pnpm it can be a **shell script**, which PM2 mis-runs as JS (`SyntaxError: missing ) after argument list`). pasarbesar/pahala's shim happened to be node-executable; **rekrutme's was a shell shim and errored** ‚Üí relaunched via the real node entry `node_modules/next/dist/bin/next --interpreter node`. Use the `dist/bin/next` path for reliability.

### Caps applied

| Group | Apps | Cap | Effective? |
|-------|------|-----|------------|
| **Relaunched direct (next-server leakers)** | `pasarbesar` :7002, `rekrutme` :7007, `pahala` :7005 | 400 M | ‚úÖ now binds (monitors real next-server) |
| **Direct-exec ‚Äî Next SSR / static serve** | `iccn-web`, `hoteru-public`, `hoteru-marketing`, `ytgrab-web`, `asiaweek-web`, `seenow-web-public` | 400 M | ‚úÖ |
| **Direct-exec ‚Äî Node/Bun APIs & workers** | `grobiz`, `iccn-api`, `wabacall`, `hoteru-api`, `asiaweek-app`, `asiaweek-api`, `seenow-api`, `seenow-workers` | 300 M | ‚úÖ |
| **Direct-exec ‚Äî Vite preview** | `hoteru-management` | 250 M | ‚úÖ |
| **Pre-existing caps (unchanged)** | `miaw-api` 500 M, `miaw-web` 300 M, `eventopia-web-public`/`console` 600 M, `ytgrab-api` 800 M, `pm2-logrotate` 500 M | ‚Äî | mixed |

Result: **24 / 38 apps capped, all online, none errored.** All 3 relaunched leakers verified serving HTTP 200 on their ports, `restarts=0` (no flapping), pasarbesar monitored memory now reads its true 321 M (under the 400 M cap). `pm2 save` run ‚Üí persisted to `/home/deploy/.pm2/dump.pm2`, survives reboot.
