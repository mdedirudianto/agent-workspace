## Category source — endpoint decision

The DuitKas client base URL is **hardcoded** to `https://app.duitkas.com/api/v1/baznas` (`apps/api/src/integrations/duitkas/client.ts` `DEFAULT_CONFIG`); `index.ts` calls `createDuitKasClient(apiKey)` with the key only — no env override for the URL in this version. The DB schema carries a per-org `baseUrl`, but the runtime ignores it.

Probed both endpoints with **lpekin's** key (`GET {base}/category/map/{income,expense}`):

| endpoint | result |
|---|---|
| `/api/v1/lpekin/category/map/*` | **404** (does not exist) |
| `/api/v1/baznas/category/map/*` | **200** — returns live map (`makanan` income = **1318**, expense = **1319**, …) |

Decision (user): *try `/lpekin`, fall back to `/baznas`.* Since `/lpekin` 404s, **both tenants use `/baznas`** — which is the hardcoded default, so **no code change was needed** (the planned `DUITKAS_API_BASE_URL` env var was dropped). lpekin's key authenticates against `/baznas` and the category IDs are global, so categories resolve correctly for both.

Note: the live API IDs (`makanan` income `1318`) differ from the **stale** static fallback in `categories.ts` (`1239`) — confirming why fetching from the API matters.

## New env (set in root `.env` + `apps/api/.env` per client)

| client | `REDIS_KEY_PREFIX` | DuitKas base | Redis db |
|---|---|---|---|
| baznas | `baznas:` | `/api/v1/baznas` (default) | `10.0.0.1:6379` db0 |
| lpekin | `lpekin:` | `/api/v1/baznas` (default; `/lpekin` 404) | `10.0.0.1:6379/5` db5 |

`DUITKAS_CATEGORY_CACHE_TTL` left at default (6h). Redis DBs were already isolated by number; the prefix is the new intended isolation mechanism (belt-and-suspenders).
