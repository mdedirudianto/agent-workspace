# Session 1 — Security Audit & Hardening

**Date:** 2026-04-19
**Server:** `root@numa` → `46.250.239.44` (vmi3211902.contaboserver.net, Contabo VPS)
**OS:** Ubuntu 24.04.4 LTS, kernel 6.8.0-106-generic

---

## Audit Findings

### Critical
| # | Issue | Detail |
|---|-------|--------|
| 1 | Password auth ON for root | Config conflict: `50-cloud-init.conf` (`yes`) overrode `60-cloudimg-settings.conf` (`no`) — confirmed by `sshd -T` |
| 2 | No firewall | `ufw` inactive, no iptables rules. SSH on `0.0.0.0:22` fully exposed |
| 3 | No brute-force protection | 42,839 failed SSH attempts in 7 days, no fail2ban, no rate limiting |

### High
| # | Issue | Detail |
|---|-------|--------|
| 4 | `PermitRootLogin yes` | Direct root SSH allowed |
| 5 | 10-year-old RSA key | `rsa-key-20160704` in authorized_keys (legacy algorithm) |
| 6 | Root had active password | Brute-forceable with no lockout |
| 7 | 5 distinct source IPs logged in as root | Some unrecognized |
| 8 | `X11Forwarding yes` | No legitimate use on headless server |

### Medium
- Conflicting SSH config — cloud-init could reset it on reboot
- No non-root admin user
- `/dev/shm` missing `noexec`
- `MaxAuthTries 6` (default), `LoginGraceTime 120` (too long)

---

## Actions Taken

| # | Action | Result |
|---|--------|--------|
| 1 | UFW enabled — port 22 allowed first | `ufw allow 22/tcp && ufw --force enable` — active |
| 2 | Disabled password auth definitively | Added `PasswordAuthentication no` to main `sshd_config`; overwrote `50-cloud-init.conf`; created `/etc/cloud/cloud.cfg.d/99-disable-ssh-manage.cfg` with `ssh_pwauth: false` |
| 3 | Installed fail2ban | `bantime=86400`, `maxretry=3`, `findtime=600` — already banning, 5 IPs banned within minutes |
| 4 | Locked root password | `passwd -l root` — status `L` confirmed |
| 5 | Removed old RSA key | `rsa-key-20160704` removed from `~/.ssh/authorized_keys` |
| 6 | Disabled X11Forwarding | `X11Forwarding no` in sshd_config |
| 7 | Tightened SSH settings | `MaxAuthTries 3`, `LoginGraceTime 30` |
| 8 | Hardened `/dev/shm` | Added `tmpfs /dev/shm tmpfs rw,nosuid,nodev,noexec 0 0` to fstab; remounted |

## Incident During Step 5
The 2016 RSA key was the active key for this workstation (`dedi@mini.local`). Removing it briefly lost SSH access. Recovery: added `dedi@mini.local` ed25519 key via the live session. No interruption.
