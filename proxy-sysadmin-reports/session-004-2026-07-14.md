# Session 4 — Fail2ban Lockout: Diagnosis & Fix

**Date:** 2026-07-14
**Server:** `root@proxy` (`46.250.234.153`, internal `10.0.0.2`)
**OS:** Ubuntu 24.04.3 LTS
**Goal:** User reported `ssh root@proxy` failing from their Mac; suspected fail2ban. Diagnose and restore access.

---

## Root Cause

Two separate fail2ban `sshd`-jail bans, both triggered by single mistyped/invalid SSH connections rather than any attack:

| IP | Why banned | Evidence |
|----|-----------|----------|
| `125.166.0.127` (user's Mac) | A connection logged in as user **`dedi`** (not `root`) — 3 failed auth events (`Failed none` + 2× `Failed password`) in one connection, at `2026-07-14T10:28:51`, instantly satisfying `maxretry=3`. No `Host proxy` alias exists in the user's local `~/.ssh/config`, so this was likely a bare `ssh 46.250.234.153`/`ssh proxy` typed without `root@`, defaulting to the local Mac username. | `journalctl`/`auth.log` on proxy |
| `10.0.0.5` (`app`'s internal IP) | Self-inflicted during diagnosis: a `devops@app → root@10.0.0.2` probe with no valid key produced 2× `Failed password` + connection-closed, tripping the same jail against app's internal cluster IP. This blocked app's internal (`10.0.0.2`) path to proxy while the public-IP path (`46.250.234.153`) stayed open (different apparent source IP after routing). | `auth.log` on proxy; before/after `nc -zv 10.0.0.2 22` |

`proxy`'s `sshd` jail (`/etc/fail2ban/jail.local`) is configured aggressively: `maxretry=3`, `findtime=10m`, `bantime=24h`, and counts failed **auth events**, not distinct connections — so a single wrong-username or wrong-key SSH attempt (2-3 failed events in one client invocation) is enough for an instant 24h ban.

Working access route found during diagnosis: `root@app`'s key is authorized directly on `proxy` via its **public** IP — used as the route in (`ssh root@app` → `ssh root@46.250.234.153`) since the internal route was itself banned at the time.

## Actions Taken

| # | Action | Detail | Result |
|---|--------|--------|--------|
| 1 | **Confirmed fail2ban state** | `fail2ban-client status sshd` via the working public-IP route from `app`. Found 74 IPs currently banned fleet-wide-noise, including both `125.166.0.127` and `10.0.0.5`. | Diagnosed |
| 2 | **Unbanned both IPs** | `fail2ban-client set sshd unbanip 125.166.0.127` / `unbanip 10.0.0.5` | Done — both cleared |
| 3 | **Verified restored access** | Direct `ssh root@proxy` from user's Mac succeeded; `app`'s internal path (`10.0.0.2:22`) succeeded again | Confirmed both routes working |
| 4 | **Backed up `jail.local`** | `cp /etc/fail2ban/jail.local /etc/fail2ban/jail.local.bak-<timestamp>` | Done |
| 5 | **Added internal cluster subnet to `ignoreip`** | `/etc/fail2ban/jail.local`: `ignoreip = 127.0.0.1/8 ::1 103.119.54.100` → `... 103.119.54.100 10.0.0.0/24` | Done |
| 6 | **Reloaded fail2ban** | `fail2ban-client reload` (preserves existing bans, unlike a restart) | OK — 71 other scanner IPs remained banned after reload |

---

## Config Change

`/etc/fail2ban/jail.local` `[DEFAULT]` section:

```diff
- ignoreip = 127.0.0.1/8 ::1 103.119.54.100
+ ignoreip = 127.0.0.1/8 ::1 103.119.54.100 10.0.0.0/24
```

Rationale: `ufw` already trusts `10.0.0.0/24` wholesale (`Anywhere ALLOW IN 10.0.0.0/24`, rule #4) — fail2ban didn't have the same trust boundary, so any cluster-internal host (app/db/monitor/backup/erp) making a login mistake against proxy could ban itself out of the internal network path. This closes that gap without touching `maxretry`/`bantime`, which are doing their job against the ~70+ genuine internet scanner IPs currently banned.

---

## Follow-up / Memory

Saved to memory: valid SSH login usernames fleet-wide are only `root`, `deploy`, `devops` (never `dedi` or other); and a new working rule — stop and inform the user after 2 failed SSH login attempts rather than retrying, since retries are what trip these bans in the first place (see `feedback-ssh-stop-after-2-fails` in the memory system).
