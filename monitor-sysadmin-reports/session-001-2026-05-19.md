All inbound HTTPS comes via `proxy` (`10.0.0.2`) on the internal LAN; no public-facing service should be reachable directly on `217.15.162.56` (this was broken pre-session — see Findings).

## Actions Taken (this session)

1. **SSH key-only auth** — added `/etc/ssh/sshd_config.d/00-hardening.conf` setting `PasswordAuthentication no`, `PermitRootLogin prohibit-password`, `MaxAuthTries 3`, `X11Forwarding no`, `ClientAliveInterval 300`. Verified a fresh session before continuing. Same playbook as `backup` session-001.
2. **Pruned authorized_keys** — removed the stale `rsa-key-20160704` (2016-vintage PuTTY RSA key, unknown provenance — identical comment to the one removed on `backup`). Backup at `/root/.ssh/authorized_keys.bak.2026-05-19`. 4 keys retained: `root@prod`, `klo@mini`, `klo-cloud@numa`, `dedi@mini.local`.
3. **fail2ban installed & enabled** — was not present. `/etc/fail2ban/jail.local` with `bantime=1h`, `findtime=10m`, `maxretry=5`, `mode=aggressive`, ignoreip `10.0.0.0/22`. **5 IPs already banned within seconds of starting** (43.130.0.58, 92.118.39.235, 103.172.236.15, 45.148.10.147, 80.94.92.186) — this host was being actively brute-forced (158 failed-password attempts in the hour before hardening).
4. **🔴 Rebound all public Docker ports to `10.0.0.3` only** — every container was publishing on `0.0.0.0` (`8000`, `8888`, `3001`, `3100`, `3200`, `3333`), so all the services proxied by HTTPS-on-proxy were *also* reachable in cleartext directly on the public IP `217.15.162.56`, bypassing the proxy/Cloudflare/TLS entirely. Edited the 5 compose files in `/opt/{vaultwarden,glitchtip,uptime-kuma,uptime-kuma-mcp,openpanel/self-hosting}/docker-compose.yml` to prefix every port mapping with `10.0.0.3:`. Backups at `docker-compose.yml.bak.2026-05-19`. `docker compose up -d` to recreate each stack. Verified public IP now returns no response on those ports; HTTPS-via-proxy still 200/302/307.
5. **Bound Netdata to internal IP** — added `/etc/netdata/netdata.conf` `[web] bind to = 10.0.0.3 127.0.0.1`. Previously listened on `0.0.0.0:19999` (also publicly reachable). Reachable via `https://monitor.biji.uk` through proxy unchanged.
6. **chmod 600 on env files with secrets** — `/opt/glitchtip/.env` (Postgres password, Django SECRET_KEY), `/opt/openpanel/self-hosting/.env` (Postgres + ClickHouse passwords, cookie secret), `/opt/uptime-kuma-mcp/docker-compose.yml` (Kuma admin password in plain compose env), `/opt/vaultwarden/docker-compose.yml` (ADMIN_TOKEN in plain compose env). All previously 644.
7. **Stopped & disabled unused services** — `ModemManager`, `wpa_supplicant`. Both masked.
8. **Reclaimed ~4 GB `/var/log`** — `journalctl --vacuum-time=14d` freed 3.8 GB (4.1 GB of journal). Truncated `/var/log/btmp` (42 MB) and removed `/var/log/btmp.1` (130 MB) — pure brute-force login noise. With key-only SSH + fail2ban these won't regrow at the same rate.
9. **apt full-upgrade + reboot** — 87 pending updates including kernel `6.8.0-100` → `6.8.0-117`. `/var/run/reboot-required` was set (last boot 83 days ago). Upgrade + reboot performed end-of-session. Post-reboot verification: kernel `6.8.0-117-generic`, all 8 containers up, all ports bound to `10.0.0.3` only, all 4 HTTPS vhosts (`errors`, `analytics`, `status`, `vault`) return 200/302, public IP probes on `8000/8888/3001/3100/3200/3333/19999` all return `000`.

## Docker Port Binding (the headline fix)

Before this session, every container in `/opt/*/docker-compose.yml` used short-form port mappings (`"8000:8000"`, `"3001:3001"`, etc.), which Docker publishes on `0.0.0.0`. Result: the entire monitoring stack — including unauthenticated Netdata and the Vaultwarden admin panel — was reachable in plain HTTP directly on the public IP, bypassing the proxy's HTTPS termination *and* Cloudflare's WAF entirely.

After this session every mapping is prefixed with `10.0.0.3:`. Sample verification:

```text
# before — public IP responds directly
$ curl -sI http://217.15.162.56:8000  → HTTP/1.1 200 OK (glitchtip)
$ curl -sI http://217.15.162.56:8888  → HTTP/1.1 200 OK (vaultwarden, Rocket server header)
$ curl -sI http://217.15.162.56:3001  → HTTP/1.1 302 /dashboard (uptime-kuma)
$ curl -sI http://217.15.162.56:3100  → HTTP/1.1 307 /login (openpanel dashboard)
$ curl -sI http://217.15.162.56:3200  → HTTP/1.1 404 (kuma-mcp)
$ curl -sI http://217.15.162.56:3333  → openpanel api responds

# after — same probes
$ for p in 8000 8888 3001 3100 3200 3333; do
    curl -m2 -sI -o/dev/null -w "%{http_code}\n" http://217.15.162.56:$p
  done
000  # all six no response

# proxy paths still 200/307
https://errors.biji.uk     → 200
https://analytics.biji.uk  → 307 /login (Cloudflare-fronted)
https://status.biji.uk     → 302
https://vault.biji.uk      → 200
```
