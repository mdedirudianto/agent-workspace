# Session 9 — Uptime Kuma update (2.1.3 → 2.4.0)

**Date:** 2026-06-22
**Server:** `root@monitor` (`217.15.162.56` public, `10.0.0.3` internal)
**Goal:** Apply the in-app update Uptime Kuma was prompting for. Routine version bump.

---

## Context

Uptime Kuma runs as a Docker Compose stack at `/opt/uptime-kuma`, image `louislam/uptime-kuma:2` (floating major tag), data on the named volume `uptime-kuma_uptime-kuma-data`. Public at `status.biji.uk` via `proxy`; internal bind `10.0.0.3:3001`. A second container `uptime-kuma-mcp` (`davidfuchs/mcp-uptime-kuma`) sits alongside — left untouched.

Running app version was **2.1.3**; latest stable on the `2` tag is **2.4.0** (2.2.0 → 2.4.0 in between; all non-prerelease).

---

## Actions Taken

| # | Action | Detail | Result |
|---|--------|--------|--------|
| 1 | **Backed up data volume** | `docker run --rm -v uptime-kuma_uptime-kuma-data:/data -v /root/kuma-backups:/backup alpine tar czf …` → `/root/kuma-backups/uptime-kuma-data-2026-06-22.tgz` | 43M ✓ |
| 2 | **Pulled new image** | `cd /opt/uptime-kuma && docker compose pull` — `:2` tag now resolves to 2.4.0 | Done ✓ |
| 3 | **Recreated container** | `docker compose up -d` — container recreated, named volume carried over (no data migration) | Done ✓ |
| 4 | **Verified version** | `docker exec uptime-kuma cat /app/package.json` → `"version": "2.4.0"` | 2.4.0 ✓ |
| 5 | **Verified health** | After ~40s warmup: `Up (healthy)`; HTTP 302 on `http://10.0.0.3:3001/` and `https://status.biji.uk/` (normal redirect to dashboard) | Healthy ✓ |

---

## Notes

- The `:2` floating tag means future patch/minor 2.x releases are a plain `docker compose pull && up -d` — no compose edit needed. Major-version jumps (3.x) would require changing the tag.
- SQLite + all monitor/notification config persists in the named volume; recreating the container is non-destructive. The pre-update tarball is the rollback path (`docker run --rm -v …:/data -v /root/kuma-backups:/backup alpine tar xzf …` into a stopped container's volume).
- `tar` warned `./run/mariadb.sock: socket ignored` during backup — expected (a live socket, not data); harmless.

`docker ps` shows `uptime-kuma  Up (healthy)` on 2.4.0; both internal and public URLs return 302 to the dashboard. All existing monitors/notifications intact (volume unchanged).
