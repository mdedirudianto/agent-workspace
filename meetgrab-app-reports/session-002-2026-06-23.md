## Goal

Redeploy meetgrab (web/api/etc.) to production from `main`. Analyze first, then deploy with approval. Check for migrations / backfill / env changes.

## Key analysis findings

| Finding | Implication |
| --- | --- |
| Prod was already at **`99a6d1e`** (not the session-001 `6e2cfb5` the report/memory documented — a web GA4 fix had been deployed since without a report) | Real gap to redeploy is `99a6d1e..main`, **smaller than it looked** |
| The only **server-deployed** code changes `99a6d1e..main` are in **`api`** (v0.4.1→**0.4.2**) + its build deps `shared-types`/`ui` | **api-only redeploy** |
| `web` (Astro static) has **no source change** since prod — the sole web change (GA4 `set:html` fix `99a6d1e`) is already live | **web skipped** (rebuild would be a no-op; user confirmed skip) |
| All other `main` commits are **extension** (speakers table, attribution footer, caption fixes) → Chrome Web Store, separate release path | **out of scope** for a server deploy |
| API is a **stateless LLM proxy** — no Postgres/Redis/Mongo | **no migrations, no backfill** |
| v0.4.2 reads two env keys: new `SUMMARY_MAX_OUTPUT_TOKENS` (default 16000) + `TRANSCRIPT_TOKEN_LIMIT` semantics changed (`0` = auto-derive from model context window; prod had hard `60000`) | **env update required** to realize the feature |

## Decisions

- **`TRANSCRIPT_TOKEN_LIMIT=0`** (auto-derive) — chosen over keeping `60000`. The whole point of `1128c20` is to size the transcript cap from the *configured* model's context window (Haiku 4.5 = 200K input ≈ ~14h meeting); a hard `60000` (~5h) would defeat it.
- **Skip web redeploy** — already current at `99a6d1e`.
