## Incident: SSH to `proxy` refused mid-session

An early `ssh proxy` attempt (bare, no explicit user) defaulted to the local shell user and got a `Permission denied`. Shortly after, all SSH to `proxy` (port 22) started returning `Connection refused` from this machine — HTTPS (443) stayed reachable throughout, and `app`'s SSH was unaffected, pointing to **fail2ban banning the IP** after the failed-auth attempt. `devops@app` has no valid credentials for `root@proxy` either (confirmed `Permission denied (publickey,password)`), so there was no relay-through-app workaround — this matches the existing convention that app→proxy isn't a trusted path; local-machine→proxy is the only route for the static-file relay. Paused and asked the user to clear the ban out-of-band (Contabo console); access was restored and the deploy resumed cleanly. **Going forward: always pass an explicit user in SSH commands to fleet servers (`root@proxy`, not bare `proxy`) to avoid tripping fail2ban.**
