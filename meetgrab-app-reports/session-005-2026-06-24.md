## Goal

Redeploy the latest `main` API to production. Unlike sessions 2–4 (small env tweaks on an unchanged contract), this catch-up crossed a **breaking API rewrite**: the summary endpoint went from **synchronous** to an **async enqueue/poll** model.

## Why this was urgent, not routine

| | Old (live `5d14025`) | New (`91309a1`) |
| --- | --- | --- |
| `POST /v1/summaries` | blocks, returns the summary in the body | **`202 {jobId, state:"queued", position}`** — enqueues to an in-memory, process-local queue (`apps/api/src/queue/summary-queue.ts`) |
| Get result | n/a | **`GET /v1/summaries/:jobId`** poll → `processing` → `done` with `result.summary` |

The **async-capable extension (v1.0.x) is already live on the Chrome Web Store** (confirmed by user). So before this deploy the published extension was POST-ing and polling against an API that still answered synchronously — a **live client↔server mismatch**. Deploying the async API **closes that break**; it doesn't introduce one.

Also shipped in this range: adaptive model escalation (Haiku→Sonnet `claude-sonnet-4-6` for oversized transcripts, TRD §8.6), transcript phantom-dedup, slow-loris receipt timeouts (`server-timeouts.ts`), and OS-notification/queue-position client UX.

## Pre-deploy analysis (no surprises at boot)

- **No new *required* env vars.** `apps/api/src/env.ts` `REQUIRED` is unchanged (`GOOGLE_AI_API_KEY`, `GLITCHTIP_DSN_API`, `OPENPANEL_CLIENT_ID_API`, `OPENPANEL_SECRET_API`, `IP_HASH_SALT`) — all present. New knobs (`LLM_LARGE`→`anthropic:claude-sonnet-4-6`, `SUMMARY_QUEUE_MAX_DEPTH`=20, `SUMMARY_JOB_RESULT_TTL_MS`=300000, `ANTHROPIC_SONNET_COST_*`) all carry safe defaults in `loadEnv()`. Deployed **as-is on the existing OAuth auth** (user decision) — new vars ride defaults.
- **Queue is process-local — no Redis/DB** (matches the stateless no-DB design). A restart drops in-flight jobs; the client re-enqueues.
- **nginx needs no change.** session-004 already moved the `location = /v1/summaries` 100s block into the 443 server. Async POST returns instantly; fast `GET /:jobId` polls match `location /` (60s, ample). The timeout ladder is no longer the bottleneck.
