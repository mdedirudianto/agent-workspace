## Two pre-existing gaps found & fixed (admin panel was 500 / login impossible)

### Gap 1 — web process had no server-side secrets → `/admin` 500
Hitting `/admin*` threw `DATABASE_URL is not set` / `BETTER_AUTH_SECRET is not set`. The admin pages are **server components** that query the DB and run Better Auth server-side, but `apps/web/.env` held only `NEXT_PUBLIC_*`. Non-admin pages never needed runtime secrets, so this stayed hidden since session-009.

**Root cause:** the Next **standalone** `server.js` does **not** load `.env` at runtime (no `@next/env`/`loadEnvConfig`; it only `process.chdir(__dirname)`). It was launched as plain `node server.js`, so only the build-time-inlined `NEXT_PUBLIC_*` were available.

**Fix:** added the 5 server secrets to `apps/web/.env` (`DATABASE_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `RESEND_API_KEY`, `ADMIN_EMAIL_FROM`) and relaunched the web process with Node 22's native **`--env-file`** (keeps `node` the direct process so PM2's mem cap works — no dotenv wrapper needed):
```
pm2 start /usr/bin/node --name ytgrab-web --cwd ~/ytgrab/apps/web --max-memory-restart 500M \
  -- --env-file=/home/devops/ytgrab/apps/web/.env .next/standalone/apps/web/server.js
```
**Sub-gotcha:** recreating the PM2 entry dropped the old `PORT`/`HOSTNAME` env → the server bound `0.0.0.0:3000` instead of `10.0.0.5:3009` (would 502 the public site). Added `HOSTNAME=10.0.0.5` + `PORT=3009` to `apps/web/.env` too. `pm2 save`.

### Gap 2 — nginx sent Better Auth endpoints to the API backend → magic-link 404
The auth-client uses same-origin, so the browser calls `youtubegrab.com/api/auth/*`. The proxy vhost had `location /api/ → ytgrab_api (:7009)`, which swallowed `/api/auth/*` — but those are served by the **web** app (`:3009`). Result: `/api/auth/get-session` and `sign-in/magic-link` → 404; login impossible.

**Fix (proxy `ytgrab-web.conf`):** added a longer-prefix location (wins over `/api/`):
```nginx
location /api/auth/ {
    proxy_pass http://ytgrab_web;
    proxy_read_timeout 120s;
}
```

**⚠️ nginx gotcha on `proxy` — two separate bugs hit here:**
1. `sites-enabled/ytgrab-web.conf` is a **real copy, not a symlink** to `sites-available`, and the two had **diverged** (enabled had a `location = /api/metrics` ACL block the available copy lacked → available was stale). Editing `sites-available` did nothing. **Always edit the `sites-enabled/` file on this host** (or `nginx -T | grep` to confirm what's actually loaded). Re-synced `sites-available := sites-enabled` after the fix.
2. `nginx.conf` includes `sites-enabled/*` (glob), so a `*.bak` saved **inside** `sites-enabled/` gets loaded → `duplicate upstream` emerg, reload fails. **Never leave backups in `sites-enabled/`** — backups now go to `/root/nginx-bak/`.
