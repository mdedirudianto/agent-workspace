# Session 18 — CPU analysis + migration plan: ytgrab-api LB across numa + db; web stays on app

**Date:** 2026-06-25
**Status:** Phase 2 complete — ytgrab-api LB live across numa (10.0.0.7, weight=2) + db (10.0.0.1, weight=1, nice -n 10)

---

## Trigger

App server (`10.0.0.5`) at sustained 100% CPU (confirmed via Netdata 60-minute average). ytgrab-api PM2 snapshot showed 100% CPU; load average 3.78 on 6 CPUs. User requested analysis and migration plan.

---

## Findings

### App server — CPU saturated

| Metric | Value |
|--------|-------|
| CPUs | 6 (AMD EPYC) |
| Load avg | 3.78 / 2.24 / 2.07 |
| CPU idle (Netdata, last 60 min) | **0% — fully saturated** |
| PM2 processes | 38 |
| ytgrab-api CPU (PM2) | **100%** |
| seenow-workers CPU (PM2) | 66.7% |
| Swap used | 1.8GB / 4GB |
| ytgrab-api restarts | 31 in 16h (network proxy errors, not OOM crashes) |

Root cause: app server is running too many workloads simultaneously. ytgrab-api's yt-dlp subprocess extraction is CPU-bound per request (spawns a new child process for each extraction), which spikes under concurrent traffic. bgutil-pot (PO token provider, 706MB) adds another process.

Error log review: restarts are caused by caught network errors (`UND_ERR_SOCKET` from proxy-side connection resets) and yt-dlp bot-detection failures — not process crashes. API remains functional.

### All-server capacity snapshot

| Server | CPUs | RAM | CPU idle | Internal IP | Notes |
|--------|------|-----|----------|-------------|-------|
| app | 6 | 12GB | **0%** 🔴 | 10.0.0.5 | Source — saturated |
| db | 12 | 47GB | 97.5% | 10.0.0.1 | Lots of headroom but critical DB server |

| proxy | 4 | 6GB | ~75% | 10.0.0.2 | nginx only, not for app workloads |

**Key discovery:** Numa IS on the internal cluster network at `10.0.0.7`. Ping from proxy confirmed: 1.3ms avg RTT. This means ytgrab on numa can reach `ytgrab_prod` PostgreSQL on db at `10.0.0.1` via internal network — no public IP exposure needed.

## Decision: Load balance ytgrab-api across numa (10.0.0.7) + db (10.0.0.1)

### Why both numa AND db (load balanced)

Running ytgrab on both servers is better than either alone:

| Factor | Numa only | DB only | **Both (LB)** |
|--------|-----------|---------|---------------|
| Total CPUs available | 4 | 12 | **16** |
| Redundancy | None | None | **Yes — 1 down, other serves** |
| DB isolation risk | ✅ safe | ⚠️ ytgrab spikes compete with PG | ✅ mitigated by `least_conn` + `nice` |
| Traffic capacity | Limited | High | **Best** |

**Key insight — `least_conn` protects db naturally:** When PostgreSQL is handling queries, db has more active connections → nginx automatically routes more traffic to numa. ytgrab-api on db also runs with `nice -n 10` so yt-dlp subprocesses are lower priority than database processes. This gives access to db's 12 CPUs during traffic bursts without risking DB query starvation.

### What runs where

| Process | App (10.0.0.5) | Numa (10.0.0.7) | DB (10.0.0.1) |
|---------|:--------------:|:---------------:|:--------------:|
| ytgrab-web (:3009) | ✅ **stays, untouched** | — | — |
| ytgrab-api (:7009) | ❌ move off | ✅ primary (nice 0) | ✅ secondary (nice -n 10) |
| bgutil-pot (127.0.0.1:4416) | ❌ remove after | ✅ required | ✅ required |

**Why web stays on app:** ytgrab-web is 0% CPU at 314MB — it is not contributing to the CPU problem at all. All the heaviness is yt-dlp subprocesses in ytgrab-api. Keeping web on app means zero migration risk for the frontend, no Next.js build on numa/db, and ytgrab_web nginx upstream needs no change. The web calls the API via the public browser-side URL (`api.youtubegrab.com`) — web and api don't need to be on the same host.

bgutil-pot must run locally on each ytgrab-api node — it communicates on `127.0.0.1:4416`.

### Memory fit — what moves to numa

| Process | Mem (current on app) | Notes |
|---------|---------------------|-------|
| ytgrab-api | 274MB | Hono API + proxy pool — **moves** |
| bgutil-pot | 706MB | PO token provider — **moves** |
| ytgrab-web | 314MB | Next.js SSR — **stays on app** |
| **Moving total** | **~980MB** | 7GB free on numa — 7× headroom |

### Network architecture after migration

```
Internet → CF → proxy (10.0.0.2)
                  ├── api.youtubegrab.com → upstream ytgrab_api (least_conn)
                  │                               ├── 10.0.0.7:7009  (numa, weight=2, nice 0)
                  │                               └── 10.0.0.1:7009  (db,   weight=1, nice -n 10)
                  └── youtubegrab.com     → upstream ytgrab_web  ← NO CHANGE
                                                  └── 10.0.0.5:3009  (app — stays here)

app (10.0.0.5):
  └── ytgrab-web   (PM2, :3009) — unchanged, still on app

numa (10.0.0.7):
  ├── ytgrab-api   (PM2, :7009, nice 0)  → ytgrab_prod on db (10.0.0.1:5432)
  └── bgutil-pot   (PM2, 127.0.0.1:4416)

db (10.0.0.1):
  ├── ytgrab-api   (PM2, :7009, nice -n 10) → ytgrab_prod on localhost:5432
  └── bgutil-pot   (PM2, 127.0.0.1:4416)

Outbound egress: Webshare residential + datacenter proxies (yt-dlp + undici, from both api nodes)
```

Proxy nginx config (final state — only ytgrab_api changes):
```nginx
# ytgrab_web upstream — NO CHANGE, web stays on app
upstream ytgrab_web {
    server 10.0.0.5:3009;  # unchanged
}

# ytgrab_api upstream — THIS is what changes
upstream ytgrab_api {
    least_conn;
    server 10.0.0.7:7009 weight=2;  # numa — primary, isolated box
    server 10.0.0.1:7009 weight=1;  # db — secondary, nice -n 10 on server
}
```

## Future: Further scaling

Once numa + db are both serving ytgrab and traffic keeps growing, the next lever is a **dedicated CPU-optimized VPS** added to the upstream:

```nginx
upstream ytgrab_api {
    least_conn;
    server 10.0.0.7:7009 weight=2;   # numa
    server 10.0.0.1:7009 weight=1;   # db
    server <new-vps>:7009 weight=3;  # dedicated — highest weight
}
```

**Trigger:** When Netdata shows numa sustaining >70% CPU and db's ytgrab processes begin competing with PostgreSQL query latency.

**VPS recommendation:** Hetzner CCX12 (2 vCPU dedicated) or CCX22 (4 vCPU) — high single-thread clock speed matters more than core count for yt-dlp (one subprocess per extraction).
