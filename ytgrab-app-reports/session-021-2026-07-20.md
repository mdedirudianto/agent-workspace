# Session 21 — Rotate `RESEND_API_KEY` fleet-wide (admin magic-link login broken)

**Date:** 2026-07-20
**Servers:** `devops@app` (10.0.0.5), `root@numa` (10.0.0.7), `root@db` (10.0.0.1)
**Trigger:** User reported `youtubegrab.com/admin` login failing with "Failed to send link." The Resend account key had been rotated but never propagated to the fleet.
**Result:** ✅ all 4 `.env` copies updated; `ytgrab-web` (app) + `ytgrab-api` (numa, db) restarted; magic-link send verified `{"status":true}`

---

## Root cause

Admin login is Better Auth **magic-link**, email sent via Resend from `packages/auth/src/email.ts` (shared by both `web` and `api`, since both instantiate `@ytgrab/auth`). All four `.env` files across the fleet still held the **old** key (`re_KkQy1vXW_MZS43GeHG5PjtjUEYY6ifVpi`); the account's key had been rotated to `re_d9SEUdpU_3W5aN2EPuQqWsLHybBXzbioy` but nothing on the servers was updated. In production, `/api/auth/*` routes to **web** (session-010 nginx fix), so `apps/web/.env` on `app` was the one actually causing the failure; `apps/api/.env` copies (app/numa/db) were updated too for fleet consistency since the shared auth package reads the same var.

No rebuild needed: it's a runtime secret loaded via Node's native `--env-file` flag at process start (`--env-file=.../apps/web/.env .next/standalone/.../server.js` on web; `--env-file .../apps/api/.env dist/index.js` on api) — not a `NEXT_PUBLIC_*` build-time var. A `pm2 restart --update-env` re-reads the file.
