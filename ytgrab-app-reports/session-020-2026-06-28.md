## Notes / follow-ups

- **Migration 0007 in prod:** `download_events.client_id` column live. Existing rows have `client_id = NULL` (no backfill — by design; audience analytics will populate from this deploy forward). Index `(client_id, created_at DESC) WHERE client_id IS NOT NULL` active.
- **Audience + Retention tabs live:** Admin panel now shows client cohort engagement and weekly retention grid. Data accumulates from this deploy; expect meaningful signal after a few days of traffic.
- **Readiness probe fix:** `GET /api/v1/status` no longer inflates the merged-download counter. Download stats are clean from this deploy.
- **Phase 3 cleanup confirmed complete:** `ytgrab-api` and `bgutil-pot` were already removed from PM2 on `app` during session-018 (LB migration). No action needed.
- **Web-on-proxy decision:** Considered moving `ytgrab-web` to static files on proxy (meetgrab pattern). Rejected — ytgrab-web has mandatory SSR (`/watch`, `/shorts/[id]`, `/admin/*`, `/api/auth/*`) unlike meetgrab-web (Astro, fully static). Keeping Next.js standalone on app is correct; app CPU is now free with ytgrab-api on numa+db. Static asset split (`/_next/static/` on proxy) adds deploy complexity for marginal gain (CF already caches hashed assets at edge).
- **Rollback:** `git reset --hard e522632` on numa + db + app; rebuild; restart. Migration 0007 leaves `client_id` column in place but 1.5.0 ignores it — no DB rollback needed.
- **Canary still on numa** (`/root/ytgrab-download-canary.sh`, API=127.0.0.1:7009) — unchanged.
