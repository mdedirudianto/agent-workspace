## Context — the trigger

The admin **Proxies** page showed **"0 active" residential** and clicking "Refresh from Webshare" didn't pick up a freshly-replaced residential plan. Root cause (diagnosed this session, see *Pre-incident* below): the v1.1.0–1.1.2 pool sourced proxies from static Webshare **"Download list" URLs** carrying a `?plan_id=NNN`. Replacing a plan on Webshare mints a **new plan_id**; the old one returns `404 {"detail":"Plan not found"}`, so refresh re-pulled a dead URL forever and the stale rows churned to disabled/recycled → "0 active". **Fix is structural and shipped in 1.1.3**, hence this redeploy.

## What 1.1.3 changes

Replaces the two **"Download list" URL** env vars (`WEBSHARE_PROXY_LIST_URL` / `WEBSHARE_RESIDENTIAL_LIST_URL`, **both removed**) with the **Webshare REST API** keyed by a single **`WEBSHARE_API_KEY`** (`Authorization: Token <key>`). Plans are **auto-discovered** (`GET /api/v2/subscription/plan/`; static **ISP** subtype → residential, **datacenter** → metadata; rotating **backbone** excluded), listed via `mode=direct` on boot + admin refresh, and **dead proxies are recycled via `POST /api/v3/proxy/replace/`** for a fresh static IP. The killed IP is kept as a **denylist** row (`recycled_at`) so a replacement never reuses it, pruned after `PROXY_RECYCLED_RETENTION_MS` (default 7d). `429`/exhausted-quota reported to GlitchTip with a per-kind ~1h backoff. **This self-heals plan swaps — no more stale plan_id.**

## Key finding — auto-discovery is clean, no plan pinning needed

Account has 5 plans, but only two are `status=active`:

| plan_id | status | subtype | proxy_count | → kind |
|---|---|---|---|---|
| `13653319` | **active** | isp | 5 | residential ✓ |
| `13641117` | **active** | default | 100 | datacenter ✓ |
| `13641118` | cancelled | isp | 5 | (old dead residential — filtered out) |
| `13402231` | cancelled | default | 100 | filtered out |
| `13400364` | cancelled | free | 10 | filtered out |

`discoverPlans` filters `status==="active"` then picks largest `proxy_count` per kind → exactly one plan per kind, no `multiple active webshare plans` warning. The dead `13641118` (the one that caused the incident) is `cancelled` and ignored. Pinning `WEBSHARE_*_PLAN_ID` would reintroduce the staleness it solves, so left unset.
