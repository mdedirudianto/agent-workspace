## Incident found + fixed: residential proxy pool exhausted (not a regression)

First real-download canary returned **502 `no healthy residential proxy`** (`pooled-download.ts:90`). IP-locked downloads pin one **residential** proxy; metadata uses datacenter so `process` still worked. DB state:

- datacenter: 32 active / 67 disabled / 1 recycled (healthy enough)
- **residential: 0 active / 5 disabled** — every residential proxy had a single failure overnight (`last_failed_at` 2026-06-21 23:26 → 2026-06-22 00:39), inside the 1-day `PROXY_DISABLE_COOLDOWN_MS`, so none had re-activated.

This is the **small-pool fragility flagged in session-012** (only 5 residential, auto-disable on a single strike), **not caused by this web-focused deploy**. Fix:

```sql
update proxies set status='active', fail_count=0 where kind='residential' and status='disabled';  -- UPDATE 5
```

then `pm2 restart ytgrab-api` to reload the in-memory selection pool (`proxy pool refreshed {datacenter:100, residential:5, loaded:105}`). Re-test:

- `process` → 200, **download → 206 / 262144 B**. ✓
- Pool after test: residential 4 active / 1 disabled (normal lifecycle — one struck during the pull).
