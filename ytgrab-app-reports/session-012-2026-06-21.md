# Session 12 — DB-backed proxy pool (extraction + IP-locked downloads) + admin page; web/api → 1.1.0

**Date:** 2026-06-21
**Scope:** Built a **proxy pool** for ytgrab so yt-dlp egress draws from the full Webshare proxy lists (100 datacenter + 5 residential) instead of a single env proxy — with a health lifecycle, usage counting, and an admin page. Developed entirely on a **local feature branch** (`feat/proxy-pool`, 8 commits) per request, then merged `--no-ff` to `main` (`e733504`) and deployed to `app`. Surfaces bumped **web + api → 1.1.0**.

---

## Why

Session-011 left downloads working via a single rotating residential proxy + IP-match retry — functional but non-deterministic (small/dirty pool). The user has Webshare lists of **static, individually-addressable** proxies (each `ip:port:user:pass`). Using one static proxy for **both** the stream-URL resolve **and** the byte fetch makes googlevideo's IP-lock deterministic (resolve IP == download IP), and a pool gives rotation + failover + dead-proxy management.

## Architecture

- **Lists in the DB (source of truth + cache), no files.** On boot (and on admin demand) the api downloads both Webshare "Download list" URLs (`WEBSHARE_PROXY_LIST_URL` datacenter, `WEBSHARE_RESIDENTIAL_LIST_URL` residential) → upserts into a `proxies` table → loads an in-memory selection pool. URLs (which carry the API token) live in `apps/api/.env`.
- **Metadata** (`process`, no IP-lock) → cheap **datacenter** pool first, **residential** fallback.
- **Downloads** (IP-locked) → **one residential** proxy pins resolve + byte fetch + ffmpeg merge (`-http_proxy`); datacenter can't (flagged for extraction). Fail over to another residential proxy on error.
- **Health lifecycle** per proxy: failure → `disabled` (re-tried after `PROXY_DISABLE_COOLDOWN_MS`, default 1 day) → 2nd failure → `recycled` (dead); a success clears strikes. `used_count` bumped per pick.
- Falls back to the legacy single `YTDLP_PROXY` when the pool is empty.
- **Admin** `/admin/proxies`: per-kind active/disabled/recycled + total uses, full table (host:port, status, used, fails, last ok/failed), and a **"Refresh from Webshare"** button (POST `/api/v1/admin/proxies:refresh` → the api process re-pulls + reloads its pool).

### Self-review fixes (`f5098ee`)
1. **Cache poisoning** — `process-video` cached IP-locked stream URLs when the pool was active (not just on `YTDLP_PROXY_PRIMARY`); a datacenter-locked URL could poison a download → now skipped whenever the pool is active.
2. **Wrong-proxy blame** — first download of an un-processed video ran metadata extraction *inside* the residential loop, so a metadata bot-challenge wrongly disabled/recycled residential download proxies. Now metadata is ensured **before** the loop.
3. **ProxyAgent churn** — `fetchViaProxy` made (and leaked) a new undici `ProxyAgent` per download; now one keep-alive agent is cached per proxy URL.
+ **pool unit tests** (no DB): selection is kind-scoped + credentialed URL, `used_count` bump, disabled→recycled, success-clears, cooldown re-activation.
