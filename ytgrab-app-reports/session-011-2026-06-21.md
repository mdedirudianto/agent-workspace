# Session 11 — Fix download failures: YouTube bot-challenge → PO-token provider + proxied downloads

**Date:** 2026-06-21
**Scope:** User reported downloads failing ("The server is having trouble right now") on web + extension. Diagnosed as a **YouTube anti-bot challenge** (not the session-010 deploy). Fixed by: updating yt-dlp, standing up a **bgutil PO-token provider** (no-cookies), routing egress through a **residential proxy**, and a **code change** (committed `d61deb6`) so the IP-locked byte download also goes through the proxy. Downloads work again (measured 10/10).

---

## Diagnosis — NOT the deploy

`POST /api/v1/videos:process` returned **503 `BotChallengeError`**; yt-dlp stderr: `Sign in to confirm you're not a bot`. The web/extension render this as "server is having trouble."

Proof it wasn't session-010's nginx/admin work: a **cached** video downloaded fine through the full public path (`youtubegrab.com` → CF → proxy → app → **206 / 262144 B**). The `/admin` change only touched `/api/auth/` routing; `/api/v1/*` extraction was untouched. The failure is at the `yt-dlp → YouTube` step.

**Root cause (two layers):**
1. **No PO token.** yt-dlp debug: `[pot] PO Token Providers: none`. In 2026 YouTube demands a Proof-of-Origin token; without one it challenges. yt-dlp was also ~3 months stale (`2026.03.17`).
2. **IP reputation.** The app's bare datacenter IP (`154.26.129.104`) is heavily flagged — ~80% of fresh extractions 503'd. Updating yt-dlp alone (`→2026.06.09`) did **not** help.

## What was done

### 1. yt-dlp updated
`/opt/ytdlp-venv` (pip venv): `2026.03.17 → 2026.06.09` (`sudo /opt/ytdlp-venv/bin/pip install -U yt-dlp`).

### 2. bgutil PO-token provider (no-cookies fix; **no Docker** per user pref)
Docker daemon is `inactive`/`disabled` on `app` and the user wants Docker avoided — so the provider runs as a **Node service under PM2**:
- `git clone Brainicism/bgutil-ytdlp-pot-provider ~/bgutil-pot`, `cd server && npm install && npx tsc`
- `pm2 start /usr/bin/node --name bgutil-pot --cwd ~/bgutil-pot/server -- build/main.js --port 4416` (`pm2 save`)
- yt-dlp plugin into the venv: `sudo /opt/ytdlp-venv/bin/pip install bgutil-ytdlp-pot-provider` → auto-loads, uses `http://127.0.0.1:4416` (provider `bgutil:http-1.3.1`).
- **Security:** provider binds `0.0.0.0:4416` (no localhost option until its next major). **UFW (active, default-deny) does not allow 4416**, verified externally blocked (`ext=000` from proxy). Internal `10.0.0.0/24` can reach it (trusted).

### 3. Code change — route IP-locked downloads through the proxy (`d61deb6`, on `main`)
googlevideo URLs are **IP-locked to the exit IP that resolved them**. yt-dlp resolves through `YTDLP_PROXY`, but the byte download (undici global `fetch`) + ffmpeg merge inputs egressed from the **app IP** → googlevideo **403** whenever a proxy was used. Fix:
- Added **`undici` pinned `^6.23.0`** — must match Node 22.22's built-in undici (`6.23.0`); `undici@8` `ProxyAgent` threw `invalid onRequestStart method` against the built-in `fetch`.
- New `apps/api/src/downloads/proxy-dispatcher.ts` → `fetchStreamWithProxyRetry()`: routes the download via an undici `ProxyAgent` and **retries with a fresh tunnel (new exit IP) until upstream is non-4xx** — needed because a rotating proxy gives a different IP per connection vs the resolver IP. No-op (single direct fetch) when `YTDLP_PROXY` unset.
- `merge/ffmpeg-runner.ts`: `-http_proxy` before each `-i` so merged downloads egress through the same proxy.
- Committed directly to `main` (`d61deb6`), pushed (husky lint hook passed). `GIT_SHA` env → `d61deb6`.

## Known limitations (need a better proxy for 100%)

1. **Reliability tail.** With the small/dirty rotating pool, an occasional download still 502s when the resolver IP is rare and 8 retries miss it (~90–100% in practice). A **dedicated clean static residential IP** (or sticky sessions) would make resolve+download share one IP → 100%, and let us drop the retry. The current static residential (`82.23.57.196`) is burned; user should **refresh it** in Webshare.
2. **Format inconsistency.** Some extractions return only **muxed itag 18 (360p)**; full DASH/HD (35 formats) appears on other attempts — a side effect of the rotating-IP + yt-dlp player-client lottery (often "android vr" client). A stable clean IP would normalize this.
3. Merged/HD downloads (ffmpeg `-http_proxy`) share the IP-lock and have no retry — less reliable than muxed until (1) is resolved.

## Addendum — proxy split + self-heal (`0a8f645`, same session)

Per user request, tested + implemented a **two-proxy split** (cost optimization). Findings (all with PO):

| Proxy | Extraction | Download (IP-locked) | Pool |
|---|---|---|---|
| Datacenter static `172.121.235.54` | 0/6 | n/a | flagged |
| Datacenter rotating `mbmkuoqx-rotate` | 4/6 | **0/6** | large (~8+ IPs) |
| Residential rotating (current) | 6/6 | reliable w/ self-heal | small (~4) |

**Key constraint:** datacenter **can't serve downloads** — its large pool means the IP-lock retry never re-hits the resolver IP (0/6). So the split is per-operation:
- **metadata/process** → `YTDLP_PROXY_PRIMARY` (datacenter rotating, cheaper, ~67%) → `YTDLP_PROXY` (residential) fallback. Offloads preview traffic off residential quota.
- **download** → `YTDLP_PROXY` (residential) only.

**Code (`0a8f645`, on `main`):**
- `env`: `YTDLP_PROXY_PRIMARY` (metadata-first proxy).
- `ytdlp.ts`: metadata walks the proxy chain, falling back only on bot-challenge.
- `process-video.ts`: when `YTDLP_PROXY_PRIMARY` set, **don't cache** the metadata-resolved stream URLs (they'd be locked to the datacenter IP and poison downloads) — downloads re-resolve via residential.
- `stream-url.ts`: `resolveStreamUrl(forceFresh)` bypasses the cached URL.
- `videos-download.ts`: **on upstream 4xx, re-resolve fresh once and retry** — self-heals a cached URL pinned to an exit IP that rotated out of the pool (this was the cause of the binary 0/4-vs-4/4 per-video pattern).
- download retry bumped 8→16.

**Result:** **15/15** downloads across fresh videos; public end-to-end 206/full bytes; `gitSha=0a8f645`. (Note: `5qap5aO4i9A` "0/4" during testing was a **live stream** with no muxed itag 18 — expected, not a proxy fault.)

Env now: `YTDLP_PROXY`=residential rotating, `YTDLP_PROXY_PRIMARY`=datacenter rotating.
