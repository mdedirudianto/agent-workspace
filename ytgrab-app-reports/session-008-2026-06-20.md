# Session 8 — Real-download canary monitoring (Telegram alert when downloads break)

**Date:** 2026-06-20
**Scope:** Added **synthetic real-download monitoring** for ytgrab production. Closes the blind spot where downloads could silently break (e.g. the session-007 IPv4 egress bug) while every existing monitor stayed green. New Uptime-Kuma **push monitor** fed by a twice-daily cron on `app` that performs an actual end-to-end YouTube extract→download and alerts Telegram on failure.

---

## Problem

The three existing Kuma monitors for ytgrab only prove **liveness**, never that a real download works:

| # | Monitor | Checks | Interval |
|---|---------|--------|----------|
| 13 | `youtubegrab.com` | web HTTP 200 | 60s |
| 34 | `api.youtubegrab.com` | HTTP (301→apex→200, **UP**, redundant) | 60s |
| 45 | `YouTubeGrab Health` | keyword `status":"ok` on `/api/v1/health` | 300s |

`/api/v1/health` reports `db`/`redis`/`ffmpeg` in-process status only — it **never touches YouTube**. So the session-007 IPv4 egress bug (downloads → 302/403/ECONNRESET on the IP-locked googlevideo byte-fetch) would have left all three green the whole time. Confirmed there was **no synthetic download monitor anywhere**: no Kuma push/heartbeat monitor, no monitor hitting `:process`/`/download`, no cron on `app`.

## What was built

**Real-download canary** — exercises the exact path the IPv4 bug broke:
1. `POST /api/v1/videos:process {url}` → 200, caches metadata.
2. Read muxed itag from `streamOptions.muxed` (itag `18` = 360p mp4 — single stream, no ffmpeg merge).
3. `GET /api/v1/videos/<id>/download?itag=18` with `Range: 0-262143` → expect 200/206 + ≥1KB. This is `resolveStreamUrl` + undici `fetch` to googlevideo — the IP-locked step.

Tries 3 stable videos (`jVGJwKsiWT0`, `aqz-KE-bpKQ`, `dQw4w9WgXcQ`); **passes if ANY one fully downloads** (a single blocked/age-flagged video won't false-page). 256KB partial keeps Webshare proxy bandwidth negligible.

### Components

| Where | Artifact |
|---|---|
| `monitor` (Kuma) | **Push monitor #46** "ytgrab Download Canary", interval 43200s (12h), retry 3600s, linked to **Telegram — Dedi** (notification id 1). Push token `33e151b9…`, URL `http://10.0.0.3:3001/api/push/<token>` (private). |
| `app` (`devops`) | `/home/devops/ytgrab-download-canary.sh` — runs the flow vs `http://10.0.0.5:7009`; on success `?status=up`, on all-fail `?status=down&msg=<reason>` (immediate Telegram). Logs to `~/ytgrab-canary.log`. |
| `app` crontab | `0 6,18 * * * /home/devops/ytgrab-download-canary.sh` (server tz **Europe/Berlin**) — twice daily. |

## Verification

- Manual run on `app`: **exit 0**, log `UP OK jVGJwKsiWT0 itag=18 262144B` — real 256KB pulled through prod API.
- Kuma heartbeat for #46: **status=1 (UP)**, msg `OK jVGJwKsiWT0 itag=18 262144B`.
- Cron installed and confirmed via `crontab -l`.
- Telegram delivery: wiring identical to the 3 existing ytgrab monitors (same notification id 1). Live DOWN→Telegram **not fired** to avoid a false alarm on the user's phone; the `?status=down` path is in the script and the notification is linked.
