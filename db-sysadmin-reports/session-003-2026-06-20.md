# Session 3 — PostgreSQL file-descriptor alert (NOFILE soft-limit headroom)

**Date:** 2026-06-20
**Server:** `root@db` (`db.biji.uk`, `109.123.233.215`, internal `10.0.0.1`)
**OS:** Ubuntu 22.04.5 LTS
**Trigger:** Netdata fleet alert — `app.postgresql_fds_open_limit = 96.6%` ("apps group postgresql file descriptors utilization").

---

## Summary

Netdata reported PostgreSQL file-descriptor utilization at 96.6%. This was a **benign false alarm** caused by the systemd service running with the default soft `LimitNOFILE` of **1024**, only just above PostgreSQL's own `max_files_per_process = 1000`. A busy backend self-caps near 1000 open files → ~96.6% of 1024. Postgres never actually exhausts fds (it self-limits), but the gauge pins near 100% permanently. Fixed by raising the soft limit to 65536 via a systemd drop-in + restart.

---

## Diagnosis

| Check | Value |
|-------|-------|
| `max_files_per_process` | **1000** (Postgres self-cap per backend) |
| service `LimitNOFILE` (soft / hard) | **1024** / 524288 |
| busiest backend open fds | **989** → 989/1024 = **96.6%** |
| `fs.file-nr` / `fs.file-max` | 7168 / 9.2e18 (system-wide: not the constraint) |
| backends / max_connections | 61 / 200 |

The hard limit was already 524288 — only the **soft** limit (1024) was starving Postgres. Each backend touching ~1000 relation files sits just under the soft cap, which Netdata reports as ~near-100% utilization. No real risk of exhaustion; pure headroom problem. PostgreSQL docs recommend the soft NOFILE be comfortably above `max_files_per_process`.

---

## Actions Taken

| # | Action | Detail | Result |
|---|--------|--------|--------|
| 1 | Created systemd drop-in | `/etc/systemd/system/postgresql@16-main.service.d/override.conf` → `[Service] LimitNOFILE=65536` | Done |
| 2 | `systemctl daemon-reload` | Staged (inert until restart — NOFILE is a start-time limit) | Done |
| 3 | `systemctl restart postgresql@16-main` | Approved restart (brief fleet-wide connection blip ~1–2s) | Done |
| 4 | Verified new limit | MainPID `/proc/<pid>/limits` → `Max open files 65536 / 65536` | Verified |
| 5 | Verified health | service `active`; `select 1` OK; 25 backends already reconnected | Verified |

After restart the busiest backend's 989 fds is ~1.5% utilization — alert clears on next Netdata poll.
