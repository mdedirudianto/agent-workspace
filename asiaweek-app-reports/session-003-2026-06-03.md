## Context

The observability/analytics feature merged on `main` (`b43c83d`) after the session-002 prod deploy (`9bb5a09`), so production was running with **zero** observability and still on `NODE_ENV=local`. The code wires four tools per [docs/engineering/analytics.md](../../Projects/asiaweek/docs/engineering/analytics.md): GA4 + OpenPanel (consent-gated product analytics, build-time inlined), GlitchTip (Sentry-compatible error tracking), OpenObserve (server log/metric shipping). All config is optional — unset means the feature is simply omitted.

User supplied 8 keys split across the three surfaces (web / app / api) plus a request to stand up OpenObserve. User approved the plan and the `NODE_ENV → production` cutover before execution.

**Key structural finding:** prod env lives in **two** places — `.env` is sourced only at **build time** (inlines `PUBLIC_*` / `VITE_*` into the web/app bundles), while the api reads `process.env` directly via a Zod schema with **no dotenv**, so its runtime secrets come from the PM2 `env:` block in `ecosystem.config.cjs`. Both had to be updated.

**Design choices:**
- **OpenObserve over internal LAN** (`http://10.0.0.3:5080`) — `app` (`10.0.0.5`) ↔ `monitor` (`10.0.0.3`) on `10.0.0.0/24`, so log shipping skips Cloudflare/public proxy/SSL entirely.
- **Stream-per-source convention** — OO `default` org already holds `app`, `db`, `ytgrab`, … one stream each; followed it with `asiaweek_api` rather than a new org.
- **Scoped service account, not root** — the app `.env`/ecosystem never holds the OO `admin@biji.uk` password.
