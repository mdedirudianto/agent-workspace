# Session 1 тАФ Initial Audit & Hardening

**Date:** 2026-05-17
**Server:** root@backup (public IP `194.233.81.93`, internal `10.0.0.4`)
**OS:** Ubuntu 22.04.5 LTS (Jammy Jellyfish), kernel 5.15.0-171-generic (pre-upgrade)
**Purpose:** First-ever audit of this server тАФ document what it does, fix the worst issues found.

## Role

Dedicated **backup target** for the internal cluster. Pulls daily dumps from:

- `db` (10.0.0.1) тАФ PostgreSQL 16 (8 databases, dominated by `wagrab` ~39 GB) and MongoDB (`quranchat`)
- `erp` (10.0.0.6) тАФ ERPNext/Frappe v16 site `erp.gro.biz.id` (triggers remote `bench backup --with-files`, rsyncs the archive locally)

Stores everything under `/root/backups/`. No application traffic тАФ pure backup destination.

## Actions Taken (this session)

1. **SSH key-only auth** тАФ added `/etc/ssh/sshd_config.d/00-hardening.conf` to force `PasswordAuthentication no`, `PermitRootLogin prohibit-password`, `MaxAuthTries 3`, `X11Forwarding no`, `ClientAliveInterval 300`. Cloud-init's `50-cloud-init.conf` had `PasswordAuthentication yes` and won (alphabetical first match) тАФ the new `00-` prefix loads before it. Verified with a fresh SSH session before continuing.
2. **fail2ban installed & enabled** тАФ was not present. Configured `/etc/fail2ban/jail.local` with `mode=aggressive`, `bantime=1h`, `findtime=10m`, `maxretry=5`, ignoreip `10.0.0.0/22`. Sshd jail active.
3. **Pruned authorized_keys** тАФ removed `rsa-key-20160704` (a 2016-vintage PuTTY RSA key, unknown owner) and a malformed/empty trailing line. Kept 4 keys: `root@prod`, `klo@mini`, `klo-cloud@numa`, `dedi@mini.local`. Backup at `/root/.ssh/authorized_keys.bak.2026-05-17`.
4. **Locked down backup script perms** тАФ `/opt/backup/config.sh` was mode `644` with `PGPASSWORD="pgpr0d123"` and a live Telegram bot token in cleartext. Now `600`. All other backup scripts moved to `700`.
5. **Stopped & disabled unused services** тАФ `nginx` (was serving the default "Welcome to nginx" page, no UFW exposure but pointless), `ModemManager`, `wpa_supplicant`, `packagekit`, `snapd`. `ModemManager` and `wpa_supplicant` masked so they can't be re-activated by udev.
6. **Removed `certbot` snap** тАФ installed but managing zero certificates; weekly renew timer doing nothing.
7. **Reclaimed ~12 GB of `/var/log` bloat** тАФ `journalctl --vacuum-time=14d` recovered 3.5 GB; `/var/log/pcp` (legacy Performance Co-Pilot archive, no service running) deleted entirely (8.7 GB); `/var/log/btmp` + `btmp.1` (215 MB of brute-force failed logins) truncated.
8. **Added weekly backup verification cron** тАФ `verify.sh` existed but was manual-only. Now runs every Tuesday 21:00 UTC (Wed 04:00 WIB), well outside the daily backup window.
9. **apt full-upgrade + reboot** тАФ 80+ pending updates including kernel `5.15.0-171` тЖТ `5.15.0-179`. `/var/run/reboot-required` was set. Upgrade + reboot performed after the daily backup completed.

## Backup System

Backup automation lives entirely in `/opt/backup/`:

```
/opt/backup/
тФЬтФАтФА backup.sh        # Cron entry тАФ runs all jobs sequentially
тФЬтФАтФА config.sh        # Paths, DB credentials, Telegram token (mode 600)
тФЬтФАтФА notify.sh        # Telegram failure alerts
тФЬтФАтФА rotate.sh        # Tiered retention deletion
тФЬтФАтФА verify.sh        # Restore verification (small DBs full restore, large DBs format-check)
тФФтФАтФА jobs/
    тФЬтФАтФА wagrab-pg.sh
    тФЬтФАтФА tier2-pg.sh         # feyfa, pasarbesar*, miawflow
    тФЬтФАтФА tier3-pg.sh         # glitchtip, openpanel
    тФЬтФАтФА quranchat-mongo.sh
    тФФтФАтФА erp-bench.sh        # ERPNext bench backup pulled from erp (added 2026-05-17)
```

**Cron schedule:**

| Job | Schedule (UTC / WIB) | Script |
|-----|---------------------|--------|
| Daily backup all DBs | 18:00 UTC daily / 01:00 WIB | `/opt/backup/backup.sh` |
| Weekly rotation | 20:00 UTC Sat / 03:00 WIB Sun | `/opt/backup/rotate.sh` |
| Weekly verification (added 2026-05-17) | 21:00 UTC Tue / 04:00 WIB Wed | `/opt/backup/verify.sh` |

`backup.sh` also calls `rotate.sh` inline at the end, so explicit Sunday rotation is mostly redundant тАФ kept for safety.

**Retention tiers:**

| Tier | Databases | Days retained |
|------|-----------|---------------|
| Tier 1 | `wagrab` | 4 |
| Tier 2 | `feyfa`, `pasarbesar`, `pasarbesar-beta`, `pasarbesar-lpekinhub`, `miawflow`, `quranchat` (mongo), `erp` (bench) | 7 |
| Tier 3 | `glitchtip`, `openpanel` (monitoring DBs) | 3 |

**Storage layout:**
- `/root/backups/pg/<db>/<db>-YYYY-MM-DD.dump` тАФ PostgreSQL custom-format (`pg_dump --format=c --blobs --no-owner`, restorable with `pg_restore`)
- `/root/backups/mongo/<db>/<db>-YYYY-MM-DD.dump` тАФ MongoDB archive (`mongodump --archive`)
- `/root/backups/bench/erp/<timestamp>-erp_gro_biz_id-{database.sql.gz,files.tar,private-files.tar,site_config_backup.json}` тАФ ERPNext bench backup (4 files per run, ~1 MB total)

**ERPNext (`erp-bench.sh`) flow:**
1. SSH to `erp` (10.0.0.6) using dedicated key `/root/.ssh/id_ed25519_pull`
2. Run `sudo -u frappe bench --site erp.gro.biz.id backup --with-files` remotely
3. `rsync -a --delete` the entire `private/backups/` dir back to `/root/backups/bench/erp/`
4. Verify `*-database.sql.gz` exists and is non-empty
5. Telegram alert on any failure

**Verification logic** (`verify.sh`):
- Small DBs (<1 GB): full `pg_restore` to a temp DB on `db`, count tables, drop.
- Large DBs (тЙе1 GB, i.e. wagrab): `pg_restore --list` format-check only (avoids hour-long restore).
- DBs with тЙд4 dump objects classified as "empty production DB" and SKIPPED (not failed).
- MongoDB: `mongorestore --dryRun` against the archive.
- Failure тЖТ Telegram alert.
