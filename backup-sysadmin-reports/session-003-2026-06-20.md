# Session 3 — Backup coverage gap: Tier 2 switched to auto-discovery

**Date:** 2026-06-20
**Server:** `root@backup` (public `194.233.81.93`, internal `10.0.0.4`)
**OS:** Ubuntu 22.04.5 LTS
**Trigger:** Fleet Telegram alert — `1 backup job(s) FAILED — check /var/log/backup.log` (recurring nightly since 2026-06-13).

---

## Summary

The nightly `tier2-pg` job had been failing every night for a week because it dumped a hardcoded DB named `miawflow`, which no longer exists (renamed to `miawflow_baznas` + `miawflow_lpekin` during the 2026-06-17/18 DuitKas multi-tenant split). Investigation revealed the failure was the tip of a larger gap: **8 production databases had zero backup coverage** because the Tier 2 include-list was never updated as new apps launched. Fixed the immediate failure and **converted Tier 2 from a hardcoded include-list to auto-discovery**, so future app DBs are backed up automatically without editing the script.

---

## Root Cause

`/opt/backup/jobs/tier2-pg.sh` line 9 looped over a fixed list ending in `miawflow`:

```
pg_dump: error: connection to server at "db" (10.0.0.1), port 5432 failed:
         FATAL:  database "miawflow" does not exist
[tier2] FAILED: miawflow
```

The job exits non-zero on any failure → nightly `[ALERT] ... 1 backup job(s) FAILED`. First failure 2026-06-13, every night since.

## Design Note — why auto-discovery

The previous design required editing **two** lists in sync (dump list in `tier2-pg.sh`, prune list in `rotate.sh`) every time an app launched. Both drifted. The new model is an **exclude-list** (`TIER2_EXCLUDE`, owned by dedicated jobs/tiers) shared by both scripts: anything new on `db` falls into Tier 2 automatically and is pruned at 7 days. To carve a DB into a different tier in future, add it to `TIER2_EXCLUDE` and give it its own job.

## Follow-ups completed same session

### #1 — Verified the new backups actually restore

- `pg_restore --list` on all 9 newly-covered dumps → valid TOCs (e.g. `asiaweek` 737 entries, `seenow` 663, `ytgrab_prod` 82).
- End-to-end restore of `ytgrab_prod` into a scratch DB on `db` → 14 tables, real row counts (`process_events` 13,267, `download_events` 11,393, `videos` 3,975). Scratch DB dropped after.
- Note: dumps carry a `SET transaction_timeout` directive (PG17) that PG16 ignores on restore — the backup host's `pg_dump` client is a major version ahead of `db`'s PG16 server. Cosmetic; data restores intact.

### #2 — Routine backups no longer run as the `postgres` superuser

Replaced the superuser for **routine dump jobs** with a least-privileged role; `verify.sh` keeps a privileged role only for its temp-DB create/restore/drop.

- On `db`: `CREATE ROLE backup_reader LOGIN` + `GRANT pg_read_all_data` + `ALTER ROLE backup_reader BYPASSRLS`. Profile: not superuser, no createdb/createrole, read-only. (`BYPASSRLS` is required — `pg_dump` refuses a partial dump of RLS tables, which `eventopia` and `seenow` have; the old superuser bypassed RLS implicitly.)
- No `pg_hba.conf` change needed — existing `host all all 10.0.0.4 md5` already covers any user from backup; PUBLIC has CONNECT on every DB.
- `config.sh`: `PG_USER=backup_reader` (+ its password); added `PG_ADMIN_USER=postgres` / `PG_ADMIN_PASSWORD` used **only** by verify.
- `wagrab-pg.sh`: dropped `--role="wagrab"` (backup_reader can't `SET ROLE` to it, and read-all + `--no-owner` makes it unnecessary).
- `verify.sh`: `CREATE DATABASE` / `pg_restore` / `DROP DATABASE` switched to `PG_ADMIN_USER`; read-only checks stay on `backup_reader`.
- **Verified:** tier2/tier3/grobiz dump jobs all `EXIT=0` as `backup_reader`; `verify.sh seenow` restored 57 tables via the admin path and `EXIT=0` (also clears the standing "run verify.sh once" follow-up).
- Files backed up: `config.sh.bak-20260620-104604`, `wagrab-pg.sh.bak-...`, `verify.sh.bak-...`.
