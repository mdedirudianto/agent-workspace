# Session 6 — GroBiz + PasarBesar routes → local backends (no code change)

**Date:** 2026-06-08
**App:** miaw-route v0.9.7 (no redeploy — DB-only change)
**URL:** https://routes.biji.uk
**Scope:** `routes` table backend_url cutover (prod DB on `db`, `miaw_route`)

---

## Goal

miaw-route and its downstream apps (grobiz, pasarbesar) all run on the **same host**
(`app`, internal `10.0.0.5`). The two non-default routes forwarded to **public**
Cloudflare-fronted URLs, so each webhook did a wasteful round-trip
**Meta → app (miaw-route) → Cloudflare → back to app**. Point them at the local
listeners instead.

---

## Findings (checked live before changing anything)

### Local listeners on `app`
| app | PM2 | bind | webhook path (from source + live probe) |
|-----|-----|------|------------------------------------------|
| grobiz-api | id 49, v0.9.0 | **`0.0.0.0:4000`** | `/api/webhooks/whatsapp` → `POST` **401** (live) |
| pasarbesar | id 28, v0.9.2 | **`10.0.0.5:7002`** | `/api/webhooks/whatsapp` → `POST` **403** (live) |

> 401/403 are signature/origin rejections of an unsigned empty test body — they prove
> the endpoint is live at that path. Real Meta-signed forwards verify against each
> downstream's own app secret, unchanged by the network-path switch.

### Two issues found on the **PasarBesar** route (pre-existing)
1. **Wrong path.** Route pointed at `…/webhook/whatsapp`, but the deployed app only
   serves `/api/webhooks/whatsapp` (`/webhook/whatsapp` → 404, no next.config rewrite).
2. **Stale/zero traffic.** Route matches `waba_id 2033865784136567` — **0 rows in
   `webhook_logs` ever**. Actual unmatched traffic falling to Default Echo is two
   **unconfigured** WABAs: `100405506465612` (52 hits today) + `2720407334981244` (50).
   Also `api.pasarbesar.id` currently returns Cloudflare **526** (origin SSL broken).

Per decision this session: **fix URL only** (host + path); WABA-match investigation
deferred. pasarbesar reached via **`10.0.0.5:7002`** (it does **not** bind `127.0.0.1`,
so `localhost:7002` is unreachable — chose the internal IP over rebinding the app).

### Echo fallback (from session-005) is working
Default Echo: **55 success / 200** today (catching the 2 unknown WABAs) — confirms the
local echo sink is live and capturing.

## Change applied (prod DB, in-place UPDATE — ids preserved)

```sql
UPDATE routes SET backend_url='http://localhost:4000/api/webhooks/whatsapp'
  WHERE name='GroBiz - PROD';
UPDATE routes SET backend_url='http://10.0.0.5:7002/api/webhooks/whatsapp'
  WHERE name='PasarBesar - PROD';
```

| route | before | after |
|-------|--------|-------|
| GroBiz - PROD | `https://api.gro.biz.id/api/webhooks/whatsapp` | `http://localhost:4000/api/webhooks/whatsapp` |
| PasarBesar - PROD | `https://api.pasarbesar.id/webhook/whatsapp` | `http://10.0.0.5:7002/api/webhooks/whatsapp` (host **and** path fixed) |

> Route cache TTL = 60s ([matcher.ts:14](https://github.com/biji-dev/miaw-route/blob/main/src/services/matcher.ts));
> direct SQL bypasses `invalidateRouteCache()`, picked up on next refresh (≤60s). No restart.
