## Post-deploy hotfix — HTML pages served as `text/plain` (`2a159d2`)

**Symptom:** after deploy, `https://routes.biji.uk/login` (and `/dashboard`) returned the full HTML markup but the browser showed it as raw text — page "not rendering".

**Root cause (confirmed by local repro, not guessed):** `@elysiajs/opentelemetry` re-finalizes every response and **drops the `text/html` content-type** that `@elysiajs/html` sets, so HTML string responses fall back to `text/plain`. This only manifests now because OTel was disabled in session-001 (no endpoint) — enabling it activated the plugin.

| Repro (real JSX) | content-type | body == baseline |
|---|---|---|
| baseline (no OTel + `html()`) | `text/html` | — |
| OTel + `html()` + global `onAfterHandle` ✅ | `text/html` | identical |
| OTel + `onAfterHandle`, no `html()` | `text/html` | **differs** (html plugin needed for body) |
| OTel + `html()` only (the bug) | `text/plain` | — |

**Fix:** added a global `onAfterHandle` in `src/index.ts` that re-asserts `content-type: text/html` for string responses starting with `<` (mutating `set.headers`, which OTel preserves). Guard keeps JSON responses and the webhook verify-challenge string untouched. `html()` retained for body rendering.

**Verified live:** `/login` → `200 text/html` (renders), `/`+`/health` → `application/json`, webhook challenge → `text/plain` echo, traces still flowing (spans for `service=miaw-route`).

> ⚠️ **For future redeploys:** keep this `onAfterHandle` whenever the OTel plugin is enabled. If HTML pages render as raw text, this is the cause.
