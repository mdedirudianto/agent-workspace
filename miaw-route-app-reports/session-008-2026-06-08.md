# Session 8 — Diagnose high Bun CPU → migrate miaw-route from PM2 to systemd

**Date:** 2026-06-08
**App:** miaw-route v0.9.7 (no code change — process-manager migration)
**Server:** `app` (154.26.129.104), Bun 1.3.14
**Result:** steady-state CPU **23.3% → 9.8%** (~58% reduction)

---

## Symptom

`top` on `app` showed the miaw-route `bun` process at **73% CPU** in bursts, ~152 min
accumulated CPU over 8h (~31% avg) — high for a low-traffic webhook router.

## Diagnosis (evidence-based)

1. **Not app code.** Workers use plain `setInterval` (retry 1s, reconciler 300s, cleanup
   3600s), no busy loops; prod logging is JSON to stdout (pino-pretty is dev-only).
2. **`strace -c` = spinning, not work:** 67% `futex` + ~200 `sched_yield`/sec — the classic
   signature of a busy-waiting thread. Main JS event-loop thread held the CPU.
3. **PM2 fork mode is the amplifier.** PM2 can't run Bun in cluster mode
   ([oven-sh/bun#10821]), so it uses **fork mode** — wrapping Bun in
   `ProcessContainerForkBun.js` and piping all stdio/IPC through sockets (confirmed:
   process fd 0/1 are PM2 sockets). Bun's Node-compat layer for that IPC **busy-polls**.

   **Controlled isolation test** (steady-state CPU, post-warmup, same box):

   | scenario | CPU |
   |---|---|
   | standalone idle `bun` | **1.9%** (Bun's own idle baseline — [oven-sh/bun#21081]) |
   | idle `bun` **under PM2 fork** | **8.6%** (~4.5× — pure PM2↔Bun overhead, zero app work) |
   | real miaw-route under PM2 | **23.3%** (bursts ~73%) |

   ⇒ PM2 fork added **~6–7 points of pure spin** even on a do-nothing script; the rest is
   miaw-route's own work (OTel export 15s, per-request logging crossing the PM2 socket,
   polling workers).

> Severity was *waste, not outage* — load was healthy (≤1.7 on 6 cores), ~3.3GB RAM free.

## Fix: run Bun under systemd (its documented production pattern)

PM2's value-add (cluster, Node IPC) doesn't apply to Bun, and its fork wrapper is what
spins. systemd execs the `bun` binary directly — no IPC wrapper, no stdio-over-socket,
logs to journald. miaw-route is the **only Bun app** under PM2 (the other 17 are Node, which
PM2 handles fine), so it's a clean one-off split, not a fleet change.

### Unit — `/etc/systemd/system/miaw-route.service`
```ini
[Unit]
Description=miaw-route — WhatsApp Cloud API webhook router (Bun)
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=devops
Group=devops
WorkingDirectory=/home/devops/miaw-route
ExecStart=/home/devops/.bun/bin/bun --env-file=/home/devops/miaw-route/.env.production src/index.ts
Restart=always
RestartSec=2
MemoryMax=600M          # safety guard (app sits ~30–165M); mirrors PM2 max_memory_restart intent
KillSignal=SIGTERM      # app logs "shutting down gracefully" on SIGTERM
TimeoutStopSec=15
StandardOutput=journal
StandardError=journal
SyslogIdentifier=miaw-route

[Install]
WantedBy=multi-user.target
```
ExecStart mirrors PM2 exactly: `bun --env-file=.env.production src/index.ts`, cwd
`/home/devops/miaw-route`, user `devops`. Env is loaded by **Bun's `--env-file`** (not
systemd `EnvironmentFile`) to avoid quoting/parse differences with the existing
`.env.production`.

### Cutover
1. `systemctl daemon-reload`
2. `sudo -u devops pm2 delete miaw-route && pm2 save` (frees :9001; removes it from the
   PM2 dump so it won't resurrect on boot — no double-bind)
3. `systemctl enable --now miaw-route`

## Rollback (if ever needed)
```
systemctl disable --now miaw-route
sudo -u devops pm2 start /home/devops/miaw-route/ecosystem.config.cjs && pm2 save
```
The PM2 `ecosystem.config.cjs` (in the repo dir) is intact and matches the old config.

## Operational note (convention change)
**miaw-route no longer runs under PM2** — it's a **systemd service**. `pm2 restart
miaw-route` will fail (not found); use `systemctl restart miaw-route`. Redeploys: after
`git pull` + any build, run `systemctl restart miaw-route` (no `pm2 reload`). All other
`app` services remain on PM2.
