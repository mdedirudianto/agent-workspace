# Session 4 — `routes.biji.uk` Login 504s Traced to Host Swap-Thrash (ytgrab-api Leak)

**Date:** 2026-06-17
**Server:** `root@app` / `devops@app` (`154.26.129.104`, internal `10.0.0.5`) — diagnosis also read `proxy` nginx logs
**OS:** Ubuntu 24.04.4 LTS
**Goal:** User couldn't log in to `routes.biji.uk` (admin) — Cloudflare **504 Gateway Time-out**. Find root cause, fix.

---

## TL;DR

The 504 was **real but transient** — 3 timeouts in a 2-minute window (`09:03:28–09:05:33 +0700`), then self-recovered. It was **not** a miaw-route bug: the service was up, listening, and serving `/login` in ~0.3s before and after. The login `POST` timed out because the **Bun event loop froze for ~8 minutes** (app logged nothing `01:59:36 → 02:08 UTC`; nginx's TCP connect *succeeded* but no response header arrived in 60s → 504).

Root cause is **host-level memory pressure**: the app server's **swap was 95% full** (1.9/2.0 Gi, 87 Mi free) with **IO pressure stall `full avg300=0.52`** (disk stalled ~half the time = swap thrashing). The primary driver was a **memory leak in `ytgrab-api`** — grown to **2.5 GB RSS over 12 days** (20% of RAM). An idle service like miaw-route gets paged out; when the login hit it, swap-in stalled the event loop under IO contention.

Same mechanism explains **recurring upstream timeouts on the same host across days** (`foucher.co:3006`, `youtubegrab:3009`, `api.wagrab:8001` in proxy error log Jun 13–16) — this was not unique to miaw-route.

Fix: restarted `ytgrab-api` (2.5 GB → 95 MB) + `miaw-route`, freeing ~2.3 GB RAM; capped `ytgrab-api` at `max_memory_restart 800M` (persisted via `pm2 save`). Login verified 200 through the full Cloudflare → proxy → app path.

## Findings / Follow-ups

- **Structural / capacity strategy (user, 2026-06-17):** 37 PM2 entries on **11 GB RAM** (now ~41% used after the fixes) + **4 GB swap** (upsized this session). Docker disabled. Plan: **keep the box lean, upgrade RAM only when it genuinely hits the limit, and peel resource-heavy apps onto dedicated servers** rather than over-provisioning here. Caps are guardrails, not a cure.
- **DONE — swap/memory alerting wired up.** The stock Netdata `used_swap` alarm (warn 80 / crit 90) already existed and was **CRITICAL at 92.7%** — but **no notification channel was configured** (stock `health_alarm_notify.conf` = email-to-local-root only), so it never reached anyone. That was the real gap, not a missing alarm. Reused the existing **Telegram bot** ("Telegram — Dedi", chat `293832479`) already stored in **Uptime-Kuma's embedded MariaDB on `monitor`** — no new bot needed. Enabled `SEND_TELEGRAM` + token + `DEFAULT_RECIPIENT_TELEGRAM` in `/etc/netdata/health_alarm_notify.conf` on **`app`** (640 `root:netdata`; app runs health checks locally). Test alert (`alarm-notify.sh test sysadmin`) **delivered to Telegram — OK** across WARNING/CRITICAL/CLEAR.
- **DONE — fleet-wide Telegram rollout.** Replicated the same Netdata Telegram config to every host running Netdata: **`db`, `proxy`, `monitor`, `erp`** (+ `app`). Each verified `SEND_TELEGRAM="YES"`, perms `640`, and **direct egress to api.telegram.org = HTTP 200** (sent a per-host "configured on \<host\>" confirmation). `backup` and `dev` run **no Netdata** (inactive) — nothing to configure there. Token sourced from Kuma on `monitor` (see [[fleet-telegram-bot]]); secret staged to `/root/.tg_stage` per host and removed after.
- **DONE — swap hygiene (flushed, not enlarged).** Decided **against raising swap** — the 504 was caused by swap *latency* (paging a frozen service back in), not by running out of swap; more swap risks worse tail-latency, and `vm.swappiness` is already `10`. After the leak fix, memory PSI was `0.00` (no pressure) but swap sat 91% full of **stale** pages = no safety buffer. Ran `swapoff -a && swapon -a` on `app` (safe with 6.8 Gi RAM free) → swap **0 B used** as a clean OOM cushion; `used_swap` alarm back to CLEAR. *(Later upsized 2 → 4 GB — see below.)*
- **DONE — root cause of the `ytgrab-api` leak identified.** `apps/api/src/middleware/logging.ts`: `const route = c.req.routePath ?? c.req.path;` then `httpRequestsTotal.labels(method, route, status)` + `httpRequestDurationSeconds.labels(...)`. On **unmatched routes (404s)** `routePath` is undefined, so it falls back to the **raw caller-controlled URL** as a Prometheus label. A public API (`api.youtubegrab.com`) gets thousands of unique bot-probe paths → prom-client stores each as a **permanent time series** (Histogram = buckets+sum+count each) → unbounded heap growth ≈ 200 MB/day → 2.5 GB/12 d. **Fix:** `const route = c.req.routePath ?? "unmatched";`.
- **DONE — `ytgrab-api` leak fix applied, built, deployed, and merged to `main`.** Edited `logging.ts`, `pnpm build` (tsc), verified fix in compiled `dist`, relaunched. Branch `fix/prom-route-label-leak` **merged into `main` with `--no-ff`** (no PR) → merge commit **`1017f37`**; server `main` synced to it. **API-only redeploy** (user's call) — `ytgrab-web` held at `8f38ccf`, so the **14 unreleased web/mobile commits on `main` are NOT shipped**. The server checkout was 14 commits behind; deliberately kept the deploy to the one-line API fix via a `git worktree`. Full detail in [ytgrab-app-reports/session-006](../ytgrab-app-reports/session-006-2026-06-17.md).
- **DONE — capped 37/37 PM2 apps (all effective).** First pass capped 33/37; the other 4 were **wrapper launches** where pm2 monitored the wrapper not the leaking child — **`ytgrab-api` itself was one of them** (`pnpm start` → `sh` → `node`, so its earlier 800 M cap was watching pnpm, not the 2.5 GB node leaf). Relaunched all 5 via **direct exec** so pm2 tracks the real process and caps fire: `ytgrab-api`/`washop`/`wagrab-web` as `node <entry>` (SSR apps need inline `HOST=10.0.0.5 PORT=…`; ytgrab-api self-loads `.env` via `dotenv/config`); `wagrab-admin`/`feyfa-admin` as `node_modules/.bin/directus start` (auto-loads `.env`). All verified listening on their ports, `pm2 save`. directus apps now show real RSS (311/355 MB) instead of the 15 MB wrapper.

After the API-only hotfix, user approved the **full web/mobile release**: `ytgrab-web` rebuilt from `main` and shipped (`/install` live on `youtubegrab.com`) — `app` now fully on `1017f37`. Same fix applied on **`dev`** (api → `1017f37`, relaunched direct+capped, :7010; dev web left at `8f38ccf`). Detail in [ytgrab-app-reports/session-006](../ytgrab-app-reports/session-006-2026-06-17.md).
- **DONE — swap resized 2 → 4 GB (capacity strategy).** Per user ("keep it clean; upgrade RAM only when it actually hits the limit; peel heavy apps to dedicated servers"): RAM verified **41% used** (no pressure), so no RAM upgrade now. Bumped the `/swapfile` 2 → 4 GB (`dd` + `mkswap`, fstab size-agnostic so persists) purely as a bigger OOM cushion; `swappiness` stays `10` so it's a buffer not a hot path.
