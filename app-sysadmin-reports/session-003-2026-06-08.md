# Session 3 — SSH "Ban" Misdiagnosis + Fleet-Wide fail2ban Whitelist

**Date:** 2026-06-08
**Server:** `root@app` (`154.26.129.104`, internal `10.0.0.5`) — primary; changes also applied on `proxy`, `dev`, `monitor`, `backup`, `erp`
**OS:** Ubuntu 24.04.4 LTS
**Goal:** Restore reported "banned" SSH access to `app`/`deploy`, then whitelist the static office IP fleet-wide so a few failed logins can't lock us out.

---

## TL;DR

There was **no ban**. `root@app`, `devops@app`, and `proxy` were all reachable the whole time. The failing command was `ssh deploy@app`, and **`deploy` is not a user on `app`** — the deploy account there is **`devops`**. "Permission denied (publickey,password)" is an *auth* failure (wrong username), not the timeout/refused you'd see from a fail2ban DROP. Office IP `103.119.54.100` was never in any ban list.

Fix for daily use: **`ssh devops@app`** (key already authorized). Then, as requested, added the office IP to `fail2ban` `ignoreip` on every host that runs fail2ban.

## Findings / Follow-ups

- **`db` runs no fail2ban.** Not a problem for lockout, but worth noting for the security posture — SSH there has no brute-force throttling. Consider enabling fail2ban on `db` in a future session (would inherit the same office-IP whitelist).
- **Old office IP `125.166.13.120`** still lingers in `app` and `dev` ignoreip. Harmless; left in place. Can be pruned later if it's confirmed dead.
- **IPv6:** office has IPv6 (`2402:8780:1538:800c::/64`) but all these hosts are reached over IPv4, so no v6 whitelist was needed. If a host ever gets an AAAA record and SSH starts preferring v6, revisit.
