## Smoke test: guest checkout (live, real order, no login)

This pull's headline change — buying a ticket with **zero account** — hadn't been exercised in prod before, so it got a dedicated live test:

1. Signed up a fresh test org via email (`SMOKE-TEST Session-017 Org`, reusing the team `hi@eventopia.my` mailbox per convention; OTP read from Redis `otp:email:hi@eventopia.my`), created + published one event (`SMOKE-TEST Session-017 Event`) with a free ticket type (50 quota).
2. Opened the **public event page** and completed checkout **directly on the page** (no separate login/account step) — name, WhatsApp number, email — confirming the new no-account UX.
3. Confirmed the order via **reverse-WhatsApp**: no real phone was available, so — mirroring the session-008/016 technique — POSTed an **HMAC-signed synthetic inbound webhook** (`x-hub-signature-256`, real `META_APP_SECRET`) to `/webhooks/whatsapp` with the exact confirmation text the UI displayed (`Konfirmasi tiket Eventopia: VQN7H3ZZ\nEvent: SMOKE-TEST Session-017 Event`, matching `buildTicketConfirmText`/`parseTicketConfirmCode` in `packages/whatsapp/src/verify.ts`).
4. Order flipped to **`PAID`**, ticket to **`ISSUED`** within seconds — verified directly in Postgres.

**Cleanup gotcha (new, worth remembering):** the base `account` identity row (login credential — distinct from `organizer_account`, which only holds org-level display/status settings) has **no FK from `event`, `orders`, or `organizer_account`** — the only link is a Redis key `membership:acc:<accountId>` whose value carries `organizerId`. Missed this on the first cleanup pass (deleted `organizer_account` but left the `account` row behind), caught it by noticing the prod account count was 8 instead of the expected 7 baseline. To find it: pull `organizerId` from the created event/order, `redis-cli keys 'membership:*'` + grep values for that `organizerId`, extract `acc_...` from the key name.

**Cleanup performed:** 13 Postgres rows (`tickets`, `order_item`, `ledger_entry`×2, `payment_intent`, `orders`, `audit_log`, `registration_form`, `ticket_type`, `event_day`, `event`, `organizer_account`, `account`) + 8 Redis keys (`org:meta:*`, `org:slug:*`, `membership:acc:*`, `idem:*`, `wa-verify:code:*`, `order-await-confirm:*`, `order-key:*`, `ticket-req:order:*`). Verified post-cleanup row counts (7 accounts / 7 events / 22 orders) exactly match session-016's baseline — no organic growth in the 3 days between sessions, confirming a fully clean removal.
