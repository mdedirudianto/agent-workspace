## Smoke test — isolated DIRECT-Xendit repro (real Xendit sandbox, dev)

Dev had no existing organizer with a Xendit BYO/DIRECT credential, and per
`docs/engineering/xendit-gateway.md` the DIRECT scenario can only be truly exercised against
a real Xendit sandbox account (fake-mode can't produce it). With no second sandbox merchant
available, reused dev's own platform-level Xendit sandbox key as a stand-in for "the
organizer's own key" — isolated for safety:

1. Commented `XENDIT_API_KEY`/`XENDIT_WEBHOOK_TOKEN` out of dev `.env`, restarted
   `eventopia-api`+`eventopia-workers` (confirmed boot still clean — dev's
   `PAYMENTS_DEFAULT_GATEWAY=doku`, so an absent platform Xendit doesn't block boot).
2. Created a fresh test organizer (`org_4d7faa76606446de`), published one event with a real
   25,000 IDR ticket type.
3. Saved the captured platform key/token as that org's own Xendit BYO keys via the real
   credential-save endpoint — **auto-registration succeeded for all three channels**
   (`qr_code`, `ewallet`, `fva_paid`) against the new per-org URL, confirming
   `webhook-registrar.ts` works against the live Xendit API.
4. Operator-approved DIRECT (`super@eventopia.co.id`, no MFA on dev) — confirmed
   `organizer_account.xendit_direct_approved = true`.
5. Real guest checkout → real Xendit QRIS intent (`qr_501762da-…`) → simulated payment via
   Xendit's real sandbox simulate endpoint → `200 SUCCEEDED`.
6. **Finding:** the inbound webhook to the new per-org URL came back `401`, not `200`. Root
   cause: this sandbox account's actual Xendit-dashboard Callback Verification Token didn't
   match dev's `.env` value (the account had been dark/unused on dev — the stored token
   looked like a stale placeholder rather than one ever actually copied from Xendit's
   dashboard). **Not a defect in the fix** — a credential-provenance gap in this specific
   sandbox account's setup.
7. **The `payment-reconcile` worker caught it, exactly as designed:** order stayed `PENDING`
   through the webhook failure; after the grace window the reconciler polled Xendit directly
   and recovered it — `RECOVERED a captured payment the webhook never delivered: order=ord_941911d7…
   intent=pin_23f0ab37… provider=xendit`. Final state: order `PAID`, ticket `ISSUED`. This is
   the "lost webhook" floor the fix adds, verified against a real failure rather than a
   contrived one.

**Cleanup (fully completed, verified zero residue):** bare platform webhook URL

## Findings

**1. (New) Dev's platform Xendit sandbox account's real dashboard Callback Verification Token
likely doesn't match `XENDIT_WEBHOOK_TOKEN` in `.env`.** Only surfaced because this session's
test forced a real webhook call — the reconciler papered over it, so it would otherwise go
unnoticed indefinitely. Low priority (dev default gateway is `doku`, so platform-mode Xendit
isn't exercised in normal use), but worth syncing the real token from Xendit's sandbox
dashboard before the next time anyone needs a clean (non-reconciler-dependent) webhook test
on dev.

**2. Minor secret-hygiene note.** The platform sandbox `XENDIT_API_KEY`/`XENDIT_WEBHOOK_TOKEN`
values were transiently visible in intermediate command output during the isolate/restore
steps (both mine and the delegated subagent's) — dev-only sandbox credentials, no prod or
real-money exposure, not included in this report or committed anywhere.

**3. (Carried, unaffected by this session)** session-018's open item — the self-service BYO
gateway Connect form still has no sandbox/production toggle; this session's test used direct
API/DB calls rather than that form, so it wasn't re-exercised.
