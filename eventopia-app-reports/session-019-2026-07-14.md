## Smoke test — new admin order-tracking page + PII/RBAC fix (real orders, both envs)

Since neither env's payment gateways could be exercised safely for a *paid* order today (see Findings below), both smoke tests used a **free ticket type** (sidesteps any gateway — orders with `totalMinor=0` auto-settle to `PAID`/`ISSUED` immediately, no buyer confirmation step needed):

1. Signed up a fresh test org via email OTP (reusing `hi@eventopia.my` per convention), created + published one event with a free ticket type, completed guest checkout via direct API calls. Order went straight to `PAID`, ticket `ISSUED`.
2. Created two temporary operator accounts per env (via a one-off script mirroring `bootstrap-operator.ts`, deleted after): one **`SUPPORT_OPS`** (has `Order` read) and one **`COMPLIANCE_OPS`** (the one seeded role *without* `Order` read, per the fix's own commit message).
3. As `SUPPORT_OPS`: `GET /admin/orders?organizerId=...` and the order-detail endpoint both returned the real order with a full composed timeline (`financial.settled` audit entry + `payment_intent:SUCCEEDED`); `/admin/search` returned a `ticket` result card linking to the order.
4. As `COMPLIANCE_OPS`: both `/admin/orders` endpoints returned `403 FORBIDDEN`, and the same search query returned `{"results":[]}` — confirms the PII leak is closed exactly as the commit intended.
5. **Cleanup:** deleted all rows the test created (`payment_intent`, `tickets`, `order_item`, `ledger_entry`×2, `processed_webhook_event`, `audit_log`, `orders`, `ticket_type`, `event_day`, `event`, `organizer_account`, `account`) + all Redis keys (`org:meta:*`, `org:slug:*`, `membership:acc:*`, `idem:*`), plus the 2 temp operator rows, on both envs. Verified prod's baseline row counts (8 accounts / 5 `organizer_account` / 7 events / 28 orders / 13 tickets / 7 payment_intent / 3 `organizer_payment_credential`) matched exactly before and after — zero residue. Left `operator_audit_log` untouched on both envs (tamper-evident by design, per session-017's hash-chain finding — never delete from it).

## Findings (new — both pre-existing, unrelated to this pull)

**1. Self-service "platform" collection is hard-wired to Xendit, ignoring `PAYMENTS_DEFAULT_GATEWAY=midtrans` (prod, live bug since session-017 this morning).**
`apps/api/src/modules/payments/credentials-module.ts` (`collectionRoute` handler): `const activeGateway = collector === 'platform' ? 'xendit' : collector;` — and the `organizer_account.payment_gateway` column itself defaults to `'xendit'` at the schema level. Midtrans/DOKU are only reachable via BYO-keys DIRECT mode. Xendit has only a placeholder `XENDIT_API_KEY` on prod. **Net effect: any organizer using platform-mode collection cannot collect real payments right now.**

Already bit a real event today: org `org_03f297927f184c0d`, event **"Test - Festival Mbois 11 - Malang Menyala"** (published, created 09:34), has **4 real buyer orders, all `EXPIRED`** (09:14–10:49) — the organizer even self-enabled Xendit DIRECT trying to work around it. I observed live PM2 log activity (09:41–09:46 today) of what looks like the same organizer/user actively working this exact organizer in the operator console, including the new `/admin/orders` page — very likely the user investigating this in parallel.

Per user decision this session: **documented only, not fixed** — no code change made.

**2. Prod's real Midtrans production merchant account (`M073614114`) has no payment channels activated.** Tested directly against `api.midtrans.com` with the real prod Server Key (confirmed valid via the same check the app's own `verifyCredentials()` uses): VA/bank_transfer → `402 Payment channel is not activated`, QRIS → same `402`, GoPay → `404 Merchant pop id is not found`. This is a Midtrans-dashboard-side configuration gap (needs activating in Midtrans's own merchant portal), not a code bug. DOKU not tested live (avoided creating more account-side noise). **Combined with Finding 1: as of today, no payment gateway can actually process a real payment on prod.**
