## Migration/backfill/seed analysis (pre-flight, before touching the DB)

All 19 pending migrations are additive — no drops, no bare `NOT NULL` added without a
default. Two carry real data-migration logic; both were checked **against dev's actual data
before migrating**, not assumed safe:

- **`0080_kyc_mou_check_and_direct_approval`** backfills any stray `kyc_status='PENDING'` to
  `'SUBMITTED'` before adding a CHECK constraint on the canonical vocabulary. Dev had zero
  `'PENDING'` rows and every existing `kyc_status`/`mou_status` value already fit the new
  CHECK (`SELECT kyc_status, mou_status, count(*) ... GROUP BY 1,2` showed only
  VERIFIED/MORE_INFO/SUBMITTED/UNVERIFIED/REJECTED × SIGNED/SENT/UNSIGNED — all in-vocab).
- **`0087_wa_active_phone_unique`** adds a partial unique index on
  `organizer_whatsapp_config(phone_number_id) WHERE status='ACTIVE'`. Dev had zero organizers
  with duplicate ACTIVE phone numbers.
- No reseed needed — every new table is additive/opt-in; the existing `kopikita-e2e` seed
  stayed valid as-is.
- The session-009 `pnpm db:migrate` gotcha (bare drizzle-kit CLI silently swallows
  `DATABASE_URL` under non-interactive SSH, falls back to `localhost` → `ECONNREFUSED`) still
  applies — no persisted workaround script has landed in the repo since, so a fresh one-off
  `drizzle-orm/postgres-js/migrator` script was written and run via `node --env-file=.env`,
  same as sessions 009/018/020.
