## Real Midtrans sandbox payment test (the new part — not exercised on prod)

Signed up a fresh test org via email (dev shows the OTP code directly in the UI —
`NEXT_PUBLIC_SHOW_DEV_OTP=true`, no Redis-reading needed), created + published an event
with a **paid** ticket type (Rp 10.000, quota 20).

**Finding: the self-service Payment Settings "Connect" flow for Midtrans is
production-only by design** — `apps/console/components/payments/gateway-connect-form.tsx`
hardcodes `environment: 'production'` on every save (comment: *"Keys are always
production"*), with no sandbox/production toggle exposed in the UI. Feeding it real
sandbox keys (`Mid-server-...`) through the normal Connect form causes Midtrans to
**reject the key with a 401**, because the saved credential row points the adapter at
Midtrans's *production* API instead of the sandbox one — even though the key itself is
completely valid. Confirmed this independently by curling Midtrans's real sandbox API
directly with the same key (`GET /v2/.../status` → `200`, `"Transaction doesn't exist"`,
the exact "valid key" signal the app's own `verifyCredentials()` checks for). Root-caused
by inspecting `apps/api/src/modules/payments/midtrans-config-resolver.ts` +
`credential-verifier.ts`: the adapter's base URL is driven entirely by the saved
`organizer_payment_credential.environment` column, which the console never lets you set
to `sandbox`.

**Workaround used:** connected Midtrans through the normal UI (Server/Client key +
Merchant ID — the same platform-level sandbox credentials already sitting in dev's
`.env`, since dev has real DOKU/Midtrans/Xendit sandbox creds configured, not payment
doubles), then patched `organizer_payment_credential.environment` from `production` to
`sandbox` directly in Postgres. Re-ran **Test connection** — `Midtrans accepted the
Server Key` — then selected Midtrans as the event's active collector (self-service DIRECT
mode).

**The actual test:** checked out the Rp 10.000 ticket as a guest (no account) on the
public event page, chose **Virtual Account** as the payment method, and reached a real
Midtrans-issued BCA VA (`14114683402176397315721`). Confirmed via a direct Midtrans API
call that this was a genuine sandbox transaction (`transaction_status: pending`,
real `transaction_id`). Went to **https://simulator.sandbox.midtrans.com/bca/va/index**
(Midtrans's own hosted payment simulator — not a synthetic webhook), entered the VA
number, and clicked **Pay** — `"Simulated payment is successful."`

Within seconds, two real inbound `POST /webhooks/payments/midtrans` calls landed from
Midtrans's actual sandbox servers (both `200` in the api logs) — the order flipped to
**`PAID`**, the `payment_intent` to **`SUCCEEDED`**, and the ticket to **`ISSUED`**. This
is the first time in this workspace's history that a payment-gateway integration has
been verified against the gateway's own real infrastructure rather than a signed
synthetic callback.

**Cleanup:** 13 Postgres rows (`tickets`, `order_item`, `ledger_entry`×2,
`payment_intent`, `orders`, `audit_log`, `registration_form`, `ticket_type`,
`event_day`, `event`, `organizer_payment_credential`, `organizer_account`, `account`) + 8
Redis keys (`org:meta:*`, `org:slug:*`, `membership:acc:*`, `idem:*`×2,
`pay:webhook:seen:midtrans:*`×2, `order-key:*`). Applied the `account`-row lesson from
session-017 proactively this time (found via `redis-cli keys 'membership:*'` → matched
`organizerId` → `acc_...`). Verified zero residue: 0 accounts, 0
`organizer_payment_credential` rows, and no matching Redis keys post-cleanup. The dev DB's
pre-existing seed data (`kopikita-e2e`, 14 events / 31 orders) was untouched.
