# Session 13 — Move operator console `operator.eventopia.my` → `admin.eventopia.co.id`

**Date:** 2026-07-10
**App:** `eventopia` (github.com/biji-dev/eventopia)
**Hosts:** `proxy` (`root@46.250.234.153`) — nginx + certbot; `app` (`devops@154.26.129.104` / `10.0.0.5`) — `eventopia-api` + `eventopia-operator` PM2 processes; Cloudflare — DNS + TLS (DNS-01)
**Env:** prod

Domain swap for the operator console, requested directly (not a `main` pull — no app code changed, no git HEAD move). Old hostname was on the `.my` TLD wildcard; new hostname sits on the separate `eventopia.co.id` zone alongside the marketing site. Ran concurrently with a separate session's `main` pull (its own session-012, console-only, `721de4a`→`00d7d27`) — no overlap: that session only touched `eventopia-console`, this one only touched `eventopia-api`/`eventopia-operator`/proxy/DNS. Re-verified after their redeploy that this session's changes held.

## What was done

1. **DNS** — added `admin.eventopia.co.id` A record (orange-cloud/proxied, `46.250.234.153`) in the `eventopia.co.id` Cloudflare zone via the existing DNS-scoped API token (same one used for DNS-01 certs).
2. **TLS** — expanded the `eventopia.co.id` LE cert (was `eventopia.co.id` + `www.eventopia.co.id`) to add `admin.eventopia.co.id` as a third SAN: `certbot certonly --dns-cloudflare ... -d eventopia.co.id -d www.eventopia.co.id -d admin.eventopia.co.id --expand --cert-name eventopia.co.id`. New expiry 2026-10-08.
3. **Nginx** (`proxy`) — in `eventopia-coid.conf`: added `admin.eventopia.co.id` to the `:80→https` redirect, and appended a new `:443` server block (cloned from the old operator block, same `eventopia_operator` upstream → `10.0.0.5:38400`) using the expanded cert. Removed the old `operator.eventopia.my` server block from `eventopia.conf` entirely (user's call — no one uses the old URL, no redirect kept). **Hit and fixed a self-inflicted bug**: the first `sed` pass to add the new hostname to the `:80` redirect line accidentally also matched the marketing `:443` block (identical `server_name` text), which would have shadowed the new operator block behind the marketing site (first exact-match wins in nginx). Caught before reload by re-reading the file, reverted the marketing block's `server_name` before running `nginx -t`.
4. **API CORS** — `apps/api/src/app.ts`'s CORS allows an explicit `CORS_ORIGINS` list *plus* any origin matching `TENANT_ROOT_DOMAIN` (`eventopia.my`) via regex. `admin.eventopia.co.id` doesn't match that pattern (different TLD), so it had to be added explicitly to `CORS_ORIGINS` in `~/eventopia/.env` on `app` (backed up first as `.env.bak-20260710`).
5. **PM2 gotcha hit again** (same one as session-008): `pm2 restart eventopia-api --update-env` does **not** re-read `.env` — `ecosystem.config.cjs` only re-parses it at `require()` time. First restart kept the stale `CORS_ORIGINS` (confirmed via a live CORS preflight — `access-control-allow-origin` was missing from the response for the new origin, present for `app.eventopia.my`). Fixed with `pm2 restart ecosystem.config.cjs --only eventopia-api --update-env`. `eventopia-operator` itself needed no rebuild — it has no baked-in self-referencing URL (`NEXT_PUBLIC_OPERATOR_BASE` doesn't exist; confirmed via source grep before starting).
6. Confirmed no cookie-based session in the admin app (Bearer-token auth only, token read from `NEXT_PUBLIC_API_BASE` via `'use client'` fetch) — cross-TLD move has no cookie-domain implications.

## Operator password reset (follow-on, same session)

User forgot the `hi@eventopia.my` `SUPER_ADMIN` password (originally generated + delivered out-of-band in session-006, never stored anywhere — by design). Checked read-only first: `operator_user` has exactly one row, `hi@eventopia.my`, `SUPER_ADMIN`, `ACTIVE`, password hash present. The existing `bootstrap-operator.ts` is idempotent-by-email and deliberately no-ops on an existing account, so it can't reset a password.

Wrote a companion script, `apps/api/scripts/reset-operator-password.ts` (untracked, same convention as `bootstrap-operator.ts`) — reuses the app's own `hashPassword()` (scrypt, `packages/api/src/admin/auth/password.ts`) and the `operatorRepo.update()` path so the result is guaranteed login-compatible, rather than hand-rolling a hash via raw SQL. Ran it once for `hi@eventopia.my`; confirmed via direct DB read that only `password_hash`/`updated_at` changed (email/roles/status untouched — schema has no PII-encrypted columns on this table, the `EVENTOPIA_PII_ENC_KEY not set` warning from the run is an unrelated global module-load warning, harmless here). Verified the new password live: `POST /admin/auth/login` from the `admin.eventopia.co.id` origin → `200` with a valid operator JWT. New password delivered to the user directly in-chat, not stored in this report or anywhere else.
