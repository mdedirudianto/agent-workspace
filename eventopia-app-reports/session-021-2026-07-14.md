## Current-state analysis

**`app` (10.0.0.5, KVM VPS, Ubuntu 24.04.4):** 6× AMD EPYC @ 2.0GHz, **11GB RAM** (4GB swap),
100GB disk (27-29% used). Runs **~40 PM2 processes** for the whole fleet (single user `devops`,
all fork-mode — no cluster mode anywhere), of which eventopia owns 5: `eventopia-api` `:38001`
(Bun/Hono), `eventopia-workers` (Bun, metrics `:9091`), `eventopia-web-public` `:38100`,
`eventopia-console` `:38200`, `eventopia-operator` `:38400`. **This is a proven noisy-neighbor
risk, not a hypothetical one**: app-sysadmin session-004 (2026-06-17) found `ytgrab-api` leaking
2.5GB RSS (unbounded Prometheus label cardinality), driving the host to 95% swap and a PSI IO
stall of `avg300=0.52` — that froze an *unrelated* app's (`miaw-route`) event loop for ~8 minutes
and caused public 504s. Nothing isolates eventopia's processes from a repeat of this with a
different offending app.

**`db` (10.0.0.1, KVM VPS, Ubuntu 22.04.5):** 12× AMD EPYC, **48GB RAM (~2GB used)**, 490GB disk
(27% used, dominated by an 89GB `wagrab` DB), 4GB swap. Runs Postgres 16, MariaDB 10.6.22,
MongoDB 8.0.10, ClickHouse 26.1.3.52, and Redis 8.6.1 for the whole fleet — **shared, not
dedicated to eventopia**. Postgres `max_connections=200` (default, never raised; 61/200 in use
at last audit). **PgBouncer is installed but sits disabled/unconfigured** — every app, including
eventopia, connects directly to port 5432. The only prior tuning was a systemd `LimitNOFILE`
fix (db-sysadmin session-003) that resolved a false-alarm fd-utilization alert, not a real
capacity change.

**`proxy` (10.0.0.2, KVM VPS, Ubuntu 24.04.3):** 4× AMD EPYC, **5.8GB RAM** (nginx has peaked at
2.0GB during a prior traffic burst), 96GB disk (17% used). nginx 1.24.0, 4 workers, 26+ vhosts
already fronting the whole fleet. **No nginx-level rate limiting (`limit_req`/`limit_conn`)
exists anywhere** — the only rate limiting in the entire stack is eventopia's own app-level Redis
fixed-window counter, and only on select routes (auth/OTP). `fail2ban` covers SSH only, not HTTP.

**eventopia's own architecture** (`/Users/dedi/Projects/eventopia`, currently at `507c15c` on
`dev`, prod on `0bc4d61`): pnpm/Turborepo monorepo, Bun/Hono API, Drizzle ORM over `postgres.js`.
Connection pool defaults to `PG_POOL_MAX=10` per process (api + workers ⇒ ≤20 direct connections
from eventopia today), with `PG_STATEMENT_TIMEOUT_MS=30000` / `PG_LOCK_TIMEOUT_MS=10000` /
`PG_IDLE_IN_TX_TIMEOUT_MS=30000` already designed in to bound a stuck lock's blast radius. Job
queue is **BullMQ over ioredis, ~20 queues, all inside one `eventopia-workers` process** —
`payments`, `ticket-delivery`, `wa-send`, `order-expiry`, `payment-reconcile`, etc. **No
`concurrency` option was found on any `new Worker(...)` call**, so every queue defaults to
processing one job at a time.

**The repo already has a real load-testing harness** (`scripts/load/01-on-sale-spike.js`,
`02-door-rush.js`, `03-broadcast-fanout.ts`, a custom Bun/TS `04-order-storm.ts`, orchestrated by
`run-load-tests.sh` / `order-storm.sh`), with results logged in
`scripts/load/RESULTS.md`/`ORDER-STORM-RESULTS.md`. Best prior result: 600 concurrent orders
against a quota of 200 → exactly 200 succeeded, 400 correctly rejected `QUOTA_EXCEEDED`, zero 5xx,
full financial/inventory/idempotency reconciliation green. The same run measured **p95 latency
going from 96ms at 120 concurrent racers to 587ms at 600**, tracing to a single pessimistic
`SELECT ... FOR UPDATE` row lock on the hot ticket-type row — the repo's own
`docs/testing/concurrency-and-idempotency.md` calls out a **throughput ceiling of ≈139 req/s for
one hot ticket type**, against ~5900 req/s on the uncontended idempotent-create path. **Every one
of these runs was against `localhost`/dev — none has ever exercised the real `proxy→app→db`
production path.**

Two more findings from the repo's own `docs/launch/prelaunch-review.md` (2026-06-05, never
fixed): CRM `resolveSegment`/`aggregateHistory` load **all** of an org's orders+items into app
memory with no pagination — will degrade as order volume grows toward 20k; `contacts` cursor
pagination is missing a composite `(organizer_id, id)` index. And a documentation note: the
repo's own `infra/DEPLOY.md`/`infra/docker-compose.prod.yml` describe a Docker-Compose prod
deploy that **doesn't match reality** — both dev and prod are hand-maintained bare-metal PM2.
Any DR/rollback procedure referencing those docs (`infra/DR.md`) is stale.

## Target architecture

Reuse `db`'s large idle headroom to run a second eventopia compute node, load-balanced by
`proxy`. Postgres/Redis stay a **single, unreplicated instance** — both nodes are stateless
compute pointed at the same data layer (the `db`-side node just reaches it locally instead of
over the private network):

```
Client → Cloudflare → proxy (nginx)
                         │
                 upstream "eventopia_api" (least_conn, passive health
                 checks via max_fails/fail_timeout — open-source nginx,
                 no active probes)
                         │
              ┌──────────┴──────────┐
        app:38001                db:38001
     eventopia-api            eventopia-api
     eventopia-workers        eventopia-workers
              │                       │
              └───────────┬───────────┘
                 Postgres 16 + Redis (single instance, on db;
                 app-node reaches it over 10.0.0.0/24, db-node locally)
```

This sidesteps an open question about whether Bun's PM2 integration supports true cluster mode
(`-i max`) — instead of multi-processing one box, `proxy` load-balances across two independent
processes, which is a well-understood nginx pattern. It also directly fixes the BullMQ
concurrency=1 bottleneck: BullMQ workers are designed for multiple processes to safely consume
the same queue (jobs are atomically claimed), so running `eventopia-workers` on both nodes gives
real parallelism on `payments`/`ticket-delivery`/`wa-send` without depending solely on
in-process concurrency tuning — the two techniques compound. And it buys real redundancy: if
`app` has another session-004-style noisy-neighbor incident, `proxy` can route around the wedged
node instead of eventopia going down with it.

**The one hard requirement this creates:** `db` now runs both the Postgres engine and
eventopia's compute. Unconstrained, eventopia's own processes could starve Postgres (or other
tenants' DBs) exactly the way `ytgrab-api` starved `miaw-route` on `app`. PM2 memory caps +
CPU niceness/cgroup limits on the `db`-side eventopia processes are mandatory from Phase 1, not
a later hardening pass.

## Phased plan

| Phase | Goal |
| --- | --- |
| 0 | Stand up the two-node+LB topology in an isolated staging environment (separate DB, no prod data) |
| 1 | Apply hardening on staging: PgBouncer, BullMQ worker concurrency, resource isolation on the `db`-side node, proxy rate limiting/tuning, targeted query/index fixes |
| 2 | Load test on staging: extend the repo's existing k6/order-storm harness with a sustained-sale profile + the existing door-rush burst profile; measure before/after each Phase 1 fix |
| 3 | 1-2 guarded rehearsals against real prod topology (isolated test organizer, off-peak window, kill switch), including a deliberate kill-one-node test to prove failover |
| 4 | Monitoring/alerting runbook for the live multi-week sale + event-day check-in, and a documented decision point for going beyond dual-node if the data says so |

### Phase 0 — Staging environment

Second `eventopia-api`/`eventopia-workers` pair on `db`, PG connections to a new isolated
Postgres DB + Redis index on the same box, plus a matching two-node nginx upstream (test vhost
on `proxy`, or a local equivalent for staging-only runs) with `least_conn` + passive health
checks. This is what gets iterated on in Phases 1-2 — never touches prod data or the shared
`app` box's existing footprint.

### Phase 1 — Hardening

- **DB:** re-enable and configure PgBouncer (transaction-pooling mode) — currently every app
  connects directly to Postgres; fix the CRM `resolveSegment`/`aggregateHistory` unscoped
  full-org-load query and the missing `contacts(organizer_id, id)` index (both from the repo's
  own prelaunch review, never fixed).
- **App:** explicit `concurrency` on the ~20 BullMQ `Worker()` calls for high-volume queues
  (`payments`, `ticket-delivery`, `wa-send`, `order-expiry`); **mandatory resource isolation**
  (PM2 `max_memory_restart` + `nice`/cgroup CPU limits) on the `db`-side eventopia processes so
  they can never starve Postgres or other tenants; a documented deploy/migration runbook (code
  deploys touch both boxes, migrations run from exactly one designated node).
- **Proxy:** `limit_req`/`limit_conn` for eventopia's public routes (none exists today); audit
  `worker_connections`/keepalive/buffer sizing given `proxy`'s 5.8GB RAM and its prior 2.0GB
  nginx memory peak during a burst.

### Phase 2 — Load testing on staging

Extend `scripts/load/` (k6 + order-storm) to target staging instead of `localhost`. Add a
**sustained-sale profile** (steady order arrival over hours, matching the confirmed sale
pattern) alongside the existing **door-rush** profile for event-day check-in. Verify `least_conn`
distribution is sane under load and confirm no session/local-state assumption breaks
round-robin (the API should be fully Redis/Postgres-backed — treated as a verification step, not
an assumption). Track p50/p95/p99 latency, error rate, Postgres connection count and lock-wait
time, BullMQ queue depth, and CPU/RAM/swap on every host involved. Re-run after each Phase 1 fix
to get real before/after deltas rather than a single end-state number.

### Phase 3 — Guarded production rehearsal

Once staging numbers hold, run 1-2 short rehearsals against the real `proxy→app→db` path: an
isolated, clearly-marked test organizer/event, scheduled off-peak, with an abort mechanism
(swap/PSI-stall/error-rate thresholds) to protect co-tenants on shared infra. Includes a
deliberate **kill-one-node test** (take down the `app`-side or `db`-side eventopia process
mid-rehearsal) to confirm `proxy` fails over cleanly within a defined window with no more than
in-flight-request loss. Since payment-gateway fixes are explicitly out of scope, this validates
order/checkout/inventory flow up to payment initiation, not full real-money capture.

### Phase 4 — Runbook & scale decision

Monitoring runbook for the live multi-week sale and event-day check-in (dashboards, alert
thresholds, BullMQ backlog alerts). The dual-node setup is the baseline, not a deferred
fallback — this phase documents the concrete trigger metrics for going further (e.g. a third
dedicated node, or resizing an existing box) if guarded-rehearsal or early real-sale data shows
strain, with rough cost/effort ready to act on without re-deriving from scratch.

## Success criteria

- Sustained order-flow throughput holds with p95 checkout latency in the low hundreds of ms,
  zero 5xx, zero oversell, at a realistic "worst moment" concurrency (a marketing-push spike,
  not an instant flash)
- Door-rush check-in scenario (20k attendees over a few hours on event day) handles its own
  burst profile cleanly
- No co-tenant app on `app`/`db`/`proxy` is destabilized by eventopia's load (the session-004
  ytgrab/miaw-route failure mode must not repeat, in either direction — including eventopia
  destabilizing Postgres on `db`)
- Failover from a killed node completes within a defined window with zero oversell/duplicate-
  charge risk (idempotency keys should already guarantee this — the rehearsal proves it)
- Evidence-based answer on whether the dual-node reuse-capacity approach is sufficient, or a
  dedicated server is warranted
