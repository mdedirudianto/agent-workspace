## Part 2 — Fixed nginx drift left by session-013, closed its OpenPanel follow-up

While verifying this deploy, `operator.eventopia.my/login` returned a 404. Before treating it as a bug, checked the workspace and found **session-013** (a concurrent session, same day) had already moved the operator console to `admin.eventopia.co.id` and *deliberately* retired the old URL with no redirect — the 404 was correct, expected behavior, not a regression. Two real gaps did exist, both from the same root cause:

- `proxy:/etc/nginx/sites-enabled/eventopia.conf` and `eventopia-coid.conf` had been edited directly as standalone files instead of through `sites-available` + symlink (breaking the fleet-wide `sites-enabled/* -> sites-available/*` convention every other site on this proxy follows). `sites-available/eventopia.conf` still had the old, now-dead `operator.eventopia.my` block that `sites-enabled` had already dropped — a purely cosmetic/tracking mismatch (live behavior was already correct) but a real trap for the next person reading `sites-available` expecting it to reflect reality.
- Session-013's own open-follow-ups explicitly flagged: *"`eventopia-operator`'s OpenPanel project CORS allowlist almost certainly still references the retired `operator.eventopia.my`... fix alongside [the OpenPanel] bug"* — which is exactly what Part 1 of this session fixed. Confirmed: `projects.cors` for `eventopia-operator` was still `{https://operator.eventopia.my}`.

**Correction worth noting:** my first pass at this added a `301` redirect from `operator.eventopia.my` to the new domain, reasoning it'd be a nice continuity touch — but that directly contradicted session-013's explicit, documented decision ("no redirect kept... user's call"). Caught it by reading session-013's report before writing this one, and reverted the redirect before it had been live more than a couple of minutes.

### What was done

1. Promoted the working `sites-enabled/eventopia-coid.conf` (session-013's correct, live `admin.eventopia.co.id` block) into `sites-available/eventopia-coid.conf`, making it the tracked source of truth. Backed up first (`/root/eventopia-coid.conf.pre-migration-fix-20260710.bak`).
2. Removed the dead `operator.eventopia.my` block from `sites-available/eventopia.conf` entirely, matching session-013's actual live state (backed up first as `/root/eventopia.conf.pre-migration-fix-20260710.bak`). Briefly replaced it with a redirect, then removed the redirect too once session-013's "no redirect, user's call" note was found.
3. Removed both standalone files from `sites-enabled` and restored proper symlinks to `sites-available`.
4. `nginx -t` (clean) → `systemctl reload nginx`, twice (once after the accidental redirect, once after removing it).
5. Updated OpenPanel: `UPDATE projects SET cors = ARRAY['https://admin.eventopia.co.id'] WHERE id = 'eventopia-operator'` on `db`.
