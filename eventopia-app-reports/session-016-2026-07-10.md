Largest pull since session-009 (48 commits, `38b3b3c`→`7b49712`), three feature merges: DOKU BYO-keys hardening (a frozen-`collection_mode` webhook-verification fix closing a latent money-loss bug, applied to all three gateways), a new **buyer checkout verification layer** (reverse-WhatsApp + email magic-link, alongside the existing email-OTP), and **Xendit bring-your-own-keys** (operator-gated DIRECT collection, mirroring the DOKU BYO pattern). First redeploy since the platform went genuinely live — prod now holds real user data (orders, accounts, events), so this session added a `pg_dump` backup step that prior sessions skipped when tables were empty.

## What changed upstream

| Area | Fix/feature | Notes |
| --- | --- | --- |
| Payments (all 3 gateways) | Frozen-`collection_mode` webhook verification | Selecting which token/secret verifies an inbound notification now uses the intent's frozen collection mode at collect time, not the organizer's current gateway toggle — closes a race where revoking/switching BYO keys mid-flight could 401 a real, already-captured payment |
| DOKU | Refund guard on DIRECT orders, webhook empty-token hardening | DIRECT-collected orders settle into the organizer's own account and can't be refunded through the platform ledger — both the refund route and initiator now refuse it (409) |
| Buyer checkout | Reverse-WhatsApp verification (`BuyerWaVerify.tsx`) | Buyer opens WhatsApp pre-filled with a code, sends it to the Eventopia number; inbound webhook binds the sender, checkout tab polls and auto-completes. No template needed (unlike forward-OTP) |
| Buyer checkout | Email magic-link verification (`EmailVerifyPanel.tsx`) | Default email-tab method now; buyer clicks a one-time link (can be on another device), originating tab polls and completes. Falls back to the existing OTP-code method via "receive a code instead" |
| Payments | Xendit bring-your-own-keys | Operator-gated DIRECT collection (`organizer_account.xendit_direct_approved`), mirrors DOKU's BYO pattern; ships dark on prod (see below) |
| DB | 2 new migrations (0075–0076) | Widen `organizer_payment_credential.provider` CHECK to include `xendit`, add `organizer_account.xendit_direct_approved boolean NOT NULL DEFAULT false` — both additive, no backfill |

**No new required env vars** (verified by diffing `.env.example` and grepping every new `process.env.*` reference against what's already set on prod) and **zero lockfile changes** (`pnpm install` was a no-op).

**Confirmed safe to ship Xendit dark:** prod's `PAYMENTS_DEFAULT_GATEWAY=midtrans` and `XENDIT_API_KEY` is still a placeholder (`REPLACE_ME_XENDIT_AP...`) — the new code's boot-time fail-closed check (`apps/api/src/modules/payments/provider-registry.ts:243`) only throws when Xendit is the *default* gateway, which it isn't. No crash-loop risk.

**Pre-existing fossil found and fixed:** `WEB_DATA_SOURCE=api` was live only inside PM2's saved process env (`~/.pm2/dump.pm2`) — session-007 removed it from `.env` believing it dead, not realizing it wasn't disk-backed (PM2 merges rather than replaces env on `--update-env`, so the original value from session-001/002 never actually cleared). It was silently load-bearing for every checkout code path this whole time. Re-added explicitly to `.env` this session with an explanatory comment so a future `pm2 delete`+`start` can't silently lose it and fall back to mock checkout behavior.

## Smoke test: new buyer verification methods (live, real orders)

Signed up a fresh test org via email (`SMOKE-TEST Session-016 Org`, reusing the team `hi@eventopia.my` mailbox per established convention), created + published one event with a free ticket type, then drove **both new verification methods** through real orders:

| Method | How it was exercised | Result |
| --- | --- | --- |
| Reverse-WhatsApp | Started the challenge via the real checkout UI, then — since prod has no dev-simulate endpoint (correctly gated off) and no real phone was available — POSTed a **HMAC-signed synthetic inbound webhook** (`x-hub-signature-256`, real `META_APP_SECRET`) to the API's own `/webhooks/whatsapp`, mirroring session-008's proven technique. The checkout tab's poll picked up `verified` within seconds and **auto-completed the order with no further input** | Order `PAID`, ticket `ISSUED` |
| Email magic-link | Requested via the real checkout UI; read the link token from `email-verify:token:*` in Redis (same class of technique as reading OTP codes in sessions 010/014/015); loaded `/verifikasi-email?token=…` and clicked confirm — matching real buyer behavior of opening the link on another device. Confirmed `status`/`complete` via the real API (the original checkout tab can't be "returned to" from an agent — completed the placement via the same API calls the tab would have made, with the resulting attendee token) | Order `PAID`, ticket `ISSUED`, `email_delivery_status=SENT` |

**Incidental finding (not a bug):** the email magic-link's `/verifikasi-email` route, when visited on bare `eventopia.my` (no subdomain), 301s through to `eventopia.co.id` before rendering — harmless extra hop from the existing apex-redirect rule (session-003), since web-public is the same instance behind both hosts.

**Cleanup:** deleted both test orders + tickets, the registration form, ticket type, event day, event, and account (9 Postgres rows total across `tickets`/`order_item`/`orders`/`form_field`/`registration_form`/`ticket_type`/`event_day`/`event`/`account`), plus 6 Redis keys (`org:meta:*`, `org:slug:*`, 2× `membership:*`, 2× `idem:*`). The `wa-verify:*`/`email-verify:*` challenge keys were already self-cleaned by `consume()`/`complete()` — no manual deletion needed there. Verified row counts match pre-session + organic growth only (7/7/22 accounts/events/orders post-cleanup vs 6/7/21 pre-session — the delta is entirely new real signups/orders during the session, confirmed by inspecting the newest rows, none matching test naming).
