# Session 8 — Register eventopia's WABA with `miaw-route` (fix dead webhook)

**Date:** 2026-07-10
**App:** `eventopia` (github.com/biji-dev/eventopia)
**Hosts:** `app` (`devops@154.26.129.104` / `10.0.0.5`) — `eventopia-api`; `db` (`10.0.0.1`) — `miaw_route` Postgres; public router `routes.biji.uk` (systemd `miaw-route` on `app`)
**Env:** prod

Closes the "register eventopia with `miaw-route`" follow-up open since session-006/007. Investigation found the problem was worse than assumed: eventopia's prod WABA (`2271807923577496` / `wa.me/6285178123275`, phone `1187552854445262`) was subscribed to **no Meta app at all** — its dedicated app's own webhook config still pointed at a dead `local.biji.uk` tunnel (502). Zero inbound WhatsApp traffic (messages, status callbacks) was reaching eventopia.

## Approach — reused `BIJI Dev`, not a new dedicated `apps` row

Session-007 (a concurrent session, see below) proposed registering eventopia's *own* Meta app (`307017735009770`) as a new `apps` row in `miaw-route`. This session took a different path after checking `GET /{waba-id}/subscribed_apps` and finding it empty: rather than wire up a second app, the WABA was subscribed directly to the already-working shared **BIJI Dev** app (`1125824895132994`) — the same app wabacall/pasarbesar/grobiz already route through. This needed zero new `miaw-route` infrastructure (no new `apps` row, no new verify_token, no Meta app-level webhook change — BIJI Dev's callback already points at `routes.biji.uk`), just a `routes` row plus fixing eventopia to hold BIJI Dev's secret instead of its own.

## What was done

1. **Backed up** `~/eventopia/.env` on `app`, then corrected `META_APP_ID` → `1125824895132994` and `META_APP_SECRET` → BIJI Dev's secret (pulled from `miaw_route.apps` on `db`), replacing the values for eventopia's unused dedicated app.
2. **Subscribed the WABA to BIJI Dev**: `POST /{waba-id}/subscribed_apps` via Graph API (eventopia's existing access token already had `whatsapp_business_management` rights on the WABA) → `{"success":true}`.
3. **Added a `routes` row** in `miaw_route` on `db`: `name='Eventopia - PROD'`, `phone_number_id=1187552854445262`, `waba_id=2271807923577496`, `backend_url='http://10.0.0.5:38001/webhooks/whatsapp'`, `app_id=<BIJI Dev>`, `fan_out=false`. Reloaded miaw-route's route cache (`POST /api/routes/reload`, admin key from session-001).
4. **PM2 gotcha hit and fixed:** `pm2 restart eventopia-api --update-env` does **not** re-read `.env` — `ecosystem.config.cjs` only re-parses it when the config file itself reloads (`const E = parseEnv(...)` runs at `require()` time). The first restart kept the stale (wrong) secret, causing signed test webhooks to fail `WEBHOOK_SIGNATURE_INVALID`. Fixed with `pm2 restart ecosystem.config.cjs --only eventopia-api --update-env`.
5. **Verified end-to-end** with HMAC-signed synthetic webhooks (own script, `x-hub-signature-256` computed with BIJI Dev's real secret) posted to the public `routes.biji.uk/webhook/d7745f14-…` URL: `webhook_logs` shows `status=success`, `http_status=200`, forwarded to `Eventopia - PROD` → eventopia's backend.

## Concurrent session collision (worth flagging)

While this was in progress, a **separate** Claude Code session pulled eventopia's `main` forward on both `dev` and `prod` (`2e3994a`→`3f2c631`, its own session-007) and deployed an upstream refactor that — coincidentally — formalizes exactly this pattern: it removed the now-dead `GET /webhooks/whatsapp` verify-handshake route and updated the module's own doc comment to describe miaw-route ownership of the handshake. Their `.env` cleanup dropped `META_WEBHOOK_VERIFY_TOKEN`/`META_WABA_ID`/`META_BUSINESS_ACCOUNT_ID`/`WEB_DATA_SOURCE` (none of which this session touched) and left `META_APP_ID`/`META_APP_SECRET` alone, so this session's fix survived their redeploy untouched. Re-ran the signed-webhook test **after** their redeploy (HEAD `3f2c631`, restart count flat at 65) to confirm compatibility — still `status=success`.

Their session-007 report's proposed miaw-route registration (new dedicated `apps` row for `307017735009770`) was superseded before it was ever executed — the actual live config is the BIJI Dev reuse described above.
