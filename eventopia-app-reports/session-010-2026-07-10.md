**Date:** 2026-07-10
**App:** `eventopia` (github.com/biji-dev/eventopia, commit `34a5385` on prod — unchanged this session)
**Hosts:** `app` (`devops@154.26.129.104` / `10.0.0.5`) — 5 PM2 procs; `db` (`root@109.123.233.215` / `10.0.0.1`) — Postgres + Redis db6; `monitor` (`10.0.0.3`) — GlitchTip + OpenPanel queried read-only

First-ever real organizer journey exercised on `eventopia.my` prod: email-OTP signup → event creation → FREE ticket type → publish → buyer checkout → order PAID via the free-order path → ticket ISSUED. Before this session, `account`/`organizer_account`/`event`/`orders` were all **zero rows** on prod — this was genuinely the first live use of the app-user side of the platform (the only prior real identity was the operator-side `hi@eventopia.my` SUPER_ADMIN bootstrapped in session-006, a separate identity system). ## Why email, not the documented dev flow

`docs/launch/dress-rehearsal.md` / `scripts/r1-rc-proof.sh` assume a dev-only shortcut (`devCode` in the OTP response body) that prod explicitly disables (`NODE_ENV==='production'` gate, `apps/api/src/platform/auth/otp.ts:95`). Read the actual code instead of guessing the workaround:

- OTP codes live in Redis **plaintext**, 5-minute TTL. Phone: `otp:<phone>`. Email: `otp:email:<canonical>` (canonicalization = trim+lowercase for non-Gmail domains — `hi@eventopia.my` needs no folding).
- Confirmed by reading `apps/api/src/platform/auth/otp.ts` + `container.ts`, not by inference from the dev scripts.
- Reading the code from Redis only solves "how do I know the code" — production's `OtpSender` still enqueues a **real** send (WhatsApp via the live WABA, or email via the now-verified Resend domain) for whatever identifier is used. This shaped every decision below.

Real account row created (`acc_dc3a3c2d-…`), real organizer (`org_c08e9d28668b4b70`).
2. **Event creation** — through the actual 5-step console wizard (not the API): "SMOKE-TEST Prod Smoke Test Event", one day (2026-07-25, 10:00–22:00 WIB), one FREE ticket type ("Free Entry (Smoke Test)", quota 10), empty registration form, published via the real confirm-dialog flow.
3. **Public page** — `https://organizer-baru.eventopia.my/smoke-test-prod-smoke-test-event` rendered correctly (title, schedule, ticket selector); GA4 tag `G-8YBGDDMSQ3` confirmed present in served HTML.
4. **Buyer checkout — routed around a real blocker (see Finding 2)**: the actual public checkout form requires a WhatsApp number with no email-OTP alternative, and submitting it fires a **real** WhatsApp OTP send via the live WABA (confirmed by tracing `apps/web-public/components/event/RegisterCheckout.tsx` → `lib/checkout/actions.ts` → `apps/api/src/platform/whatsapp/otp-sender.ts`, a genuine `wa-send` BullMQ job, not a stub). Rather than fabricate a phone number (which could belong to a real stranger) or ask to borrow one, exercised the identical backend path directly: `POST /v1/auth/email/request` + Redis-read code + `POST /v1/auth/email/verify` with `intent:'attendee', organizerSlug:'organizer-baru'` (same `hi@eventopia.my`) → `POST /v1/events-orders` (needs an `Idempotency-Key` header, undocumented in the old e2e helper) → order `ord_0200cef1-…`, `PENDING`, `totalMinor:0`.
5. **Free-order path** — polled `GET /v1/events-orders/:id`; flipped `PENDING`→`PAID` in ~2s (the immediate fire-and-forget settlement path, not the 2-minute sweep backstop). `tickets` row `ISSUED`, `delivery_status=PENDING` (no phone given, expected), `email_delivery_status=ENQUEUED`.
6. **Cleanup** — deleted in FK-safe order inside one transaction (`tickets` → `orders` → `event`, cascading `event_day`/`ticket_type`/`order_item` → `account`), then 5 Redis keys (`org:meta:*`, `org:slug:*`, 2× `membership:*`, 1× `idem:*`). Verified: `account`/`event`/`orders` back to `0` rows; public page now `404`; no leftover Redis keys under any of the test's identifiers.

## Recommendation: turn this into a reusable prod-safe smoke-test script?

**Yes, for the organizer→event→free-order path — no, for the buyer-checkout path as currently built.**

What made this safe and repeatable:
- Email OTP + Redis-key-read is clean, fast (didn't need the 2-minute sweep even once), and requires no fabricated PII.
- The only two non-obvious gotchas versus the old dev-only e2e helpers: `POST /v1/events-orders` needs an `Idempotency-Key` header (undocumented in `scripts/e2e/helpers/api.ts`), and email OTP is a genuinely separate endpoint pair (`/v1/auth/email/request` + `/v1/auth/email/verify`), not a variant of the phone route.
- Cleanup is mechanical and verifiable: delete `tickets`→`orders`→`event` (cascades handle the rest) →`account`, then 5 well-known Redis key patterns, then re-assert row counts back to baseline.

What blocks fully automating the buyer side: there is no safe identifier for attendee checkout through the real UI — it's WhatsApp-only and always sends. A script could still drive the **backend** attendee path over email (as this session did) to prove the money/ticketing pipeline, but that skips exercising the actual public-facing checkout form and its WA-OTP screen. Recommend either (a) accept that gap and let the script cover organizer+event+free-order+backend-attendee only, clearly documented as not touching the WA UI, or (b) file a product ask for an email-OTP alternative on buyer checkout (Finding 2), which would also fix a real accessibility gap for non-WhatsApp buyers.

Suggested shape: `scripts/prod-smoke-test.sh`, sibling to `r1-rc-proof.sh`/`r2-rc-proof.sh` but explicitly prod-aware (refuses to run unless pointed at a prod-like `API_BASE_URL`, forces FREE-tier-only ticket types, no PAID path ever), driven by an env-configured team mailbox, with the same delete-and-reconcile cleanup pattern used manually here.
