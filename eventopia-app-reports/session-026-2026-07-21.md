# Session 26 — Unblock BRImo sandbox testing on dev (missing `BRIMO_SECRET_KEY`, gateway pin)

**Date:** 2026-07-21
**App:** `eventopia` (github.com/biji-dev/eventopia, branch `main`)
**Hosts:** `dev` (`deploy@154.26.130.96`) — `.env` edited, 2 PM2 procs restarted twice, no rebuild, no migration
**Env:** dev only

Follow-up to session-025 (which closed dev's 344-commit gap and landed the BRImo code, but
did not activate it). A separate agent session, working on dev to test BRImo, hit
`503 BRIMO_NOT_CONFIGURED` — `BRIMO_SECRET_KEY` was absent from dev's `.env`, and separately
`PAYMENTS_DEFAULT_GATEWAY=doku` on dev meant an unpinned test org would 422 on a BRImo collect
(`BRIMO_CAPABLE_GATEWAYS = ['xendit','midtrans']`, DOKU can't mint a BRI VA).

## What was checked first

- **Xendit sandbox credentials — where do they live?** Confirmed dev already has a real,
  working sandbox key (`XENDIT_API_KEY=xnd_development_…`, plus `XENDIT_WEBHOOK_TOKEN`, both
  non-placeholder). Checked prod (`app:~/eventopia/.env`) too, per the user's request — it has
  **neither var at all**, not even a placeholder, confirming the existing README note is still
  accurate. Nothing to pull from prod; dev was already self-sufficient.
- **Is `BRIMO_SECRET_KEY` actually secret?** No — it's BRI's own **publicly published**
  integration-guide dev key (`docs/engineering/brimo-guide.md` has it in plaintext in the
  committed repo, `b2f8d243cf2bf86afed17aaa3c27410e`). The local machine's root `.env` already
  had this exact value (33-byte match, confirmed by length before reading it). Safe to reuse
  as-is, not environment-specific.

## What was done

1. Backed up dev `.env` (`.bak-20260721-1310`-ish, pre-edit snapshot).
2. Appended `BRIMO_SECRET_KEY` (piped directly local→remote via `grep | ssh cat >>`, never
   echoed in any command output) and `BRIMO_ENVIRONMENT=dev` to dev `.env`.
3. **First pass (later reverted — see below):** flipped `PAYMENTS_DEFAULT_GATEWAY` dev-wide
   `doku`→`xendit`, restarted `eventopia-api`+`eventopia-workers --update-env`. Flagged to the
   user that this affected every unpinned organizer on dev, not just a BRImo test org, and that
   `docs/launch/brimo-sandbox-validation.md` actually recommends pinning a single test org
   instead. User confirmed: scope it back to one org.
4. **Chose a test org.** Queried all 22 organizers on dev (`organizer_account` joined to
   `event` counts). Most are session-005's named showcase seed seed data (0 events, no gateway
   pin, varied statuses for operator-console UI testing — not usable for a real payment test).
   Five have real event data from actual test sessions; one, **`org_3c6828c4cdd245af`**
   ("Festival Mbois 11 - Malang Menyala", created 2026-07-14, 2 events both `PUBLISHED`), was
   **already** `payment_gateway='xendit'` + `xendit_direct_approved=true` — no DB change
   needed, it was already correctly configured for exactly this test. Confirmed
   `organizer_account.status` (`PENDING` for this org) doesn't gate payment collection in the
   payments module — the capability ladder (KYC/DIRECT-approval) does, and this org already
   clears it.
5. **Reverted** `PAYMENTS_DEFAULT_GATEWAY` back to `doku` (dev-wide default unaffected for
   every other organizer), restarted `eventopia-api`+`eventopia-workers --update-env` again.
