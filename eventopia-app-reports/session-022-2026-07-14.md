## `org_03f297927f184c0d` (Malang Creative Fusion) — the user's explicit ask to verify

Migration correctly grandfathered this organizer into `organizer_direct_approval`
(`APPROVED`, `xendit`) — `payment_gateway`, `xendit_direct_approved`, and their enabled BYO
credential all came through the migration untouched.

**First live test (real guest checkout against their published "Test - Festival Mbois 11"
event, reusing their own "Test 50 Rupiah" ticket type) failed immediately:**
`ORGANIZER_CANNOT_COLLECT` — KYC `UNVERIFIED`. This organizer was manually operator-approved
for DIRECT collection in an earlier, pre-ladder session and never went through a formal KYC
submission; the new ladder correctly does not forge a KYC status that was never real (the
grandfather migration only backfills the DIRECT-approval table, by design).

**Resolved this session, at the account owner's explicit instruction** (real bank info to
follow later): captured **placeholder** bank details
(`bankCode=PLACEHOLDER`, `accountNumber=PENDING-REAL-BANK-INFO`,
`accountName="Malang Creative Fusion (placeholder, pending real bank info)"`) via a one-off
script mirroring `POST /v1/payouts/kyc` exactly (same repo calls, same audit-log shape),
moving `kyc_status` `UNVERIFIED→SUBMITTED`. Then approved through the **real, audited**
operator endpoint (`POST /admin/onboarding/kyc/{id}/decision`, `hi@eventopia.my`
`SUPER_ADMIN` — password reset per the session-013 precedent, forgotten/rotated) →
`kyc_status=VERIFIED`, `payoutEligible=true`, recorded in `operator_audit_log` as
`kyc.approve`.

Order creation then succeeded. **Payment creation (`POST /v1/payments`) failed 3/3 times**
with `503 PAYMENT_GATEWAY_UNAVAILABLE`, including after a fresh `eventopia-api` restart
(ruling out a stale connection-pool hypothesis). A standalone diagnostic reproducing
`resolveXenditConfig` + `createXenditProvider().createPayment()` directly **succeeded** —
real Xendit QRIS intent created, proving the organizer's credential, decryption, and network
path all work.

**Root cause, traced to `packages/payments/src/capability.ts`:**

```
canUseDirect = canSellPaid && directApproved[gateway] && enabledCredential[gateway] && gatewayAvailable[gateway]
```

`gatewayAvailable[gateway]` reflects whether the **platform's own** credentials for that
gateway are configured (`registry.isConfigured`) — not the organizer's. Prod's platform
Xendit key is still the deferred placeholder (deploy-doc Step 1, intentionally not done this
session — see below), so `gatewayAvailable['xendit'] = false` platform-wide. This forces
`canUseDirect = false` for **every** Xendit-DIRECT organizer regardless of their own valid
BYO key and operator approval. `resolveByName` (`provider-registry.ts`) then falls through to
the boot-registered platform Xendit provider, which is unconfigured →
`GatewayNotConfiguredError` → the 503.

**Net effect: no Xendit-DIRECT organizer — including this one — can currently collect a real
payment on prod**, until either the platform's real Xendit keys land (deploy-doc Step 1) or
the `gatewayAvailable` gate is reconsidered for the DIRECT case. This is a stronger
dependency than the deploy doc implies (it frames Step 1 as unblocking *platform-mode*
revenue specifically; it turns out to also gate DIRECT-mode via this formula). **User decision
this session: document only, no code change.**
