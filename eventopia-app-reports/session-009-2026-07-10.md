## Key finding — object storage ACL conflict (flagged, not fixed)

The new S3 adapter sends no per-object ACL on upload, so the DO Spaces bucket's default ACL applies uniformly to *everything* — public cover/poster images and private KYC/MoU/certificate documents alike. Making the bucket public-read to fix public images would also make KYC documents world-readable by key (UUID-based, so not trivially enumerable, but not the intended access model either). **Left the bucket as-is this session** — public cover images stay broken (same as before, no regression) until a follow-up code change adds prefix-conditional `x-amz-acl`.

**Gotcha hit: GlitchTip DSN format.** Creating `ProjectKey` via the Django ORM shell returns `public_key` as a Python `uuid.UUID` — printing it directly (dashed, 36 chars) produces a DSN the Sentry SDK rejects ("Invalid Sentry Dsn"), which crash-safe-degraded to errors-disabled on the first restart (api/workers logged the error but stayed online — not a crash-loop, just silent). The rest of the fleet's working GlitchTip DSNs all use the dash-less 32-char hex (`public_key.hex`). Fixed by correcting all 3 DSNs in `.env` and **rebuilding all 4 frontends** (web-public/console/admin/checkin — the DSN is baked in at build time) plus restarting api/workers.

**Gotcha hit: shared root `.env`, but admin needs its own OpenPanel client id.** `apps/console/.env` and `apps/admin/.env` are both symlinks to the same root `.env`, and both read the identical env var name `NEXT_PUBLIC_OPENPANEL_CLIENT_ID` — but the app's own doc requires admin ≠ console (operator behaviour must never mix with organizer behaviour). Root `.env` carries console's client id; admin was built with an explicit shell override (`NEXT_PUBLIC_OPENPANEL_CLIENT_ID=<admin-id> pnpm run build`) so its bundle bakes in its own distinct id. Confirmed by grepping the built `.next/static` chunks for each id.

**Known gap, not resolved this session: OpenObserve receives nothing from the app.** `.env` is correct (confirmed via `pm2 env`, `NODE_ENV=production`), and a manual `curl` straight to the ingest endpoint with the same auth succeeded (200, 1 doc) — so the infra side works. But 25s after multiple `/readyz` hits, `eventopia_api`/`eventopia_checkin_sync_backlog`/`eventopia_queue_depth` streams all still show 0 docs, and neither api nor workers logs mention OTel/tracing initialization at all (no errors either — just silent). Likely cause: `@opentelemetry/sdk-node` auto-instrumentation and/or the custom pino OpenObserve transport may not fully activate under **Bun** (api/workers run via `bun run`, not `node`) — worth a follow-up investigation into Bun compatibility. Errors (GlitchTip) and analytics (OpenPanel/GA4) are unaffected — this only concerns OpenObserve logs/traces/metrics.

## Migration DB reliability gotcha (new finding, worth carrying forward)

`pnpm db:migrate` (`drizzle-kit migrate`) **silently failed twice** with no usable error output — the CLI's spinner UI swallows the underlying exception under a non-interactive SSH pipe (confirmed reproducible with `CI=true`, `NO_COLOR=1`, and even under a pseudo-TTY via `script`). Root-caused by writing a standalone script using `drizzle-orm/postgres-js/migrator` directly: the *real* problem was that neither `drizzle-kit` (bare `node_modules/.bin/drizzle-kit migrate`) nor a plain `pnpm db:migrate` picks up `DATABASE_URL` from `.env` automatically — it falls back to `drizzle.config.ts`'s dev default (`postgresql://eventopia:eventopia@localhost:5432/eventopia`), which `ECONNREFUSED`s since the DB is remote. Bash `source ../../.env` doesn't work either — this `.env` now has unquoted values containing `<`/`>`/spaces (`EMAIL_FROM=Eventopia <tiket@eventopia.my>`), which bash's `source` chokes on (`syntax error near unexpected token`) even though the app's own dotenv-style parsers handle it fine. **Working fix: `node --env-file=../../.env <script-using-drizzle-orm-migrator>`** (Node's native `--env-file` flag uses permissive dotenv parsing, unlike bash). Migrations 0072-0074 applied cleanly this way, verified against the DB directly (`75` rows in `drizzle.__drizzle_migrations`, `account` table + RLS policy + `tickets.email_delivery_status` all present). **Recommendation for next session:** don't trust a clean `pnpm db:migrate` exit under SSH without independently checking the migration count in the DB — this failure mode passes `pnpm install` and `git merge` cleanly and only manifests at the migrate step, and its silence makes it easy to miss.

## Build gotcha: bun not on PATH over non-interactive SSH

`apps/api`'s build script (`bun run scripts/gen-openapi.ts && tsc --noEmit`) failed with `bun: not found` under the full `pnpm build` (turbo) run — `~/.bashrc`'s `BUN_INSTALL` PATH export is skipped for non-interactive SSH shells. Fixed with `export PATH="$HOME/.bun/bin:$PATH"` before build commands.

## Build gotcha: `tsc --noEmit` OOM under full parallel `pnpm build`

Running `pnpm build` (turbo, all 5 apps in parallel) OOM'd `apps/api`'s `tsc --noEmit` step (`FATAL ERROR: JavaScript heap out of memory`) — 5 concurrent heavy build processes competing for the server's 11GB RAM. web-public/console/admin/checkin all succeeded in the same run; only api's type-check exhausted its default V8 heap. Fixed by rebuilding api alone with `NODE_OPTIONS='--max-old-space-size=4096'` — succeeded cleanly (exit 0, no errors). Not needed for api/workers to *run* (Bun executes TS directly, no build artifact required at runtime) — this is a type-check gate only, but worth clearing for confidence.
