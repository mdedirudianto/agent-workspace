## Part 2 — Custom domains: Cloudflare setup + **Enterprise blocker**

The prod `proxy` nginx vhost (`eventopia-custom-domains.conf`) and CF-IP allowlist were already installed (undocumented, likely a prior session). This session added, via the user-supplied CF API token (Zone→SSL:Edit on `eventopia.my`):

1. DNS `custom.eventopia.my A <proxy IP>` **proxied** — the fallback-origin check first failed ("DNS records not setup correctly") because a wildcard `*.eventopia.my` masked the missing explicit record; user added the explicit proxied A record → Fallback Origin went **`active`**.
2. Confirmed the three env vars on `app`.

**Blocked, not live.** A real end-to-end test (`event.biji.uk`, org's own subdomain) passed **TXT ownership verification** but failed at cert issuance:

> "Access to setting a custom origin SNI has not been granted for this zone or for this account."

Confirmed against Cloudflare's current docs: **`custom_origin_sni` (SNI Rewrite for Custom Origin) is Enterprise-exclusive.** The `eventopia.my` zone is on the **Free** plan. `custom_origin_server` alone is Free/Pro/Business, but the SNI-rewrite half — which is load-bearing for this feature (Full-strict origin behind our `*.eventopia.my` cert) — is not. **This is a plan/billing gate, not a code bug.** Options documented in `docs/engineering/custom-domains.md` (committed `f104ea1`): (a) Cloudflare Enterprise (sales-negotiated, not self-serve), or (b) revert to the repo's prior per-domain-cert approach (Caddy on-demand TLS, `infra/caddy/` — deleted in this same merge, recoverable from git). **Undecided; user to choose.** The provisioner has **no prod guard** — it silently falls back to the fake impl when `CF_*` unset, so the console "Custom Domains" UI is live but no organizer can reach `Live`.

## Part 5 — Landing-page UI fixes (feature code, committed `1749989` + pushed)

Driving the live Mbois page on desktop + mobile via Playwright exposed three real bugs in the **shared** landing-page feature (fixed in `apps/web-public` + `packages/contracts`, deployed, committed, pushed to `origin/main`, prod git reconciled clean):

| Fix | Detail |
| --- | --- |
| **Scroll-reveal hid content** | `.lp-reveal` set `opacity:0` and relied on `animation-timeline: view()` (`animation-range: entry 0% cover 30%`) to fade back. An element **taller than the viewport** (the tickets block, the map) can never satisfy an entry/cover range → **permanently `opacity:0` on Chromium** → buyers couldn't see the tickets. Proven live. Now **transform-only** (slides, never hides). |
| **Text too small** | `.lp-eyebrow` `0.7rem`→`0.8rem`; countdown unit labels bumped (were ~11px). |
| **Ugly CSS orb → real artwork** | Removed the built-in CSS "plasma orb" motif; added a generic optional hero **`sideImage`** field. Set the Mbois hero to the festival's own **glowing purple orb** (pulled from the original site, optimized 2.4MB→689KB, self-hosted at `apps/web-public/public/lp/hero-orb-mbois.png`). Assets served from `/lp/*` — excluded from the tenant rewrite in `proxy.ts` (and `sharp` isn't in the standalone build, so `/_next/image` can't optimize them; referenced by raw path). |

**Gotcha (worth carrying):** the public page's `PageDoc` is parsed in **`apps/api`** (`page-port.ts` → `parsePageDocLenient`), not web-public. The api runs **Bun source in-memory**, so a new contracts field (`sideImage`) is stripped until **`eventopia-api` is restarted** — rebuilding web-public alone wasn't enough. A clean `.next` rebuild was also needed once (Next cached the transpiled contracts). Verified final: 17 blocks + orb render on desktop (orb hidden `<lg` by design), text legible, all sections visible.
