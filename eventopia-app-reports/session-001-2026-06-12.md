## Migrations (the drizzle-kit gotcha)

`drizzle-kit migrate` **exits 1 with no error printed** (it swallows the pg error). Root
cause: migration `0002` does `CREATE ROLE app_tenant` and `eventopia` (a plain role, not
the container superuser the Docker deploy assumes) lacked the privilege. Applied the 31
migrations directly via `psql -1 -f <file>` in journal order (`psql` treats
`--> statement-breakpoint` as a comment), recording each in `drizzle.__drizzle_migrations`.
Result: **31 migrations, 78 tables, 76 RLS policies.**

## Seed

`scripts/e2e/seed-e2e.ts` (API-driven; needs api+workers+pg+redis up + fake payments) →
org **`kopikita-e2e`** (stable per `SEED_ORG_PHONE=+628110000100`), upcoming/free/past

## TLS

Let's Encrypt **HTTP-01** (grey DNS reaches origin), one multi-SAN cert **`eventopia-dev`**:
`dev` · `www` · `api` · `app` · `checkin` · `kopikita-e2e`.dev.eventopia.my. Auto-renew scheduled
(expires 2026-09-10). nginx `/var/www/eventopia-acme` webroot kept for renewals.

> **Wildcard-TLS limitation:** no Cloudflare DNS-01 token, so only the issued SANs have valid
> certs. A **new** organizer slug created during testing serves a cert-name mismatch until
> issued. One-liner to add one:
> ```
> certbot certonly --webroot -w /var/www/eventopia-acme -n --cert-name eventopia-dev \
>   --expand -d dev.eventopia.my -d www.dev.eventopia.my -d api.dev.eventopia.my \
>   -d app.dev.eventopia.my -d checkin.dev.eventopia.my -d kopikita-e2e.dev.eventopia.my \
>   -d <newslug>.dev.eventopia.my && systemctl reload nginx
> ```
> Or drop a scoped CF token and switch to a real `*.dev.eventopia.my` DNS-01 wildcard.

## Findings (the load-bearing ones)

1. **`eventopia` needs `BYPASSRLS`.** The app's tenancy model assumes the connection role
   is the DB **superuser** — platform/cross-tenant queries run with RLS bypassed, and only
   tenant queries drop to `SET LOCAL ROLE app_tenant` (which re-enables RLS). In Docker the
   container's `POSTGRES_USER=eventopia` *is* superuser. On the shared cluster `eventopia`
   is a plain owner, and `FORCE ROW LEVEL SECURITY` blocked the settlement worker's pre-tenant
   `resolveIntentTenant` lookup → "unknown provider_payment_id" → settlement `ignored` → seed
   failed. Fix = `ALTER ROLE eventopia BYPASSRLS` (NOT full superuser; `app_tenant` has no
   bypass, so tenant isolation still holds).
2. **web-public block sends NO `X-Forwarded-Proto`/`X-Forwarded-Host`.** With `XFP=https` on an
   HTTP origin, the Next-standalone middleware rewrite (`<org>` → `/_t/<sub>`) re-proxies to
   `https://localhost:38100` → `EPROTO` → 500 on every tenant page. TLS is terminated at nginx;
   web-public's absolute URLs come from `PUBLIC_URL`/`NEXT_PUBLIC_*` (https), so http-internal
   is fine. (api/app keep the forwarded headers.)
3. **`.env.example` is partly stale** — ticket key env names (`TICKET_TOKEN_*`) and the docker
   `postgres` host are not what the code reads. Always cross-check against `process.env.*` usage.
