## The multi-tenant backfill problem (the crux)
Prod is **per-tenant ERPNext mode** (`ERPNEXT_SITE_NAME` unset). 4 active tenants, each its own Frappe site (3 still on pre-claim-rename `pool-` site names → backfills key off DB `site_name`):

Two backfills ship in the range; they behave **differently** on prod:

1. **`backfill:loyalty-permission`** — natively per-tenant aware (loops tenants, builds a client from each tenant's own System-Manager key + `site_name`). Ran as-is. Grants `Loyalty Point Entry` create/write so manual point adjust ("Sesuaikan Poin") stops 403-ing. **Applied to all 4 tenants, 0 errors.**

2. **F&B Custom Field `allow_on_submit` reconcile** — ⚠️ the shipped `backfill:fnb-loyalty` **hard-exits in per-tenant mode** (`if (!ERPNEXT_SITE_NAME) process.exit(1)` — "targets shared mode"), and `ensureFnbCustomFields` runs **only at provisioning time** (`provisioning.ts:175`; the kitchen route does **not** self-heal at runtime). So existing F&B tenants whose `custom_kitchen_status` field lacks `allow_on_submit=1` would have **kitchen-status-on-submitted-invoice rejected by ERPNext**. Handled with a **one-off per-tenant script** (`apps/api/src/scripts/backfill-fnb-fields.ts`, the purpose-built tool from the `feat/dine-in-qr` branch — imports only `ERPNextClient` + `ensureFnbCustomFields`, both present on `main`). It filters to F&B tenants, builds a client per tenant (`site_name` + own key), calls `ensureFnbCustomFields` (create-if-missing + reconcile, additive/idempotent, no System/Manufacturing Settings writes). scp'd → dry-run → `--yes` → **deleted** (tree clean).

## Lessons / carry-forward
- **Per-tenant vs shared backfill is a load-bearing distinction on grobiz.** `backfill:fnb-loyalty` is shared-mode-only (hard-exits without `ERPNEXT_SITE_NAME`); for prod F&B field reconcile use the per-tenant `backfill-fnb-fields.ts` pattern (build client from each tenant's `site_name` + own key). `backfill:loyalty-permission` handles both modes itself.
- The kitchen route has **no runtime self-heal** for `allow_on_submit` — the field must be reconciled by a backfill after a deploy that adds/changes managed F&B Custom Field attrs, or kitchen-status-on-submitted breaks for pre-existing F&B tenants.
