## The feature in one paragraph
The **server** now decides how an invoice is paid (`resolvePaymentMode()` in `core/billing/midtrans.ts`): `mock_auto` (dev, no key → settle synchronously), `snap` (a real `MIDTRANS_SERVER_KEY` is set → mint a real Snap token), or `manual` (prod with no key → bank transfer + admin verify). `GET /api/billing/subscription` returns `paymentMode` so the web app renders the right CTA before any click. `POST /api/billing/invoices` supersedes any existing open invoice, creates the invoice, and (in `snap`) mints a Snap token via the sandbox/prod Snap API. The **only** thing that settles a real Snap payment is the webhook `POST /api/webhooks/midtrans` (`verifySignature` → `settleInvoicePaid`): invoice → `paid`, subscription → `active` **on the invoice's plan**, tenant unsuspended, payment recorded, owner notified via the `payment_success` WA template. **There is no settlement poller** — the dashboard notification URL is mandatory for real payments.
