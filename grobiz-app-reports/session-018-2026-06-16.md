## ⚠️ Manual / things to know for testing (real sandbox)
- **Dynamic QRIS (Mode A & B) calls the Midtrans Core API `/v2/charge`** — this must be **enabled on the sandbox merchant** (Snap is on by default; Core API is separate). Without it, dynamic-QRIS charges error. Mode C (static QR, manual-confirm) and pay-at-cashier don't need it.
- **QRIS settlement webhook:** `https://api.dev.gro.biz.id/api/webhooks/midtrans/orders` (distinct from the subscription Snap webhook `…/api/webhooks/midtrans`). Dynamic charges send it per-charge as `X-Override-Notification` (built from `API_BASE_URL`, already set), so no dashboard change is strictly required; you may also set it as the merchant's default Payment Notification URL.
- **How to test (from `docs/testing/qris-modes-testing-guide.md`):**
  - **Mode A** (GroBiz holds + disburses): owner → **Settings → Pembayaran QRIS** = "Dikelola GroBiz" (default); **Settings → Rekening Bank** = a payout destination; admin → **Settlement** sets the global fee (e.g. 1.5%). Ring a plain (no-discount, online) cart → dynamic QR → pay in sandbox → settlement dashboard shows held + net; admin records a payout.
  - **Mode B** (merchant's own Midtrans key): owner enters their sandbox key (encrypted at rest with `ERPNEXT_CRED_ENC_KEY`, already set) → fee accrues → fee-invoice.
  - **Mode C** (static QR): manual-confirm sale; fee accrues.
  - POS dynamic-QR **falls back to manual-confirm** when the cart has a discount or the browser is offline (v1 limitation).
