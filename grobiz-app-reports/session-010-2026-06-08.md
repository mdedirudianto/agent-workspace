### How the rename works (and why it's cheap)
Frappe has **no `bench rename-site`** — a site's identity is purely its directory name under `sites/`, and `site_config.json` holds only the hashed `db_name` (+ creds), so the **MariaDB database is decoupled from the site name and survives a plain `mv` untouched**. With `dns_multitenant` on, Frappe routes by Host header to the matching directory, so `renameSite` is just `mv <SITES_DIR>/<old> <SITES_DIR>/<new>` + a cache clear (`clearSiteCache`, non-fatal). Admin API keys (stored in the site DB) stay valid. **It is a local op on `erp` (30s timeout) — NOT a chatty DB workload, so it is unaffected by the erp↔db latency that blocks `new-site` (session-009).** Idempotent across retries: old==new → no-op; source missing → no-op; target present (stale prior attempt) → drop-then-mv.

### ROOT CAUSE FOUND + VERIFIED — it's `-tt`, not latency (corrects session-009)
On user request, cleaned the pool (dropped the hung/available unclaimed sites + 2 session-009 orphans on erp, kept the 3 live claimed tenants) and re-enabled for a clean retry. Builds **failed again** — but the pm2 logs gave the real cause: every failed build's captured **`stdout` was `"Enter mysql super user [root]: "`**. `bench new-site` is **blocking on an interactive prompt** for the mysql super user and hanging forever (no stdin over non-interactive ssh) — which is why it sits at 0 tables with no MariaDB connection. The trigger is the **`-tt` flag added in `772ebde`** (remote-kill feature): `ssh -tt` forces a remote pseudo-TTY → bench goes interactive → prompts → hangs.

**This overturns the session-009 latency theory.** Those weren't slow-DDL timeouts; they were builds **hung at this prompt** until the 30-min timeout killed them. The erp↔db link is fine (idle ping ~2.5ms). ICMP latency was a red herring.

**Verified fix (empirical):** ran `bench new-site nott-test1 … --mariadb-root-password … --admin-password …` over ssh **without `-tt`** and **stdin `< /dev/null`** → ran normally: `Installing frappe… Updating DocTypes 0→75%`, **196 tables created**, exit 0 (only cut off by my 140s test cap). Dropped the throwaway after.

**The fix is a code change in `apps/api/src/core/erpnext/bench.ts`:** stop forcing `-tt` for `new-site`/`install-app` (the `opts.tty` → `'-tt '` path in `runBenchCommand`), and pipe stdin from `/dev/null` for defense-in-depth. The `-tt`-based remote-kill it removes was *already* broken (session-009: the timeout SIGTERMs the `sh` wrapper, not ssh), so removing it loses nothing; a proper remote-kill (process-group) is a separate follow-up. After the code fix + redeploy, re-enable `SITE_POOL_ENABLED=true` and the warmer should fill normally.
