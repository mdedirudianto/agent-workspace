## 🐞 Bug found + fixed during prod verification — `ca885c6`
On first activation the warmer logged **"Pool warmer started (target 3 sites)"** but then **threw on every poll tick** and the pool stayed empty:

> `recoverStaleBuilds` (`core/tenant/pool.ts`) interpolated a JS `Date` (`cutoff`) into a **raw `` sql`` `` template** passed to `db.execute`. Unlike the query-builder calls in the same file (which serialize Dates via column type), a raw template hands the bare `Date` to the **postgres.js** driver, which can only bind `string`/`Buffer` params → `TypeError: … Received an instance of Date` at `recoverStaleBuilds` → `pool-warmer.poll`.

**Fix:** rewrote `recoverStaleBuilds` to use the drizzle query builder (`update().set().where(and(eq(status,'building'), lt(updatedAt, cutoff))).returning(...)`), matching `markPoolSiteFailed` directly above it. No schema change. Why tests missed it: `site-pool.test.ts` / `pool-warmer.test.ts` are mock-based and never exercise the real postgres.js bind path.

**Shipped (per user decision — commit to main + redeploy):**
1. Local: edit → `tsc` clean → 9/9 pool tests pass.
2. `git commit ca885c6` → `git push origin main` (fast-forward from `7d7f880`).
3. Prod: `git pull` → `7d7f880` → `ca885c6`; rebuilt `@grobiz/api`; `pm2 reload grobiz-api`.
