# Session 6 — grobiz backup mechanism (db + erp + app → backup server)

**Date:** 2026-06-07
**Servers touched:** `backup` (`10.0.0.4`, all changes), `app` (`10.0.0.5`, one additive SSH key), `erp` (`10.0.0.6`, read-only `bench backup`), `db` (`10.0.0.1`, read-only `pg_dump`)
**Design spec:** [docs/superpowers/specs/2026-06-07-grobiz-backup-design.md](../docs/superpowers/specs/2026-06-07-grobiz-backup-design.md)

## Goal

grobiz went to prod 2026-06-05, **after** the `/opt/backup` framework was last touched (2026-05-25), so almost none of its stateful data was being backed up. Add full coverage — Postgres app DB (db), per-tenant ERPNext sites (erp), app `.env` (app) — landing on the `backup` server, by **extending the existing framework** rather than building anything parallel.

## Decisions

| Topic | Choice |
|---|---|
| Where | **Backup server only** (extend `/opt/backup`); offsite/encryption remains an open fleet follow-up |
| Frequency | **Every 6h** — `0 0,6,12,18 * * *` WIB (server is `Asia/Jakarta`) |
| Retention | **GFS, actively pruned on backup server:** 4 intraday (last 24h) + 7 daily (00:00 WIB anchor) + 4 weekly (Sun 00:00, 30d) |
| Tenant capture | **Frappe-native `bench backup --with-files` per site** (over direct mysqldump / LVM snapshot) |
| Safety | **Never delete/drop on app/db/erp.** All source ops read-only; verify non-destructive; rsync without `--delete`. Deletion happens *only* on the backup server's snapshot store. |

| File | Change |

| `config.sh` | **modified** — added `GROBIZ_INTRADAY_H=24`, `GROBIZ_DAILY_D=7`, `GROBIZ_WEEKLY_D=30` (backup `.bak.2026-06-07`) |

| `verify.sh` | **modified** — excluded `grobiz` from the destructive full-restore loop; added **non-destructive** `verify_grobiz()` (`pg_restore --list` + `gzip -t`, no temp DB / no DROP) (backup `.bak.2026-06-07`) |

**On `app`:** appended one locked-down line to `devops` `authorized_keys` (backup `.bak.2026-06-07-grobizbackup`) — reuses the existing `backup-pull@backup` ed25519 key, restricted `from="10.0.0.4"` + `command="/bin/cat /home/devops/grobiz/.env"` + no-pty/no-forwarding. The key can do nothing but emit `.env` from the backup server's IP. **No new key generated, no other access granted.**

## Safety posture (honored "no delete/drop on live")

Every operation against app/db/erp this session was read-only: `pg_dump`, `bench backup --with-files` (writes only into erp's own backup dir), and a forced `cat .env`. Verification uses `pg_restore --list` + `gzip -t` — **no temp DB created or dropped on db**. `rsync` runs without `--delete`. The only deletions occur on the `backup` server's own snapshot store, which is the explicitly-authorized cleanup.

## Restore runbook

- **Postgres `grobiz`:** `pg_restore --no-owner -d grobiz pg/grobiz/grobiz-<stamp>.dump`.
- **Tenant site:** copy the 4-file set to erp → `bench --site <site> restore <db.sql.gz> --with-public-files <pub.tar> --with-private-files <priv.tar>` (or `bench new-site <site> --restore` if the shell is gone). Keep MariaDB host `10.0.0.1` + `dns_multitenant on`.
- **`.env`:** copy `appconfig/grobiz/env-<stamp>` → `/home/devops/grobiz/.env` on app, `chmod 600`, `pm2 restart grobiz-api`.
- Restore Postgres and tenant sites from the **same-stamped run** so the registry and sites agree.
