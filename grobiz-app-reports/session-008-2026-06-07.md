## The hazard (root cause, from the 0009 header)
`now()`/`defaultNow()` columns were written in the **DB session timezone (prod = UTC+8)** → stored as +8 local wall-clock; JS `new Date()` columns stored as UTC. Old reads interpreted everything as UTC, so `now()`-written values read back ~8h **in the future** (e.g. `provisioning_jobs.created_at > completed_at` → empty "Provision duration"). 0009 fixes this in one pass: each `ALTER ... USING ... AT TIME ZONE '<zone>'` carries an **explicit** zone (`Asia/Singapore` for NOW columns, `UTC` for JS/MIXED columns) so the backfill is correct **independent of who runs the file**, and the whole thing is a single guarded `DO $$` block (sentinel: `provisioning_jobs.created_at` type) → idempotent and **atomic** (partial application impossible).

## Blocker caught by the dev rehearsal: missing `devices` table
0009 references **31 tables**; **`devices` did not exist in either DB.** It's defined in `schema/devices.ts` (FCM push tokens) and referenced by live code (`push-notification.ts`, `notification-triggers.ts`, `routes/devices.ts`) but was **never created** — `deploy.sh`'s `drizzle-kit push` is deliberately bypassed (the `feature_flags` interactive-hang footgun), and no migration creates it. First dev run of 0009 hit `ERROR: relation "devices" does not exist` and **rolled back atomically** (0 rows touched) — confirming 0009's all-or-nothing safety, but it won't apply until `devices` exists. (0010 is independent/additive and applied fine.)

**Decision (user):** create the empty `devices` table from its schema, then migrate. No data risk — table is empty. Created with plain `timestamp` columns (pre-0009 shape) so 0009 converts it uniformly with the rest; FKs verified valid (`users.id` PK, `tenants.tenant_id` unique).

```sql
CREATE TABLE IF NOT EXISTS "devices" (
  "id" uuid PRIMARY KEY DEFAULT gen_random_uuid() NOT NULL,
  "user_id" uuid NOT NULL, "tenant_id" varchar(50) NOT NULL,
  "fcm_token" varchar(512) NOT NULL,
  "platform" varchar(20) DEFAULT 'android' NOT NULL, "device_name" varchar(255),
  "created_at" timestamp DEFAULT now() NOT NULL, "updated_at" timestamp DEFAULT now() NOT NULL);
ALTER TABLE "devices" ADD CONSTRAINT "devices_user_id_users_id_fk"   FOREIGN KEY ("user_id")   REFERENCES "users"("id");
ALTER TABLE "devices" ADD CONSTRAINT "devices_tenant_id_tenants_tenant_id_fk" FOREIGN KEY ("tenant_id") REFERENCES "tenants"("tenant_id");
CREATE UNIQUE INDEX IF NOT EXISTS "devices_fcm_token_idx" ON "devices" ("fcm_token");
CREATE INDEX IF NOT EXISTS "devices_user_id_idx" ON "devices" ("user_id");
CREATE INDEX IF NOT EXISTS "devices_tenant_id_idx" ON "devices" ("tenant_id");
```

## Deploy procedure (both servers — **no `deploy.sh` / `drizzle-kit push`**)
> ⚠️ `drizzle-kit push` would do a *naive* `ALTER TYPE timestamptz` using the session TZ, bypassing 0009's per-column `AT TIME ZONE` backfill and **corrupting data**. This deploy used manual `pnpm --filter` builds + `psql -f` migrations only.

1. `git pull`; `pnpm install --frozen-lockfile`; build `shared` + `api` + `admin` (clean).
2. `CREATE` the empty `devices` table.
3. **Tight window:** `pm2 reload grobiz-api --update-env` (new code → session UTC) → **immediately** `psql -f 0009` → `psql -f 0010`.
4. reload `grobiz-admin` + `grobiz-landing` (dev: rsync `admin`/`landing` dist → `/var/www/grobiz/...`); `pm2 save`.
