## Notes for future grobiz dev/app redeploys (pattern to remember)

**Pre-flight checklist, every redeploy:**
1. Diff `<deployed-SHA>..origin/main` — commit count, file count, LOC.
2. **DB migrations**: find highest numbered file in `apps/api/drizzle/`, list every new one, read its SQL, flag anything destructive (drops, non-idempotent UPDATEs, uniqueness constraints that could collide with live data — pre-check collisions on the target DB before applying).
3. **ERPNext**: grep the diff for `erpnext`, `Custom Field`, `custom_`, `ensureFnbCustomFields`, `backfill`, `DocType`. New custom fields/doctypes only auto-provision for **new** tenants — existing tenants need the matching `backfill:*` script run. Dev is **shared-site** ERPNext (`ERPNEXT_SITE_NAME=dev.gro.biz.id`) so one backfill run covers all tenants; **prod is per-tenant** ERPNext, so a prod backfill must loop each tenant's own site (see sessions 015/016).
4. **Env**: diff `.env.example`; grep for new `process.env.*` refs not already covered. Also sanity-check any new fail-closed boot invariants (e.g. secret-distinctness checks) against the live `.env` *before* deploying, not after a crash-loop.
5. **Deps**: check `pnpm-lock.yaml` + each `package.json` for new/changed deps, especially anything needing native compilation or `file:`-path vendored tarballs.
6. **App scope**: check which of api/web/admin/order/landing actually changed (file count) — skip rebuilding+rsyncing anything untouched.

**Hard rule: never use `drizzle-kit push` (`db:push` / `deploy.sh`) for grobiz.** It hangs on an interactive truncate prompt on the `feature_flags` index under non-TTY SSH, and has previously risked silently corrupting a `timestamptz` backfill by doing a naive session-TZ-dependent `ALTER TYPE`. Always apply new migration files individually via `psql -1 -f apps/api/drizzle/00NN_*.sql` (transactional per-file — a failure rolls back cleanly, doesn't half-apply).

**Build order convention:** `shared → api → web → admin → order → landing`, only rebuilding what changed. `web` is served same-origin by the API (`@fastify/static`, no rsync needed) — but `admin`, `order`, and `landing` are static dirs under `/var/www/grobiz/{admin,order,landing}` that need an explicit rsync after build. `order`'s `.env.production` hardcodes the **prod** API URL, so a dev build always needs `VITE_API_URL=https://app.dev.gro.biz.id` passed inline.

**Verification convention:** PM2 status+version, health endpoint (postgres/redis/erpnext), vhost 200s for every touched static app, and — the step that actually catches a stale-deploy bug — re-fetch the served page and grep for the bundle hash to confirm it matches what was just built, not a cached prior build.
