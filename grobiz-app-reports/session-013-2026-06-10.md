## Per-tenant tax-account backfill (the "backfill to existing users")
Prod is **per-tenant sites**, so — unlike dev's one-shared-site backfill — the script iterates active tenants and builds a **per-tenant** ERPNext client, exactly mirroring `core/tenant/erpnext.ts` `buildTenantErpnext()`: swap the tenant's `site_name` onto the hostname of `ERPNEXT_URL`, authenticate with the tenant's **own** stored `erpnext_api_key/secret` (plaintext columns; the per-tenant `api-<id>@grobiz.local` user is a System Manager, so it can type the account + create the template). For each tenant it fetched the site's single Company, then called `provisionTaxConfig(erpnext, company, abbr, businessType)` — the exact step-12b logic.

- One-off script `apps/api/src/scripts/backfill-tax-accounts-prod.ts`, run via `pnpm --filter @grobiz/api exec tsx`. **Default dry-run; `--apply` to write.**
- **Dry-run** (fresh read): all 4 tenants reachable (via the Cloudflare hairpin that runtime tenant calls already use), each `2130 account_type=''`.
- **Apply:** 4/4 → `account_type='Tax'`. **Independent fresh re-read** (second dry-run, separate process/client) confirms all 4 now read `'Tax'`. Script deleted afterward; tracked tree clean.
