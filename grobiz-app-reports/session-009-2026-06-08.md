## Verification → POST-MORTEM: the pre-warm pool is non-viable on current infra

The note's third ask — "after deploy, verify a pool build reaches `available`" — **failed**, repeatedly and informatively.

### Timeline of attempts
| new-site timeout | build | outcome |
|---|---|---|
| 300000ms (default) | `pool-d5ea0ba6` | **timed out at 300s** (`bench new-site`) |
| 600000ms (bumped) | `pool-192820d9` (clean, current-api child, no reload interference) | **timed out at 602s** (`bench new-site`) |
| 1800000ms (bumped) | `pool-073e89f2` (clean) | **timed out at 1802s — `bench new-site` never finished in 30 min** |

`new-site` — not `install-app` — is the step that times out. It builds the framework schema by running **tens of thousands of sequential DDL/INSERT** statements against the tenant's MariaDB.

### Root cause: erp↔db is a high-jitter overlay link, not a LAN
MariaDB lives on the shared `db` server (`10.0.0.1`); `bench` runs on `erp` (`10.0.0.6`); every query is a network round-trip over the "private" `10.0.0.0/24`. Measured **idle** (warmer off, zero builds):
- `ping 10.0.0.1` ×20: **min 1.28 / avg 11.75 / max 136.4 / mdev 29.0 ms** — the floor proves it *can* be fast, but the avg + 136ms spikes show it's a jittery routed/overlay link, **not** true L2-local.
- At ~12ms avg × tens of thousands of round-trips, `new-site` is inherently minutes-to-tens-of-minutes and **wildly variable** → blows past even 30 min.

This is **architectural**: ERPNext provisioning assumes a local (sub-ms) DB. Putting tenant MariaDB on a remote shared host over this link is the real blocker. The note's "remote-MariaDB → ~200s new-site" was an optimistic earlier sample; current conditions are far worse.

### Second bug found: the `-tt` remote-kill does NOT kill orphans
The release's headline "no orphaned ~200s builds" claim does **not hold**. On timeout, Node's `exec()` SIGTERMs the **`/bin/sh -c` wrapper**, not the `ssh` grandchild → the `ssh` (and its remote `bench`) **survive, reparented to init (ppid=1), and keep running**. Confirmed live: `pool-192820d9`'s ssh was still executing `bench new-site` minutes after its row was marked `failed`. Combined with the warmer respawning to refill the pool, this produced a **death spiral**: failed builds leave live orphans + the warmer launches new builds → **8 concurrent `bench new-site` processes** all hammering the shared MariaDB → latency worse for all → all time out → more spawn. (Note: ~half of my own confusion early on was self-inflicted — `pm2 reload` mid-build also orphans in-flight ssh; but the post-timeout orphan is a genuine code defect independent of reloads.)

### Action taken: warmer disabled (reversible) to stop active harm
The spiral was degrading a **shared production DB server** (used by other apps). Set **`SITE_POOL_ENABLED=false`** in prod `.env` + `pm2 reload`. Killed all in-flight/orphaned bench ssh server-side (`setsid` detached kill — the interactive ssh kept dropping at exit 255 under the thrash; also note `pgrep -f "ssh.*frappe@…"` matches its **own** command line, a false-positive that cost a connection). Purged the 8 new failed rows. Load recovered (12+ → ~2.2).

**Final prod state:** api `0.9.2` healthy · `SITE_POOL_ENABLED=false` · `ERPNEXT_NEWSITE_TIMEOUT_MS`/`INSTALL` left at `1800000` · pool = **1 `available` + 3 `claimed`** (reserve intact) · **0** in-flight builds · **0** orphan ssh · **0** failed rows. Signups fall back to on-demand provisioning (same latency, but no background pileup).

## Notes / learnings
- **The deploy is the right code; the blocker is infra, not the merge.** Credential redaction works, timeouts are now tunable — both good. But no timeout value fixes a `new-site` that exceeds 30 min because of per-query network latency.
- **`exec()` timeout kills the shell, not the grandchild.** A real fix needs `spawn(..., {detached:true})` + `process.kill(-pid)` (kill the process group), or spawn `ssh` directly without a shell. The `-tt` SIGHUP path only works if the *ssh* itself is signalled.
- **Warmer concurrency made it catastrophic.** Even with the kill fixed, the warmer should build **serially (concurrency 1)** against a high-latency DB — N concurrent new-sites make each one slower.
- **`pgrep -f`/`pkill -f` on a self-describing pattern matches itself** — bracket the pattern (`[s]sh …`) or `ps | grep` to avoid killing your own shell.
