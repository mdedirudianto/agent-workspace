**Date:** 2026-06-16
**App:** `seenow` (github.com/biji-dev/seenow, branch `main` @ `5849ec9`)
**Host:** `dev` — `deploy@154.26.130.96`, Ubuntu 22.04.5 (no Docker; system nginx + PM2 + local Postgres 14/Redis)

**Env:** `dev` — `NODE_ENV=production` posture, seeded data, real Anthropic OAuth. NOT production.

Seenow is a Bun + Turbo monorepo — an AI-tutor LMS (Hono modular-monolith API, BullMQ workers,
a Next.js public site, and four Vite/React SPAs). Upstream ships a Docker-Compose deploy
(`deploy/docker-compose.yml`); this session adapts it to the dev host's bare-metal model
(system nginx + PM2 + local PG/Redis), the same way eventopia is deployed here.

## Environment

Root `.env` (`~deploy/seenow/.env`, `0600`) is read by Bun (api + workers). Build-time frontend
env lives per-app (Next/Vite load from their own dir, not the monorepo root):

- `apps/web-public/.env.production` — `NEXT_PUBLIC_API_URL` + portal URLs (`https://*.dev.seenow.ac`).
- `apps/web-{learner,teach,b2b,admin}/.env` — `VITE_API_URL`, `VITE_PUBLIC_URL`, `VITE_TEACH_URL`.

Key root values: `DATABASE_URL`=owner role · `APP_DATABASE_URL`=`seenow_app` (RLS) ·
`REDIS_URL=…/3` · `BETTER_AUTH_URL=https://api.dev.seenow.ac` · `SESSION_COOKIE_DOMAIN=.dev.seenow.ac`
· `WEB_ORIGINS`=the 6 portal origins · `API_PORT=4900` · `AI_PROVIDER=anthropic` +
`ANTHROPIC_AUTH_MODE=oauth` + `ANTHROPIC_AUTH_TOKEN` (Claude subscription) + `EVAL_MODEL=claude-opus-4-8`.
Generated secrets (`BETTER_AUTH_SECRET`, `AUDIT_CHAIN_PEPPER`, both DB passwords) are stashed
root-only at `/root/.seenow-secrets`. Object storage (`S3_*`) left at defaults — **deferred**
(the API code does not consume them yet). OTEL endpoint empty (telemetry no-op).

## Findings / gotchas

1. **pgvector 0.8.0 was already installed on the local PG14** — `CREATE EXTENSION vector` + HNSW
   work natively; no Docker Postgres / pg16 needed. (Biggest pre-deploy risk, eliminated.)
2. **`bun db:migrate` fails with a `:seenow` password auth error** — the root script uses
   `--filter @seenow/api`, which runs in `apps/api`, where Bun looks for `.env` (none) and
   `DATABASE_URL` falls back to its `postgres://seenow:seenow@…` default. Fix: run the migrator
   from the repo root (`bun run apps/api/src/db/migrate.ts`) so Bun loads the real root `.env`.
3. **Frontend build-time env must be per-app** — turbo strict env mode doesn't forward
   `NEXT_PUBLIC_*` / `VITE_*` from root; Next/Vite load env from their own app dir. Wrote
   `apps/web-public/.env.production` + `apps/web-*/.env`. (Stale `localhost:4900` strings remain in
   the minified bundles as the dead `?? "…"` fallback — the active baked value is the real URL.)
4. **PM2 only treats `*.config.cjs` files as ecosystems** — a `seenow-pm2.dev.cjs` got run as a
   single script; renamed to `seenow.config.cjs`.
5. **Owner role kept off full superuser** — `rls.sql`/seeds need to bypass `FORCE RLS`, so the owner
   role has `BYPASSRLS` (not cluster-wide superuser); the `vector` extension was pre-created as
   postgres. More contained on this shared PG cluster.
6. **CF token scope** — `seenow.ac` was not in the existing dev token; user supplied a dedicated
   `seenow.ac` token (works for dev + prod), stored separately so other zones' renewals are untouched.
